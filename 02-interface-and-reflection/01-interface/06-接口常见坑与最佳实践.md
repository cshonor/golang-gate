# 06 - 接口常见坑与最佳实践

前置：[05-类型断言](./05-类型断言.md)。与 [02-空接口eface与nil](./02-空接口eface与nil.md) 中的 nil 语义对照阅读。

---

## 1. 常见坑

| 坑 | 说明 |
|----|------|
| **接口值非 nil，但 data 为 nil** | 类型已绑定，值为 `nil` 指针；与「纯 nil 接口」不同，见 [02](./02-空接口eface与nil.md)。 |
| **断言不检查 `ok`** | 失败时 panic；用 `v, ok := x.(T)`。 |
| **大接口滥用** | 方法多的接口难以 mock、测试与演进；优先小接口（如 `io.Reader`）。 |
| **在热路径上频繁接口转换** | 有类型断言与分配成本；能写具体类型时不必强行接口化。 |

---

## 2. 实践习惯

- **接受接口，返回结构体**（常见 API 设计建议）：参数用接口表达能力，返回具体类型降低调用方负担。  
- **接口由使用方定义**（Go 惯例）：在消费方包内定义小接口，而非大而全的「通用接口层」。  
- **版本演进**：给接口**新增方法**会破坏实现方；更稳妥是新增接口组合或新包名。
