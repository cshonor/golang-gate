# 为什么要用 CAS 以及 Go 内存屏障实现

> 升华：**内存屏障**管顺序与可见性；**CAS** 把「判断 + 修改」变成**原子**的一步——两者层次不同，常配合使用。

## 1. 为什么要用 CAS

普通写法：

```go
if v == expected {
    v = newValue  // 两步之间可能被别的线程打断
}
```

两步之间**不是原子的**，会出现 lost update。

**CAS** 在硬件支持下把「读-比较-写」做成**一条原子操作**，从根上消除这一步的竞态。

**结论**：需要**无锁**或**极短临界**且只动一个字长时，CAS 是核心工具；复杂不变量仍应用 Mutex。

## 2. CAS 比「只靠内存屏障」强在哪

- **内存屏障（memory barrier / fence）**：主要保证 **指令重排边界** 与 **可见性**（何时把写刷到别的 CPU 可见），**不**把「if + 写」变成原子。
- **CAS**：在屏障/原子指令配合下，提供 **RMW（read-modify-write）原子性**。

面试表述：

> 屏障解决「看见什么、以什么顺序看见」；CAS 解决「这一整块判断+写入是否被打断」。

## 3. Go 里没有 Java 的 `volatile`，靠什么？

Go 内存模型依赖：

- **`sync/atomic`**：原子操作隐含必要同步，用于无锁计数、标志位等。
- **`sync.Mutex` / `RWMutex`**：临界区内外可见性由锁保证。
- **`channel`**：send/receive 与 happen-before 关系。

**不要**自己用「普通变量 + 想当然顺序」模拟跨 goroutine 可见性；该用 atomic、锁或 channel。

## 4. 与 Mutex 快路径的关系（闭环）

- Mutex **快路径**里用 CAS 尝试抢锁 → 依赖原子性与屏障保证状态一致。
- 失败进**慢路径** → 排队与唤醒，同样要在正确点发布内存可见性。
