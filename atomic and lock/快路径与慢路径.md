# 快路径与慢路径

> 这是整个锁体系的地基：先理解「快/慢」，再理解 Mutex、自旋、futex、atomic 各自站哪一侧。

## 1. 什么是快路径（fast path）

- **在用户态完成**，尽量不陷入内核。
- 常用手段：**CAS**（Compare-And-Swap）等原子指令，尝试把锁从「未锁」改成「已锁」。
- **不阻塞、不睡眠**：当前 goroutine 继续跑，没有「挂起等锁」。
- **极快**：一次或几次原子操作就抢到锁，没有系统调用开销。

直觉：**能在家门口解决的事，就不进内核。**

## 2. 什么是慢路径（slow path）

- 快路径**抢锁失败**（锁已被占用）后进入。
- 往往涉及：**排队、阻塞、睡眠**，把 CPU 让给别的线程/goroutine。
- 可能走到 **futex** 等路径：用户态登记「我要睡」，内核里真正阻塞，被唤醒后再回来抢锁。
- **省 CPU**：避免在锁上死循环空转。

直觉：**抢不到就排队睡觉，别霸占 CPU 自旋。**

## 3. Mutex 的核心设计思想（背）

**先走快路径（CAS 尝试加锁），失败则进入慢路径（排队/阻塞/可能睡眠）。**

- 快路径成功：加锁成本极低。
- 快路径失败：进入慢路径，保证**公平性、调度友好**，避免纯自旋拖死整个 P 上的调度。

## 4. 和「自旋」的关系

- 部分实现会在**进入睡眠前**短自旋（混合策略）：赌锁很快释放。
- **纯手写 CAS + 无限自旋** = 自旋锁，**不是** Go 官方 Mutex 的完整故事；Mutex 最终要落到「让出」与慢路径。
