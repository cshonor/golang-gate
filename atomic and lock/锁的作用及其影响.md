# 锁的作用及其影响

> 课表：**锁实现原理 · 基础版** — 第 1 节

## 1. 锁要解决什么

- **临界区（critical section）**：多线程/多协程同时读写共享数据时，需要 **互斥** 保证「同一时刻最多一个执行者」。
- **没有锁**：可能出现 **竞态条件（race）**，结果依赖调度顺序 → 数据损坏、逻辑错误。


- Go 对应场景：多个 Goroutine 同时操作共享变量、切片、Map 等非线程安全结构时，包裹操作的代码块就是临界区。

## 2. 锁带来的影响（代价）

| 方面 | 说明 |
|------|------|
| **正确性** | 避免数据竞争，保证不变量 |
| **性能** | 串行化临界区；高竞争时吞吐下降 |
| **延迟** | 等待锁 → 自旋耗 CPU 或阻塞耗调度 |
| **死锁/活锁** | 加锁顺序不当、重复加锁等 |

- 本质：程序的执行结果依赖于 Goroutine/线程的调度顺序，是并发编程的经典 Bug 源。
​

- 典型危害：
​
- 共享数据损坏、计数错误（如计数器多次自增结果异常）
​
- 业务逻辑错误、状态不一致
​
- 偶现、难以复现的线上故障

- Go 检测工具：可通过  go run -race  编译运行，自动检测数据竞争。

## 3. 与「更轻量」手段的边界

- **原子操作（atomic）**：适合简单计数、标志；复杂不变量仍常要锁。  
- **无锁结构**：难度大，需严格内存序；一般团队优先 mutex。

## 4. 自检

- 举例：哪些场景「有锁」比「atomic 一把梭」更合适？  

核心判断标准：临界区是否包含「多步操作/多变量/复杂逻辑」

 
- ✅ 适合用锁的场景：

​
1. 多变量/多步操作：比如「先判断库存，再扣减，再更新订单状态」，原子操作无法保证多步的原子性，必须用锁。
​

2. 复杂数据结构：比如  map 、链表、自定义结构体，原子操作无法保护整个结构，必须用锁。

​
3. 临界区逻辑复杂：包含条件判断、循环、IO 操作等，原子操作无法覆盖。
​

4. 代码可维护性优先：团队对无锁/原子操作不熟悉，用锁更易维护，减少线上故障。
​

- ❌ 不适合用锁的场景：

简单计数、单标志位，用  atomic  性能更高，无锁竞争

- 锁粒度粗 vs 细各有什么权衡？

锁粒度	优点	缺点	适用场景

粗粒度（大锁）	实现简单，代码易维护，锁竞争少	临界区大，并发度低，性能差	低并发场景、逻辑复杂的操作，优先保证正确性

细粒度（小锁）	临界区小，并发度高，性能好	实现复杂，易出现死锁、锁顺序问题，维护成本高	高并发核心链路，极致性能优化
