# 锁的作用及其影响

> 课表：**锁实现原理 · 基础版** — 第 1 节

## 1. 锁要解决什么

- **临界区（critical section）**：多线程/多协程同时读写共享数据时，需要 **互斥** 保证「同一时刻最多一个执行者」。
- **没有锁**：可能出现 **竞态条件（race）**，结果依赖调度顺序 → 数据损坏、逻辑错误。


- Go 对应场景：多个 Goroutine 同时操作共享变量、切片、Map 等非线程安全结构时，包裹操作的代码块就是临界区。

## 2. 锁带来的影响（代价）

| 方面 | 说明 |
|------|------|
| **正确性** | 避免数据竞争，保证不变量 |
| **性能** | 串行化临界区；高竞争时吞吐下降 |
| **延迟** | 等待锁 → 自旋耗 CPU 或阻塞耗调度 |
| **死锁/活锁** | 加锁顺序不当、重复加锁等 |

- 本质：程序的执行结果依赖于 Goroutine/线程的调度顺序，是并发编程的经典 Bug 源。
​

- 典型危害：
​
- 共享数据损坏、计数错误（如计数器多次自增结果异常）
​
- 业务逻辑错误、状态不一致
​
- 偶现、难以复现的线上故障

- Go 检测工具：可通过  go run -race  编译运行，自动检测数据竞争。

## 3. 与「更轻量」手段的边界

- **原子操作（atomic）**：适合简单计数、标志；复杂不变量仍常要锁。  
- **无锁结构**：难度大，需严格内存序；一般团队优先 mutex。
