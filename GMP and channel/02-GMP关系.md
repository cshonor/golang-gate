# GMP 关系（调度模型鸟瞰）

## 1. 三个字母

| 符号 | 含义 | 直觉 |
|------|------|------|
| **G** | Goroutine | 要执行的函数闭包 + 栈 + 状态 |
| **M** | Machine（OS 线程） | 真正在 CPU 上跑代码的载体 |
| **P** | Processor（逻辑处理器） | 本地可运行 G 队列、执行 Go 代码所需的上下文；数量 ≈ `GOMAXPROCS` |

- P = Java 线程池的【核心线程】（数量固定，控制并发上限）
​
- M = Java 线程池的【工作线程】（可以多，可以阻塞，可以替换）
​
- G = 提交给线程池的【任务】

P ≈ CPU 核心数（逻辑处理器），M 是操作系统线程（可大于 P），G 是协程（数量无上限，远大于 M/P）

层级关系： G（协程，数量最多） >> M（线程，可多于P） > P（逻辑处理器，≈ CPU核心数）

## 2. 协作关系（简化）

- **M 必须关联 P** 才能执行 Go 代码（常见路径）。  

- **P 从自己的本地队列 + 全局队列 + 偷取其他 P** 取 G 给 M 执行（work stealing）。 
 
- G 阻塞在 channel 等上时，M 可与 P 分离或运行其他 G（取决于阻塞类型）。

1. M必须关联P才能执行Go代码：M（OS内核线程）只有绑定P（逻辑处理器），才能获得执行Go代码的上下文，否则只能处理系统调用等非Go代码逻辑。

​
2. P的任务获取逻辑（work stealing）：每个P维护本地可运行G队列，优先执行本地任务；本地为空时，从全局队列取任务，或从其他P的本地队列「偷」任务，最大化CPU利用率，减少锁竞争。

​
3. G阻塞时的动态调度：当G因channel等待、网络IO等阻塞时，M会与P解绑（或切换执行其他G），P可绑定新的M继续执行其他G，避免CPU空转。

## 3. 常见现象对照

- **CPU 打满**：多个 G 在少数 P 上时间片轮转。  
- **网络 IO**：往往用 netpoller，G 可不长期占 M（概念上「异步」等就绪）。  
- **阻塞 syscall**：可能占住 M，运行时可能需要更多 M。

现象	本质（GMP视角）

CPU打满	多个G在少数P上通过时间片轮转执行，P数量对应CPU核心数，充分利用CPU算力

网络IO	Go通过netpoller（网络轮询器）实现异步IO，G阻塞时不长期占用M，M可执行其他G，实现高并发

阻塞syscall	系统调用会阻塞M，此时P会与M解绑，Go runtime会创建/唤醒新的M绑定P，因此高syscall场景下M数量会远多于P

## 4. 与 channel 的关系

- channel 操作在 runtime 里会 **操作 G 的状态**（就绪/等待），与调度紧密相关；FIFO 等见 [10-channel阻塞原理.md](./10-channel阻塞原理.md)。

## 6. 自检

- `GOMAXPROCS=1` 时，两个无限循环的 goroutine 会怎样？ 

 ✅答案：两个goroutine会通过时间片轮转交替执行，不会出现一个死循环独占CPU、另一个永远不执行的情况。
 
- 原理：Go runtime会在G执行达到时间片阈值（默认10ms）时触发抢占式调度，暂停当前G，切换到另一个就绪G执行，保证所有G都能获得CPU时间。
​
- 补充： GOMAXPROCS=1 意味着只有1个P，因此同一时间只能有1个G在CPU上执行，两个G通过时间片轮转共享这1个CPU核心。

- 为什么说 P 是「中间层」，而不是只有 G 和 M？

✅ 答案：P的存在是Go调度器高性能的核心，解决了纯G-M模型的痛点：
 
1. 解决全局锁竞争：纯G-M模型依赖全局队列，多线程取任务会产生严重锁竞争；P维护本地无锁队列，大幅减少锁开销。
​
2. 实现work stealing负载均衡：P作为调度单元，本地队列空时可从其他P偷任务，保证CPU负载均衡，避免资源浪费。
​
3. 隔离调度上下文：P存储执行Go代码所需的全部上下文（如内存分配缓存、调度状态），M绑定P即可执行，无需全局状态，提升调度效率。
​
4. 控制并发上限：P数量对应 GOMAXPROCS ，限制同时执行的G数量，避免过多内核线程导致的上下文切换开销。
