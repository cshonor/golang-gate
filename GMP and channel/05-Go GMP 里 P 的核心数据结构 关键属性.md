# Go GMP 里 P 的核心数据结构（关键属性）

面向读源码与面试：**`runtime.p` 是什么、哪些字段最重要**。

## 1. P 是什么

- **P = Processor（逻辑处理器）**，承载可调度的上下文。
- **M 执行 G 必须绑定 P**：P 提供本地 runq、`mcache` 等 per-P 资源。
- **P 的数量**与并行度相关，通常由 **`GOMAXPROCS`** 约束（默认常接近 CPU 核数）。

一句话：**P 是 Go 用户级调度里真正的「调度单元」之一**（与全局 `sched`、窃取策略等配合）。

## 2. 核心数据结构（示意）

摘自 `runtime` 思路的**简化**示意，完整定义以 `runtime/proc.go` 等为准：

```go
type p struct {
	status uint32
	id     int32

	m muintptr // 当前绑定的 M

	// 本地可运行 G 队列（环形队列，重要）
	runqhead uint32
	runqtail uint32
	runq     [256]guintptr

	preempt bool
	// mcache、定时器、统计等字段省略
}
```

## 3. 关键字段（必须理解）

### 1. `runq` — 本地 Goroutine 队列

- **定长环形队列**（常见实现为 256 槽，以源码为准）。
- **与当前 P 绑定的 M** 主要从这里取 G，**无全局锁**，热路径快。
- 新创建的 G **优先**放入当前 P 的本地队列（策略随版本微调）。

### 2. `runqhead` / `runqtail`

- 环形队列的队头、队尾下标，配合 `runq` 使用。

### 3. `m`

- 当前与该 P 绑定的 **M**。
- **同一时刻 1P ↔ 1M**；M 无 P 一般不能直接跑用户 G。

### 4. `status`

P 的生命周期状态（名称以源码为准），例如：

| 状态 | 含义（直觉） |
|------|----------------|
| `_Pidle` | 空闲 |
| `_Prunning` | 正被某个 M 使用 |
| `_Pgcstop` | GC 等场景下暂停调度 |
| `_Pdead` | 已废弃 |

### 5. `preempt`

- 与 **协程抢占**相关；Go 1.14+ 异步抢占路径会用到 per-P / per-G 的抢占标记。

### 6. `mcache`（每 P 一份）

- 每个 P 自带 **`mcache`**，小对象分配走 **无锁** 快速路径（详见内存分配相关笔记 / `runtime/mcache.go`）。

## 4. 为什么 P 重要（浓缩）

1. **本地 `runq`** 减少全局锁与竞争，是高并发调度快的重要原因之一。
2. **M 必须绑 P** 才能稳定地跑 G（拿队列、抢 G、窃取都围绕 P）。
3. **`GOMAXPROCS`** 控制 P 规模，影响**可同时参与调度的并行度**（不等价于「G 的个数」）。
4. **工作窃取**常以 **P 的 runq** 为粒度从其他 P「偷」G。
5. **每 P `mcache`** 让堆上小对象分配尽量无锁。

## 5. 一句话背诵

**P 是带本地 runq 与 mcache 的调度上下文；本地无锁队列是 GMP 性能的关键之一。**

## 复习速记

| 字段 | 记一句 |
|------|--------|
| `runq` | 环形本地队列，取 G 主路径 |
| `m` | 当前绑定的线程 |
| `status` | P 空闲/运行/GC 停等 |
| `mcache` | 每 P 小对象分配加速 |
