# GMP 队列与阻塞流转（就绪队列 / 等待队列 / M-P 行为）

> 由 `GMP 里一共就 3 个队列.md` 与 `G 阻塞后队列归属与 M、P 行为.md` 合并整理。

## 1. GMP 里最核心的“队列分类”

从「G 在哪」的角度，把队列分成两类更不容易混：

### 1.1 就绪队列（只放 _Grunnable 的 G）

1. **P 本地 runq（环形队列）**
   - `runq [256]guintptr` + `runqhead/runqtail`
   - 目标：无锁/低锁、最快

2. **全局就绪队列 sched.runq（链表）**

```go
var sched struct {
    runq     gList
    runqsize int32
    // ...
}
```

- 所有 P 共享
- 访问需要锁（相比本地队列慢）

### 1.2 等待队列（阻塞态 G 不在就绪队列里）

核心结论（面试必背）：

1. **P 只有就绪队列，没有阻塞队列**  
   - `P.runq`、全局 `sched.runq` 只存放 **就绪态 G（_Grunnable）**
   - **阻塞态 G 不会**留在 P 或全局就绪队列
2. **G 阻塞 → 挂到对应事件的等待队列**  
   - 谁引发阻塞，G（准确说是 sudog）就挂到谁的队列里
3. **G 阻塞 ≠ M 阻塞 ≠ P 阻塞**  
   - P 是核心资源：优先保持可运行，不随 G 或 M 一起阻塞

## 2. “GMP 里一共 3 个队列”怎么理解？

如果按很多资料的口径，会说 “3 个队列”：

1. **P 本地 runq（环形队列）**：放就绪 G（最快）
2. **全局 sched.runq（链表队列）**：放就绪 G（有锁）
3. **等待队列（各种阻塞队列）**：放阻塞 G（按事件归属）

这套说法的重点是：**就绪与阻塞分开管理**。

## 3. 什么时候用哪个队列？（调度路径记忆版）

### 3.1 新建 G

- 优先放 **当前 P 的本地 runq**
- 本地满了：会把一部分 G 挪到 **全局队列**（实现细节以 runtime 为准）

更贴近实现的记忆点（不用背代码，背逻辑）：

- **本地 runq 优先**：因为它快（无锁/低锁）
- **runq 溢出会“批量搬运”**：把部分 runnable G 转移到全局 `sched.runq`，降低单个 P 的局部拥塞与后续抢锁频率

### 3.2 M 取 G 执行（优先级）

1. 先从 **本地 P.runq** 拿
2. 没有 → 去偷别的 P 的 runq（work stealing）
3. 还没有 → 去 **全局 sched.runq** 拿一批

补充两个面试常追问的点：

- **偷取通常是“偷一半”**（以 P 为单位做负载均衡，具体比例/策略以版本为准）
- **一次从全局拿一批**：减少锁的次数（不是每次只拿 1 个）

## 4. G 阻塞后：去哪个等待队列？（按事件归属）

G 发生阻塞时，根据原因进入**对应资源对象的等待队列**，不归 P：

1. **阻塞在 channel 读写**：进入 `hchan.recvq` / `hchan.sendq`
2. **阻塞在 sync.Mutex / RWMutex**：进入锁的等待队列
3. **阻塞在网络 I/O**：进入 `netpoll` 等待集合
4. **阻塞在 time.Sleep**：进入定时器结构
5. **阻塞在 WaitGroup/Cond**：进入对应同步对象队列

一句话：**阻塞 G 由事件本身管理，不归 P 管。**

### 4.1 为什么等待队列里挂的是 sudog？

- `g` 本体很大，不适合频繁入队出队  
- `sudog` 是轻量节点：能串链表、能携带必要信息（关联对象、数据地址）  
- 这也是“阻塞/唤醒”能做得很快的原因之一（结构 + 局部性）

## 5. G 阻塞时：M 与 P 的行为（两类阻塞要分清）

### 5.1 用户态阻塞（chan、锁、sleep 等）

1. G 进入阻塞，状态变为 `_Gwaiting`
2. M 把 G（sudog）放入事件等待队列
3. **M 解绑 P**，释放 P
4. P 被其他 M 绑定继续跑 runq

结果：**G 阻塞，M 不必长阻塞，P 继续干活**。

补充一个“错觉纠正”：

- 你看到某个 goroutine “卡住”，不等于 OS 线程被卡住  
- runtime 会把 G park 掉，让 M/P 去跑别的 runnable G  
- 所以 Go 里的“阻塞等待”常常是 **G 等待而非线程等待**

### 5.2 系统调用/内核态阻塞（syscall、cgo 等）

1. G 进入系统调用，M 会被内核阻塞
2. 运行时会尽量 **提前解绑 P**（让 P 不被拖死）
3. 系统调用结束后，M 再寻找可用 P 继续执行

结果：**M 可能被内核阻塞，但 P 尽量保持可运行**。

补充：这也是为什么高 syscall/cgo 场景下，你可能观察到 **M 数量明显增长**（runtime 为了维持 P 的利用率）。

## 6. 把队列知识落到 channel 一条链路（例子）

以 `ch <- v`（发送）为例，按“快路径 → 慢路径”串起来：

1. `chansend` 持锁检查 `recvq`：若有等待接收者 → **直接配对拷贝**并唤醒对方 G  
2. 否则看是否有缓冲：有且未满 → 写入环形队列返回  
3. 否则（无缓冲或缓冲满）→ 当前 G 变成 sudog 入 `sendq` 并 `park`  
4. 未来接收者到来 → sudog 出队 → G 变回 runnable → 放回某个 P 的就绪队列（本地 runq / 全局 runq 视时机而定）

看懂这条链路，就把 **“就绪队列 vs 等待队列”** 真正落地了。

## 7. 面试常问：work stealing 为什么能提升吞吐？

- 大多数时候只操作本地 runq → 少锁竞争  
- 空闲 P 不至于空转 → 去偷忙碌 P 的 runnable G  
- 负载更均衡 → CPU 利用率更高、延迟更稳定

## 8. 速记（10 行背诵版）

- **P 只管就绪队列**：`P.runq` / `sched.runq`
- **阻塞 G 不在就绪队列**，挂到对应事件等待队列
- **M 必须绑定 P 才能跑 Go 代码**
- 取 G 顺序：本地 → 偷取 → 全局
- 用户态阻塞：G 等待，M 解绑 P，P 继续跑
- syscall 阻塞：M 可能卡住，但运行时尽量释放 P
