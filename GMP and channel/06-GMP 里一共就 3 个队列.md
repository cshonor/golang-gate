# GMP 里一共就 3 个队列

这里说的「队列」指 **与 Goroutine（G）调度强相关** 的几类队列；channel 自己的 **buf** 是另一类环形缓冲（存数据，不是 G 队列），见下文补充。

## 1. 三类（G 相关）

| 队列 | 位置 / 结构 | 放什么 | 特点（直觉） |
|------|----------------|--------|----------------|
| **P 本地 `runq`** | 每个 `p.runq` + head/tail | 可运行 G | 环形、**无锁热路径**、最快 |
| **全局可运行 G 队列** | `sched.runq` 等 | 可运行 G | **全局共享、加锁** |
| **等待队列** | channel、锁、runtime 等内部链表 | 阻塞中的 G（常以 **sudog** 挂链） | 多种结构，**不是「一个全局数组」** |

## 2. 全局 Goroutine 队列在哪？

在 **`runtime` 包的全局调度器 `sched`** 里（具体字段名以源码为准），例如：

```go
// 示意：仅突出全局 runq
var sched struct {
	runq     gList  // 全局可运行 G 队列
	runqsize int32
	// 锁、统计、其他全局状态...
}
```

- **`sched.runq`**：全局 **可运行** G 队列。
- 一般为 **链表** 实现（非 P 那种环形数组）。
- **多 P 共享**，访问需 **加锁**。

## 3. 哪些地方用到「队列 / 环形」？

### 1）P 本地 `runq` — 环形队列

```go
runq     [256]guintptr // 槽位数以源码为准
runqhead uint32
runqtail uint32
```

- 基于数组 + head/tail 的 **环形队列**。
- **本 M 绑本 P** 时取 G，**无锁**。

### 2）全局 `sched.runq` — 链表队列

- **链表**、**有锁**、全 P 共用。
- 本地队列满、窃取不到等场景会与之交互（细节以 `proc.go` 为准）。

### 3）channel 缓冲 `buf` — 环形队列（存元素）

```go
// channel 内部示意
buf   unsafe.Pointer
sendx uint
recvx uint
```

- 存 **channel 元素**，**不是 G 队列**。
- 也是环形下标；与 **G 的调度队列**是不同概念。

## 4. 什么时候用哪个？（主路径直觉）

**新建 G**

- 优先放入 **当前 P 的本地 `runq`**。
- 本地满时，可能把 **一半** 挪到 **全局队列**（平衡负载，具体算法以源码为准）。

**M 取 G 执行（常见顺序，概念上）**

1. 先拿 **本地 P `runq`**。
2. 没有则 **工作窃取** 其他 P 的 `runq`。
3. 再没有则从 **全局 `sched.runq`** 取一批。

**阻塞**

- G 进入 **channel / 锁 / runtime** 等对应的 **等待结构**（常以 sudog），**不再在可运行队列里**。

## 5. 三句话背诵

1. **P 本地 `runq`**：环形、无锁、取 G 主路径。
2. **全局 G 队列**：在 **`sched`** 里，链表、有锁。
3. **调度不变量**：**M 绑 P 才能跑用户 G**；同一时刻 **1P ↔ 1M**（绑定关系可换）。

## 复习速记

| 概念 | 别和谁混淆 |
|------|------------|
| 3 类 G 队列 | 本地 runq / 全局 sched.runq / 各类等待链 |
| channel `buf` | 环形存 **数据**，不是 **G 队列** |
