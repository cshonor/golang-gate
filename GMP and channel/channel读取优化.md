# channel 读取优化（实现/性能笔记）

> 课表文件名常为 `channel读取优化.md`，不同版本 runtime 细节会变，以下记 **常见思路**。

## 1. 编译器与 runtime 协同

- 部分 `select`、简单 channel 操作可能走 **专用路径**（如直接 runtime 调用），减少通用分支开销。  
- **逃逸分析**：未逃逸的 channel 可能更利于优化（仍依赖版本）。

## 2. 与接收相关的优化直觉

- **直接收发配对**（无缓冲且对方已在等）：可 **跳过缓冲区**，减少内存拷贝次数。  
- **批量唤醒**：某些路径会合并调度开销（概念层，以源码为准）。

## 3. 使用层面的「优化」

- 高并发下 **细粒度 channel** 过多 → 锁竞争；可考虑 **分片、worker、带缓冲适度背压**。  
- 不要误用 **过大缓冲** 掩盖设计问题（内存与延迟）。

## 4. 对照实验思路

- `go test -bench` 对比：无缓冲 vs 小缓冲 vs 大缓冲在同一模式下的吞吐。  
- `go build -gcflags=-m` 看逃逸（辅助理解，不必死记）。
