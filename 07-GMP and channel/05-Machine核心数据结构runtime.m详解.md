# Machine（`runtime.m`）核心数据结构：精读导读

> Go 调度里 **M = 内核线程（执行载体）**，与 **P（调度上下文）**、**G（协程）** 绑定协作。本仓库未单独拆一篇「只讲 `m` 每个字段」的长文，本篇把 **`m` 在 GMP 里扮演的角色** 与 **应去读哪些分篇** 串起来，避免学习链断裂。

---

## 1. `m` 是什么（一句话）

- **M**：真正在 CPU 上跑指令的 **OS 线程** 在 runtime 里的代表。  
- **G** 必须通过 **M** 执行；**M** 必须关联 **P** 才能执行 **G**（常规路径）。

## 3. 面试常问（与 `m` 相关）

- **channel 阻塞的是 G**：M 仍可执行其他 G（若 P 未被 syscall 拖死）。  
- **阻塞 syscall**：可能长时间占用 M，runtime 可能 **解绑 P** 或 **新建 M**，避免整颗 P「被 syscall 饿死」（细节以版本与平台为准，读 [07](./07-Goroutine阻塞后的调度行为.md)）。  
- **与 Java 线程对比**：Go 用少量 M 复用大量 G，降低内核切换与栈内存压力（见 [01](./01-协程与线程的本质区别.md)）。
