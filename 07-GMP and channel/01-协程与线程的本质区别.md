# 协程与线程的本质区别（Goroutine vs OS Thread）

## 1. 一句话

- **Goroutine**：Go 运行时管理的 **轻量协程**，栈可伸缩，创建成本低。  
- **线程（M）**：操作系统调度实体，数量相对少、切换成本高。

进程拆成线程，线程再拆成协程。

## 2. 核心区别（面试速记表）

| 维度 | OS 线程 | Goroutine |
|------|---------|-----------|
| 调度者 | 内核 | Go runtime（用户态协作） |
| 切换成本 | 相对较高（需陷入内核等） | 相对较低（常见路径在用户态） |
| 栈 | 常较大、偏固定 | **小栈起步、按需扩缩**（由 runtime 管理） |
| 典型可承载并发量 | 千～万级需精打细算 | 可轻松上到 **大量** G（仍受内存与业务约束） |
| 同步惯用法 | 锁 + 共享内存 | **channel**（CSP）+ `sync` 原语 |

**为什么要有 GMP**：若「一个 G 永远钉死一个内核线程」，海量并发会迅速被 **内核调度成本** 和 **线程栈内存** 拖垮；GMP 把大量 G **复用到少量 M** 上，在用户态完成多数调度与切换。

## 3. 多对多（概念）

- 大量 **G** 映射到少量 **M** 上执行；由 **P** 参与调度与本地队列管理（详见 [02-GMP模型核心概念总览.md](./02-GMP模型核心概念总览.md)）。
- 不是「一个 goroutine 永远绑定一个线程」；阻塞在 channel 上的 G 会离开 M，M 可运行其他 G。

## 4. 何时会用到更多线程

- **阻塞系统调用**（如部分文件/网络路径）、**调用 C / 阻塞的 runtime 外操作**：可能 **阻塞 M**，运行时会新建或复用 M 保证其他 G 能跑。
- **`runtime.LockOSThread`**：把当前 G 与当前 M 绑定（特殊场景：GUI、某些驱动）。

## 5. 对写代码的启示

- goroutine 便宜 ≠ 可以无限起：内存、调度、channel 背压仍要考虑。  
- **CPU 密集**：通常用 `GOMAXPROCS`（默认约等于 CPU 核数）即可；增加 G 数不线性加速。
