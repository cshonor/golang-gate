# G、M、P、sudog 四个结构体的关系

文字版关系图 + 面试可背要点。

## 1. 总关系（一句话）

| 符号 | 含义 |
|------|------|
| **G（`g`）** | 协程本体：栈、运行现场、状态 |
| **M（`m`）** | OS 线程：真正执行机器指令 |
| **P（`p`）** | 逻辑处理器：本地运行队列、并行度 |
| **sudog** | 对阻塞中 G 的包装：挂在 channel 等等待结构上 |

- **执行路径**：G 要跑 → 必须挂在 M 上跑 → M 必须绑定 P。
- **阻塞路径**：G 阻塞（如 channel）→ 常以 sudog 形式进等待队列 → 让出 M。

## 2. 各自结构（只记关键）

以下均为**示意**，字段名与 layout 以当前 Go 版本 `runtime` 源码为准。

### G（`runtime.g`）

```text
g {
    stack       // 协程栈
    sched       // 调度现场（pc、sp 等，切换用）
    status      // _Grunnable / _Grunning / _Gwaiting ...
    m           // 当前绑定的 M
    waitreason  // 等待原因（chan、mutex、sleep 等）
}
```

### M（`runtime.m`）

```text
m {
    g0          // 调度栈（系统栈），非用户 G
    curg        // 当前正在执行的 G
    p           // 当前绑定的 P
    nextp       // 即将切换绑定的 P
}
```

### P（`runtime.p`）

```text
p {
    runq        // 本地可运行 G 队列（环形）
    runqhead
    runqtail
    m           // 绑定的 M
    gFree       // 空闲 G 缓存等
}
```

### sudog

```text
sudog {
    g       // 被挂起的 G
    c       // 关联的 channel
    elem    // 数据/元素相关指针（收发路径）
    next
    prev
}
```

## 3. 关系怎么记（核心）

1. M 要执行 G，**必须先绑定 P**（有 P 才有本地调度上下文与 runq）。
2. 同一时刻：**一个 M 只绑一个 P，一个 P 只绑一个 M**（可随调度切换绑定关系）。
3. 一个 P 维护**本地可运行 G 队列**；新 G 优先进当前 P 的 runq（策略细节见队列专题）。
4. G **不阻塞**时：在绑定的 M 上正常执行。
5. G **阻塞**（channel、部分锁、runtime 阻塞等）：往往经 **sudog** 进入对应等待结构，**不再占用 M** 跑用户代码。
6. **唤醒**：sudog 出队 / 完成等待 → G 重新可运行 → 进入某 P 的 runq 或全局路径，再由 M 执行。

## 4. 极简总结

- **G**：协程实体（栈、现场、状态）。
- **M**：内核线程，执行载体。
- **P**：调度上下文，持本地 G 队列，限制并行度（与 `GOMAXPROCS` 相关）。
- **sudog**：阻塞等待链路上的节点，把 G 挂到 channel 等结构上。
- **规则**：**M 绑 P 才能跑 G**；**G 阻塞**则常经 **sudog** 离开运行路径，**释放 M**。
