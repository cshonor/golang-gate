# 学 GMP 必学 chan

**结论先说**：Goroutine 的**阻塞、唤醒、切 M、再入队**，很大一部分都发生在 **channel** 路径上。只背 GMP 抽象而不看 `chan`，调度细节会对不上号。

## 1. 为什么算「一套体系」

| 概念 | 分工（直觉） |
|------|----------------|
| **GMP** | G 怎么跑、在哪排队、怎么换 M/P |
| **channel** | G 之间怎么传数据、**什么时候该等、该让出 CPU** |

Go 里常说：**不要通过共享内存来通信，而要通过通信来共享内存。**  
落地就是：**用 G 做并发，用 channel 做协作，用 runtime 做背后的调度。**

## 2. channel 会直接驱动 GMP 行为

写：

```go
<-ch    // 收
ch <- x // 发
```

在**阻塞条件不满足**时，典型会发生：

1. 当前 **G** 进入等待（如挂到 `recvq`/`sendq`）。
2. G 从当前 **M** 上卸下，M 继续找别的可运行 G。
3. 条件满足后，G 再变为可运行，进入 **P 的 runq** 或经全局路径被拉取。

这一串：**阻塞 → park → M 执行别的 G → 就绪 → 再调度**，就是 GMP 日常路径；**channel 是最高频的触发场景之一**。

## 3. 和「队列」笔记怎么串起来

- **P** 上有 **`runq`**（环形数组 + head/tail），管**可运行 G**。
- **有缓冲 channel** 里有 **`buf` + sendx/recvx**（环形缓冲），管**元素**。

都是「环形 + 生产者/消费者」直觉；搞懂一个，另一个更好记（详见 [14-环形队列.md](./14-环形队列.md)）。

## 4. 面试常把 GMP + chan 合着问

典型方向：

- G 阻塞在 channel 上时，**M** 在干什么？G 存在哪？
- **无缓冲 / 有缓冲满、空** 时，**sendq/recvq** 怎么用？
- 与 **sudog**、`hchan` 的关系？

需要同时有 **GMP** 与 **chan 实现向** 的底子（见 [08-sudog详细介绍.md](./08-sudog详细介绍.md)、[10-channel阻塞原理.md](./10-channel阻塞原理.md)）。

## 5. 一句话

**GMP 是调度与执行模型；channel 是最常见的「让 G 阻塞与唤醒」的机制之一，两者在 runtime 里是同一套故事的两面。**
