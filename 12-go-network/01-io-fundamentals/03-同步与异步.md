# 同步与异步

> **01-io-fundamentals · IO 核心理论**  
> 前置：[01-什么是 IO](./01-什么是IO.md)、[02-阻塞与非阻塞](./02-阻塞与非阻塞.md)。**同步 / 异步**描述的是：**一次 IO 的「完成语义」如何落在调用边界上、由谁把数据放进你能直接读写的缓冲里、如何被通知**——与「阻塞 / 非阻塞」**正交**，勿混为一谈。

---

## 一、核心定义（口述版）

### 1. 同步 IO（Synchronous I/O）

**直觉**：调用方发起 `read`/`write` 后，**这次调用的返回**就携带了「**至少阶段性**」的明确结果：例如 `read` 返回 `>0` 表示已把若干字节拷入用户缓冲；返回 `0` 常表示 EOF；返回 `-1` 配合 `errno` 表示失败。教材里常说：**对同步读而言，「调用返回」与「数据已到你指定的用户缓冲」在语义上绑定**（与具体是否阻塞等就绪无关）。

- **同步 + 阻塞**：未就绪则线程睡在 `read` 上，就绪后内核拷完再返回。  
- **同步 + 非阻塞**：未就绪立刻 `EAGAIN`；就绪时同一次 `read` 把可读数据拷入用户缓冲。

### 2. 异步 IO（Asynchronous I/O）

**直觉**：**提交**与**完成**分离——`aio_*`、`io_uring`、IOCP 等路径上，应用先**登记**「把这段数据读到哪 / 写到哪」，**发起调用可先返回**；真正等链路、搬运、落缓冲由内核/子系统在后台推进，**再以事件 / 完成项 / 回调**告诉你「这一笔做完了」。

- **特点**：完成通知与业务线程解耦潜力大；**状态机、生命周期、缓冲所有权**更难写对。  
- **平台差异大**：Linux 上 **POSIX AIO** 与 **`io_uring`** 能力不同；Windows **IOCP** 是另一套心智。

---

## 二、关键区分（面试表）

| 维度 | 同步 IO（以 `read` 为例） | 异步 IO（以完成通知型 API 为例） |
|------|---------------------------|----------------------------------|
| **完成感知** | 多在 **`read` 返回**时，已读到用户缓冲或明确错误 | 多在 **完成回调 / CQ 项** 上，标志整笔请求结束 |
| **谁常负责「拷入用户缓冲」** | 用户态发起的 **`read` 路径** | 依实现：可能内核直拷到登记缓冲 |
| **编程心智** | 线性、易推理 | 回调 / 状态机、易踩竞态 |
| **典型代表** | 阻塞/非阻塞 `read` + `epoll` 通知**可读** | `io_uring`、IOCP（完成端口） |

> **纠偏**：`epoll_wait` 只解决「**哪些 fd 可能就绪**」；你仍要 **`read`** 把字节取上来——整体常被归为 **Reactor + 同步读** 范式，而不是「`epoll` 本身 = 异步 IO」。

---

## 三、Reactor vs Proactor（架构口语）

### 1. Reactor（反应器）

- **事件**：**就绪**（可读 / 可写 / 异常）。  
- **动作**：用户在事件回调里 **`read`/`write`**，把数据搬进/搬出自己的缓冲。  
- **常见组合**：**非阻塞 fd + IO 多路复用**（`select`/`poll`/`epoll`）。  
- **Go**：`netpoll` + 非阻塞 socket，上层 `Conn.Read` 体验接近同步阻塞，底层是这套。

### 2. Proactor（前摄器）

- **事件**：**完成**（读请求已完成、写请求已落盘/已入发送路径等，依定义）。  
- **动作**：内核或框架已把数据放到约定位置，用户处理「业务层结果」。  
- **代表**：Windows **IOCP**；Linux 上 **`io_uring`** 的不少用法更接近这种「提交—完成」心智。

---

## 四、与「阻塞 / 非阻塞」一张嘴讲清

|  | **阻塞** | **非阻塞** |
|--|----------|------------|
| **同步** | 未就绪则睡在 `read`；返回时数据已在缓冲（或错误明确） | 未就绪 `EAGAIN`；就绪时 `read` 一次性搬走可读数据 |
| **异步** | 较少见：发起后仍可能在「等完成」上阻塞（视 API） | 发起先返回；完成靠通知（常见高性能形态） |

---

## 五、Go 里怎么记（一句人话）

- **默认 `net.Conn.Read`**：你写的是 **同步** 风格——这次 `Read` 返回时，**读到的字节已经在你的 `[]byte` 里**（或 `io.EOF`/错误）。  
- **底层**：常见是 **非阻塞 socket + netpoll（多路复用）**；**阻塞的是 G 的调度语义**，不是「每个连接占死一个 OS 线程卡在 `recv` 上」。  
- **是否「POSIX 意义的真异步 AIO」**：**默认不是**；要玩 **`io_uring`** 等需单独评估版本、平台与生态。

深入：[04-Go的IO模型到底是什么](../07-go-netpoll/04-Go的IO模型到底是什么.md)、[02-netpoll是什么](../07-go-netpoll/02-netpoll是什么.md)。

---

## 六、极简总结

- **同步**：**调用边界**上能读到「这一笔读写完 / 明确失败」的结果（与阻不阻塞无关）。  
- **异步**：**提交**与**完成通知**分离，完成语义在另一通道。  
- **Reactor**：通知**就绪**，用户 `read`；**Proactor**：通知**完成**，内核/框架搬运更多。  
- **Go 默认网络栈**：**同步 API + 非阻塞 fd + 多路复用**，别和「异步 IO」混称。
