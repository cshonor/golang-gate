# Chapter 01 — Architectural Layers（架构分层）

> 对应原书 *Network Programming with Go Language, 2nd Ed.* 第 1 章。  
> 若说后面章节教的是「用 Go **写**网络代码」，本章教的是「**如何思考**网络与分布式环境」——不了解架构与故障模型，就很难写出健壮系统。

---

## 1. 协议层与数据流（Protocol Layers & Encapsulation）

为控制复杂性，把网络行为拆成**层级**是最常用的抽象。

- **OSI 七层 vs TCP/IP**  
  书中对比**理论上的 OSI**（物理层、数据链路层、网络层、传输层、会话层、表示层、应用层）与**工程上主流的 TCP/IP 栈**。在 Go 网络编程里，日常最常打交道的是 **IP（网络层）** 与 **TCP/UDP（传输层）**，再往上是各类应用协议（HTTP、DNS、TLS 等）。

- **数据包封装（Packet Encapsulation）**  
  数据沿协议栈**向下**传递时，每一层会为本层 PDU 加上自己的 **Header**（有时还有 Trailer）。例如一个 **TFTP** 报文在真正发到链路上之前，会依次被封装进 **UDP → IP → 链路层（如以太网帧头）** 等结构中。读后续 `net`、抓包、调 MTU 时，脑子里要有「**一层套一层**」的图像。

## 2. 网络连接与通信模型（Connection & Communications）

### 2.1 两种「连接」心智

| 类型 | 直觉 | 典型例子 |
|------|------|----------|
| **面向连接（Connection-oriented）** | 像打电话：先建立会话，再双向传数据 | **TCP** |
| **无连接（Connectionless）** | 像寄明信片：每条报文相对独立，不保证顺序与到达语义 | **IP**、**UDP** |

**前瞻**：「可靠」**不必**永远等于「面向连接」。例如 **HTTP/3** 在**无连接的 UDP** 之上用 **QUIC** 实现加密、多路复用与可靠传输——后面读 HTTP 演进时可以把本章模型和 QUIC 对照。

### 2.2 两种常见通信抽象

- **消息传递（Message Passing）**  
  最基础：建立通道后收发**报文/字节流**，对端自行解析并回复。Socket、许多自定义二进制协议都属于这一族。

- **远程过程调用（RPC）**  
  把「跨网络」伪装成「像调本地函数」：客户端调 **stub**，由运行时/框架完成序列化、传输与错误映射，是构建大型分布式系统的常用抽象（本书后文有专章）。

## 3. 组件分布与中间件（Component Distribution）

任何网络应用大致都能拆成三块：**表示（Presentation）**、**应用逻辑（Application logic）**、**数据访问（Data access）**。三者如何落在客户端、边缘与服务器上，决定了架构形态。

书中借 **Gartner 的五种分布模型**（示意理解即可）说明「谁在哪算、数据在哪存」：

1. **分布式数据（Distributed data）**  
   例如移动端缓存部分数据，缺了再向远端拉取。
2. **远程数据（Remote data）**  
   客户端高度依赖远端存储与命名空间（如 **NFS** 一类心智）。
3. **分布式应用 / 分布式事务（Distributed application / transaction）**  
   典型现代 Web：浏览器里有前端逻辑，服务器有后端逻辑，分层分工。
4. **远程表示（Remote presentation）**  
   表示层主要在远端（如传统瘦客户端）。
5. **分布式表示（Distributed presentation）**  
   表示层在多处协作完成用户体验。

**中间件（Middleware）** 常被称为分布式系统的「胶水」：目录与发现、事务与一致性辅助、序列化、安全审计、消息路由等——这些往往不是单机 OS 自带的「全家桶」，而是架构里显式引入的一层。

## 4. 必读：分布式计算的八大谬误（Eight Fallacies）

几十年前总结的经验，**至今仍是踩坑重灾区**。写网络代码时若潜意识里假设「世界像本机一样完美」，就会在超时、断连、丢包面前反复翻车。

1. **网络是可靠的** — 链路、路由、DNS、对端进程都会失败。  
2. **延迟为零** — 与本地调用差数量级；设计若不考虑 RTT，吞吐与体验会崩。  
3. **带宽是无限的** — 冗余流量会挤占队列与成本。  
4. **网络是安全的** — 窃听与篡改客观存在，**TLS** 等机制才有存在价值。  
5. **拓扑永不改变** — 云与容器环境下 IP、路由、服务实例**动态**是常态；硬编码地址往往走不通。  
6. **只有一个管理员** — 排障常跨团队、跨组织。  
7. **传输成本为零** — 带宽、流量费、CPU 加解密都是成本。  
8. **网络是同构的** — 设备、协议栈、客户端版本各异，兼容性要显式设计。

## 5. 本章与后续章节的关系（收束）

第一章给的是「**预防针**」：分布式世界里有**崩溃、超时、丢包、部分失败**等单机程序不常假设的问题。  
因此后面在 Go 里你会反复看到：**显式 `error`、超时与取消、重试与退避、连接生命周期管理**——不是语言啰嗦，而是模型诚实。
