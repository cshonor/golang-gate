# Chapter 05 — Application-Level Protocols（应用层协议）

> 对应原书 *Network Programming with Go Language, 2nd Ed.* 第 5 章。  
> **性质**：偏**协议设计与演进**，少独立 Go API；与第 8（HTTP）、13（RPC）、14（REST）、15（WebSocket）形成「共性 → 具体协议」关系。

---

## 1. 为什么单独成章、又常被「合并进路线」？

- 解决的是**跨实现**的问题：版本、报文边界、状态放哪、错误语义——而不是某一门语言的语法。
- 读完第 3 章 Socket 后，你已经能**搬运字节**；第 5 章回答的是：**这些字节按什么规则解释、如何升级而不砸旧客户端**。

---

## 2. 协议设计与版本控制

- **向后兼容**：新字段 optional、旧客户端忽略未知字段、默认行为明确。
- **能力协商**：传输层已连上之后，应用层仍可能要「谈能力」（ALPN、子协议、feature flag）。
- **Web / HTTP 演进主线**（与第 8 章细节衔接）：0.9 → 1.0 → 1.1 → 2 → 3（QUIC）各自主要解决**延迟、队头阻塞、多路复用、加密与握手**中的哪一类痛点。

**自检**：新协议版本发布后，**旧客户端最少还能安全地做什么**？

---

## 3. 消息格式：字节视角 vs 文本视角

| 风格 | 典型手段 | 易错点 |
|------|-----------|--------|
| **面向字节** | 定长头、长度前缀、TLV | 大小端、长度上限、半包粘包 |
| **面向字符/文本** | 行协议、类 HTTP 头、`JSON` 行 | 编码（UTF-8）、换行符、注入与转义 |

与本书第 4 章「序列化」关系：**序列化解决「对象 ↔ 字节」**；本章更偏**在线路上如何成帧、如何分条消息**。

---

## 4. 状态（State）与状态机

- **无状态**：每个请求自带全部上下文（典型 REST 理想）；易水平扩展。
- **有状态连接**：会话 ID、连接内阶段（握手 → 业务 → 关闭）；要能画**状态转换图**。
- **状态放哪**：服务端内存、客户端 cookie、加密 token、或外部存储（Redis 等）——影响**伸缩与故障恢复**。

**自检**：连接断线重连后，**客户端和服务端各自认为会话进行到哪一步**？

---

## 5. 与后续章节的对照表（读完本章用来「串书」）

| 后续章 | 本章哪条「共性」会反复出现 |
|--------|---------------------------|
| 第 8 章 HTTP | 文本起始行 + 头 + 体；版本与持久连接；语义动词 |
| 第 13 章 RPC | 私有帧格式或借 HTTP；过程名/参数如何编码 |
| 第 14 章 REST | 资源 URI + 动词 + 无状态 + 表现层格式 |
| 第 15 章 WebSocket | 先 HTTP 协商再升级帧协议；心跳与关闭语义 |

---

## 6. 关键总结（背诵版）

**应用层协议 = 字节如何成帧 + 状态放哪 + 版本如何演进 + 失败如何表达。**  
实现可以换（Go/Java），这四问答不稳，后面所有「调库」都会在生产里以事故形式补考。
