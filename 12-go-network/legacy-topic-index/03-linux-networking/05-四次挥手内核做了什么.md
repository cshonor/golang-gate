# 四次挥手：内核做了什么

> **03-linux-networking · Linux 网络底层**  
> 上篇：[04-三次握手内核做了什么](./04-三次握手内核做了什么.md)；报文与语义见 [04-tcp · 四次挥手](../04-tcp/04-四次挥手.md)。

---

## 一、为什么需要「四次」？

TCP **全双工**：每个方向的字节流 **独立关闭**。一方结束发送 ≠ 另一方也必须立刻结束，因此 **FIN 与 ACK** 常呈现 **四次报文** 的经典形态（存在 **延合并** 时可能看到 **三次** 外观，属优化，面试仍以四步逻辑为准）。

---

## 二、经典四次（主动关闭方 A vs 被动 B）

1. **A → FIN**：A 不再发送应用数据（可仍读）。  
2. **B → ACK**：确认 FIN。  
3. **B → FIN**：B 也结束发送。  
4. **A → ACK**：确认 FIN；随后 A 进入 **`TIME_WAIT`**（主动关闭方典型），持续 **2MSL** 量级后释放（具体以内核实现与套接字选项为准）。

---

## 三、核心状态（服务端/客户端均可扮演不同角色）

| 状态 | 直觉 |
|------|------|
| **`FIN_WAIT_1`** | 已发 FIN，等对端 ACK |
| **`FIN_WAIT_2`** | 对端已 ACK FIN，等对端 FIN |
| **`CLOSE_WAIT`** | 收到对端 FIN，**本端应用尚未 close** —— **泄漏排查高频** |
| **`LAST_ACK`** | 本端发 FIN 后等对端 ACK |
| **`TIME_WAIT`** | 主动关方等待 **2MSL**，防旧连接迟滞报文干扰新连接 |
| **`CLOSING`** | 双方几乎同时 FIN 时的短暂态 |

全图衔接：[06-TCP连接状态](./06-TCP连接状态.md)。

---

## 四、`shutdown` vs `close`（工程必辨）

- **`shutdown(SHUT_WR)`**：常只关闭 **写半部**，仍可读；用于 **半关闭** 协议（如部分 HTTP 场景）。  
- **`close(fd)`**：递减引用，**读写均结束** 后释放 fd；与 **`SO_LINGER`** 组合影响 **`RST`/`FIN`** 行为（慎用暴力 `RST`）。

Go：**`TCPConn.Close`** 接近 **`close`**；需要半关闭时看 **`CloseRead`/`CloseWrite`**（存在性与语义以文档为准）。

---

## 五、`RST` 常见触发（现象层）

- 向 **无监听端口** 写数据 → 对端 **`RST`**。  
- **连接已销毁** 仍 `write`。  
- **`SO_LINGER(0)`** 类行为导致快速丢弃。  

与错误处理衔接：`ECONNRESET` 等见 **06-go-net-internals** 错误篇。

---

## 六、极简总结

- **四次挥手** = **双向发送通道**各自礼貌关闭。  
- **`CLOSE_WAIT` 多** = 本端 **未及时 `close`** 或逻辑卡死。  
- **`TIME_WAIT` 多** = 主动关连接频繁或短连接风暴 → 调优见 **`12-tuning-and-issues`**。
