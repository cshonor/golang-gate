# socket 缓冲区：SO_RCVBUF / SO_SNDBUF

## 一、每个 TCP 连接有两块内核缓冲

- **接收缓冲（receive buffer）**：对端发来的字节进入 **内核 TCP 接收队列**；应用 `read` 从这里 **拷贝到用户空间**。  
- **发送缓冲（send buffer）**：应用 `write` 将数据拷入 **内核发送队列**，由协议栈 **异步** 推向网卡。

**`SO_RCVBUF` / `SO_SNDBUF`**：用 `setsockopt` **建议**内核缓冲大小；**实际生效值**常被内核 **翻倍/对齐/夹紧**（务必 **`getsockopt` 验证**）。

---

## 二、与应用行为的关系

### 1. 接收侧

- 应用 **读得慢** → 接收缓冲 **堆积满** → 窗口缩小 → 对端 **`write` 阻塞或返回 `EWOULDBLOCK`**（非阻塞）→ **背压** 传播到对端。

### 2. 发送侧

- **高带宽 × 高 RTT**（长肥管道）需要 **足够窗口与缓冲** 才能「灌满管道」，否则 **吞吐上不去**。  
- 盲目调大缓冲 → **内存压力**（**并发连接数 × 每连接缓冲**）。

---

## 三、与 BDP（带宽时延积）的心智

**BDP ≈ 带宽 × RTT**，表示「在途数据」量级。缓冲/窗口若长期 **远小于 BDP**，则 **无法吃满带宽**；过大则 **浪费内存** 且对 **短连接** 无益。

---

## 四、Go 中常用入口

- **`TCPConn.SetReadBuffer` / `SetWriteBuffer`**：对应 **SO_RCVBUF/SO_SNDBUF** 的建议值。  
- **`net.ListenConfig` / `Dialer`/`Control`**：在 **`syscall.RawConn`** 上 `setsockopt` 做更底层控制。  
- **调优必须与测量结合**：`ss -m`、**`iperf`**、业务 pprof。

---

## 五、极简总结

- **内核双缓冲** 是 **TCP 流控与吞吐** 的核心容器。  
- **`setsockopt` 只是建议值**；以 **`getsockopt`/实测** 为准。  
- **背压** 会从 **慢消费者** 一路传到 **对端写路径**。
