# 阻塞与非阻塞

> **01-io-fundamentals · IO 核心理论**  
> 前置见 [01-什么是 IO](./01-什么是IO.md)。**阻塞 / 非阻塞**只回答一件事：**调用这一刻若数据未就绪（或暂时不可写），内核要不要把调用线程挂起等到就绪为止。**  
> 进阶地图：[五大 IO 模型总览](../02-io-models/01-五大IO模型总览.md)。

---

## 一、阻塞 IO（系统默认的常见形态）

### 1. 核心行为

当应用调用 `read` / `recv` / `accept` / `write` / `send` 等（在**未**设置 `O_NONBLOCK` 的前提下）：

- **条件已满足**（例如接收缓冲区已有数据、发送缓冲区有空间）：内核完成拷贝，**正常返回**。
- **条件不满足**（无数据可读、对端未连接可 `accept`、发送缓冲满等）：内核通常把**当前线程**置为睡眠，**让出 CPU**，直到事件就绪再唤醒，继续把本次调用走完。

也就是说：**线程在「一次系统调用」上可能长时间不返回**，但也不再消耗 CPU 自旋。

### 2. 关键特点

| 优点 | 代价 |
|------|------|
| 代码路径直：**一行调用，等完再往下执行** | **一个线程同一时刻只能堵在一个 fd 上** |
| 等待逻辑全在内核，用户态不必写重试循环 | 高并发时常态是「一线程一连接」，**线程数与栈内存、上下文切换**压力陡增 |

### 3. 直观例子

单线程 TCP 服务里连续 `accept()`：若没有新连接，线程会**一直阻塞在 `accept`**，不占 CPU，直到有客户端连上才被唤醒继续执行。

## 二、非阻塞 IO（`O_NONBLOCK`）

### 1. 核心行为

对 fd 设置 **`O_NONBLOCK`**（常见手段：`fcntl(fd, F_SETFL, fcntl(fd, F_GETFL) | O_NONBLOCK)`）后，同类 IO 调用语义变为：

- **能立刻完成**：与阻塞模式一样，读写成功并返回。
- **不能立刻完成**：**不**把线程挂起在「等数据」上，而是**立即返回**，典型错误码 **`EAGAIN` / `EWOULDBLOCK`**（具体以平台与接口为准）。

本质：**内核把「等就绪」的责任交回用户态**；线程不会因为这一次 `read` 而进入可中断睡眠（但仍可能在别的系统调用上阻塞）。

### 2. 关键特点

| 优点 | 代价 / 风险 |
|------|----------------|
| 单次调用**不卡死线程**，便于**单线程轮询大量 fd** | 必须在用户态处理 **重试、错误码、状态机** |
| 为高并发 + **IO 多路复用**（`select` / `poll` / `epoll` 等）打底 | **裸忙轮询**会空转 CPU，生产上**几乎从不单独使用** |

### 3. 典型落地

**非阻塞 fd + IO 多路复用**：用复用器**集中阻塞在「多路之一就绪」**上，而不是每个 fd 各占一个线程阻塞。详见本仓库 **`02-io-models`**、后续 **`07-go-netpoll`** 等篇。

## 三、高频易错：阻塞 ≠ 同步（二者正交）

- **阻塞 / 非阻塞**：描述的是 **调用发起后，当前线程是否会在该调用上睡眠等待就绪**。  
- **同步 / 异步**：描述的是 **由谁完成数据搬运、完成如何被感知**（调用方自己收尾 vs 内核/系统事后通知）。  

二者**正交**，面试要能口述常见组合（不必死记名称，能说清「谁等、谁通知、是否卡线程」即可）：

| 组合 | 一句话 |
|------|--------|
| **同步 + 阻塞** | 最常见默认路径：调用卡住直到本次 IO 完成，调用方自己读结果。 |
| **同步 + 非阻塞** | 立即返回；未就绪则 `EAGAIN`，用户态轮询或由复用器驱动重试。 |
| **异步 + 阻塞** | 发起后线程仍可能在「等完成通知」上阻塞（相对少见，具体看 API 语义）。 |
| **异步 + 非阻塞** | 发起即返回；完成通过回调/完成队列/信号等通知（高性能框架常见底层形态之一）。 |

展开与纠偏见：[03-同步与异步](./03-同步与异步.md)。

## 四、常见 API 在非阻塞下的表现（速查）

### 1. 如何设为非阻塞

- Unix：**`fcntl`** 对 fd 置 `O_NONBLOCK`（与 `open` 时带 `O_NONBLOCK` 等价类效果，细节查平台文档）。

### 2. 返回规则（典型）

| 调用 | 未就绪 / 暂不可写时（非阻塞） |
|------|-------------------------------|
| `read` / `recv` | 常立即返回 **`-1`**，`errno` 为 **`EAGAIN`/`EWOULDBLOCK`** |
| `write` / `send` | 发送缓冲满时同上 |
| `accept` | 无新连接时同上 |
| `connect`（TCP） | 可能**立即返回**「进行中」类结果，后续用 **`SO_ERROR` / 可写事件** 判断是否真正连上（与多路复用配合） |

## 五、与 Go 的衔接：为什么 `conn.Read`「看起来像阻塞」？

日常写 Go：`conn.Read(buf)` 一行写完，语义上像**同步阻塞**；但高并发下不会像「一线程一阻塞」那样拖垮 OS 线程。

要点（剧透，细节在 **`07-go-netpoll`** 与 **`07-GMP and channel`**）：

- 底层 socket 往往以 **非阻塞** 方式配合 **netpoll**（`epoll` / `kqueue` / `IOCP` 等）工作。  
- **数据未就绪**：通常**不阻塞 M 去空转等这一个 fd**，而是把当前 **G** 挂起，调度器去跑别的 G。  
- **就绪后**：runtime 把对应 G 重新置为可运行，继续你的 `Read` 逻辑。

## 六、极简背诵

1. **阻塞**：未就绪则**挂起线程**，内核代等，写法简单，**并发连接成本高**。  
2. **非阻塞**：未就绪则**立刻 `EAGAIN` 类返回**，**用户态**负责重试策略；生产上常配 **多路复用**，忌裸忙等。  
3. **阻塞 ⟂ 同步**：一个看**卡不卡线程**，一个看**谁收尾 / 谁通知完成**。  
4. **Go**：上层同步阻塞式 API，底层常见是 **非阻塞 fd + netpoll + GMP**，把「等」从 OS 线程挪到 **G 的调度**上。
