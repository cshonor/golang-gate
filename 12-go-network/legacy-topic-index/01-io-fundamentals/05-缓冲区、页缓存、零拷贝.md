# 缓冲区、页缓存、零拷贝

> **01-io-fundamentals · IO 核心理论**  
> 前置：[04-用户态与内核态](./04-用户态与内核态.md)。本篇把 **「数据暂存在哪」** 与 **「少拷几次」** 讲清；与 Linux 章 [SO_RCVBUF/SO_SNDBUF](../03-linux-networking/08-socket缓冲区SO_RCVBUF与SO_SNDBUF.md) 可交叉读。

---

## 一、缓冲区（Buffer）

### 1. 为什么需要？

CPU 与内存远快于磁盘、广域网。**缓冲**把快慢两端解耦：内核/协议栈先攒一截，再一次性拷给用户或发给网卡，减少 **等包 / 等扇区** 次数与 syscall 税。

### 2. 网络 IO：内核 Socket 缓冲

每个 TCP socket 在内核侧有大致两类队列心智（实现细节因内核版本而异）：

- **接收缓冲**：对端发来的字节先进内核接收路径；你的 `read` 从这里拷到用户 `[]byte`。可调 **`SO_RCVBUF`**（生效方式见平台文档）。  
- **发送缓冲**：`write` 把用户数据拷入内核发送队列，由协议栈与驱动异步推到网卡；可调 **`SO_SNDBUF`**。

**`conn.Read` / `conn.Write`**：一次成功读写，至少包含 **「用户缓冲 ↔ 内核 socket 缓冲」** 这一段 CPU 拷贝（除非走特殊零拷贝路径）。

### 3. 用户态缓冲

应用里的 `make([]byte, 4096)`、`bufio.Reader` 等：减少 **小 `Read` 次数**、做协议解析、对齐业务批量——**不是**替代内核缓冲，而是叠一层策略。

### 4. 文件 IO：stdio / bufio

C `stdio`、Go `bufio` 等在用户态再缓冲，**减少 `read`/`write` syscall 次数**；与内核 **页缓存**（下一节）是不同层次。

---

## 二、页缓存（Page Cache）

### 1. 是什么？

Linux 上普通文件读写的热路径里，文件内容常缓在 **页缓存**：把「磁盘上的页」映射为内存中的页，命中则 **读内存而非读盘**。

### 2. 读路径（概念）

1. `read` 进内核。  
2. 若页缓存命中：从页缓存 **拷贝** 到用户缓冲。  
3. 未命中：块层读盘填充页缓存，再拷到用户缓冲（可能阻塞在 IO 上）。

### 3. 写路径（概念）

`write` 常先写入页缓存并标记脏页，由 **`pdflush`/回写线程** 异步刷盘；`fsync`/`fdatasync` 才更强约束落盘语义。

### 4. 特点

- **快**：把随机读变成「读内存」。  
- **占内存**：缓存与进程 RSS、回收策略相互影响。  
- **掉电 / crash**：未刷盘数据可能丢失——持久化要 **`Sync`** 或数据库 WAL 语义。

## 三、零拷贝（Zero-Copy，工程语义）

**目标**：减少 **「用户态 ↔ 内核态」** 之间不必要的 **CPU 搬运** 与 **陷入次数**；**不是**数学上绝对「0 次数据 Touch」，而是 **「用户不参与中间缓冲」** 这类优化统称。

### 1. 传统「读文件再写 socket」的痛点（示意）

典型 `read` + `write` 路径里，数据可能经历：**磁盘 → 内核页缓存 → 用户缓冲 → 内核 socket 缓冲 → 网卡**（其中两段常是 **CPU `memcpy`**，具体因实现与优化而异）。瓶颈常在 **CPU 拷贝 + 用户态参与次数**。

### 2. 常见技术（Linux 心智）

| 手段 | 直觉 | 备注 |
|------|------|------|
| **`mmap` + `write`** | 用户通过映射直接触达页缓存映射区，减少「拷进用户临时大缓冲」一类路径 | 仍可能有协议栈侧拷贝，需 profiling |
| **`sendfile`** | 内核在 **页缓存 ↔ socket** 之间走更短路径，**用户态不传中转缓冲** | `net`/`http` 在合适条件下会用（见下） |
| **带 Scatter-Gather DMA 的网卡** | 进一步降低 **CPU 参与搬运** 的比例 | 依赖硬件/驱动/内核路径 |
| **`splice` 等** | 管道与 socket 间 **fd 缓冲指针传递** 类优化 | 使用场景更窄 |

### 3. `io_uring`

Linux **5.1+** 起逐步成熟的 **异步接口族**，可覆盖磁盘与网络等；**是否零拷贝、是否比你当前 `epoll+read` 更快**，取决于内核版本、操作类型与基准，**以压测为准**。

### 4. 与 Go 的关系

- **`net/http` 服务静态文件**：在类型与系统支持满足时，常见路径会走 **`sendfile`** 类优化（把「文件 → TCP」尽量留在内核侧）。  
- **`io.Copy` / `ReadFrom` / `WriteTo`**：在源与目标类型支持时，可能选用 **splice/sendfile** 等快捷路径（仍读源码与 `go test`/pprof 验证）。  
- **手写零拷贝**：可用 `syscall`/`x/sys` 谨慎调用，注意 **可移植性、TLS、中间人缓冲** 等现实约束。

---

## 四、极简总结

- **Socket 双缓冲**：收发在内核排队；`Read`/`Write` 是跨边界拷贝。  
- **页缓存**：文件热数据在内存；**命中快、未命中读盘**。  
- **零拷贝**：减少 **用户参与 + CPU  memcpy**；**sendfile** 是经典代表；**`io_uring`** 是演进方向之一。  
- **调大 buffer 不总更快**：内存、延迟、队头阻塞要一起算。
