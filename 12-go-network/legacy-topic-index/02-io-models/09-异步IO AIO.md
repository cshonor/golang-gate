# 异步 IO（AIO）与 `io_uring`

> **02-io-models · IO 模型全解**  
> 上篇：[08-信号驱动IO](./08-信号驱动IO.md) · 前置：[03-同步与异步](../01-io-fundamentals/03-同步与异步.md)

---

## 一、核心定义（教材语义）

**异步 I/O**：应用提交请求后**立即返回**；**等就绪 + 拷贝到用户缓冲** 尽量在**内核后台**完成；完成后再通过 **事件 / 回调 / 完成队列** 通知应用——应用 **`read` 返回时数据已在缓冲里`** 的那类模型（与「只通知就绪、仍由你 `read`」的 **多路复用 + 同步读** 对照）。

> 具体 API（POSIX AIO、`io_uring`、Windows **IOCP**）语义不同，**以各平台文档为准**。

---

## 二、特点

### 优点

- **线程不必卡在等 + 拷的同步路径上**（理想情况下）。  
- **高吞吐、低延迟** 场景有吸引力（尤其 **`io_uring` / IOCP** 生态演进方向）。

### 缺点

- **编程与调试复杂**：生命周期、缓冲所有权、完成顺序、背压。  
- **依赖内核与库成熟度**。  
- **与现有同步代码混用** 要小心线程模型与内存序。

---

## 三、Linux 侧常见分层

### 1. glibc POSIX AIO（`aio_read` 等）

- 不少资料指出：**部分路径**用线程池/worker **模拟**，**不总等于「纯内核异步网卡到用户缓冲」**。  
- **高并发 socket** 场景，**传统上仍以 `epoll` 为主流叙述**。

### 2. `io_uring`（Linux 5.1+ 起逐步可用）

- **内核提供的环形提交/完成队列**，可覆盖 **磁盘 + 网络** 等（能力随内核版本扩展）。  
- 常被称作 **Linux 异步 I/O 的重要演进方向**；**是否在你的 workload 上碾压 `epoll`**，要用基准与内核版本说话。

---

## 四、与多路复用（`epoll`）对照（口述版）

| 项 | `epoll` + 非阻塞 `read` | 真异步（理想） |
|----|-------------------------|----------------|
| 就绪通知 | 有 | 可有 |
| 谁把数据放进用户缓冲 | 常见：`read` 里拷 | 内核完成再通知 |
| 心智模型 | Reactor 为主 | Proactor 色彩更浓 |
| Go 现状（概括） | **`net`/`netpoll` 主流路径** | **生态与 runtime 持续演进**，以 **Go release note / 构建标签** 为准 |

**不要背死「Go 某版本已全面切 `io_uring`」**：以你使用的 **Go 版本文档与实验特性说明** 为准。

---

## 五、极简总结

- **异步**：**提交 ≠ 完成**；完成靠**完成通知**。  
- **POSIX AIO** 与 **`io_uring`** 别混为一谈。  
- **Go**：写业务仍以 **`net` + `epoll`/`kqueue` 心智** 为主；关注 **`io_uring`/`sendfile` 等** 的演进即可。
