# 阻塞 IO（BIO）

> **02-io-models · IO 模型全解**  
> 上篇：[01-五大IO模型总览](./01-五大IO模型总览.md) · 前置：[02-阻塞与非阻塞](../01-io-fundamentals/02-阻塞与非阻塞.md)

---

## 一、核心定义

**阻塞 IO（Blocking IO）**：进程对 fd 发起读（如 `recv` / `read`）时，若**接收缓冲无数据**，内核让该调用**阻塞**（线程进入睡眠），直到**至少有一些数据可读**并完成从内核空间到用户缓冲的拷贝后，调用才返回。

### 以网络读为例的典型流程

1. 应用调用 `recv` / `read`。  
2. 内核检查 socket 接收缓冲：无数据 → **阻塞当前线程**。  
3. 对端数据到达 → 数据进内核缓冲。  
4. 内核把数据拷到用户缓冲。  
5. 调用返回，应用继续。

---

## 二、特点

### 优点

- **编程简单**：「调一次就等到有结果」，少分支。  
- **阻塞期不占 CPU**：线程睡眠，不空转。  
- **单次 syscall 语义清晰**：一次调用完成「等 + 读」。

### 缺点

- **伸缩性差**：**一线程一连接**时，连接数上去后线程、栈、调度成本爆炸。  
- **大量时间阻塞在等 IO**：线程资源利用率低（从「线程数」视角）。  
- **不适合万级并发**：与事件驱动模型对比明显。

---

## 三、典型场景

- 教学、脚本级小服务、逻辑极简单的阻塞式 demo。  
- 与「线程池 + 短连接」结合时仍可见，但高并发网关一般不以此为主模型。

---

## 四、和 Go 的关系

**标准库 `net` 的 `Conn.Read` 会「阻塞」**——阻塞的是 **goroutine**，底层 fd 多为 **非阻塞**，由 **`netpoll` + epoll/kqueue** 等在就绪时唤醒 G。因此：**体验像 BIO，实现不是「一线程卡死在 `recv` 上」那种经典 BIO 线程模型**。

---

## 五、极简总结

- BIO：**等就绪 + 拷贝**在同一次阻塞调用里完成。  
- 简单、不占 CPU 空转；**高并发下一连接一线程**是硬伤。  
- Go：**G 级阻塞 + 多路复用**，别和「OS 线程阻塞在 `recv`」混为一谈。
