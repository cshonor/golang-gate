# IO 多路复用：`select`

> **02-io-models · IO 模型全解**  
> 上篇：[03-非阻塞IO NIO](./03-非阻塞IO%20NIO.md)

---

## 一、核心定义

**IO 多路复用**：把「等多个 fd 就绪」合并到 **一次阻塞调用**（如 `select`），返回后再对每个就绪 fd 做 `read`/`write`。这样避免 **NIO 裸轮询** 的 CPU 浪费。

`select` 是 POSIX 上最老、最广为人知的多路复用接口之一，适合理解 **「fd 集合 + 超时 + 返回后用户遍历」** 这一范式。

### 典型流程

1. 用 `fd_set` 组织关心的读/写/异常集合。  
2. 调用 `select` → 线程阻塞，内核检查这些 fd。  
3. 至少一个就绪（或超时）→ `select` 返回。  
4. 用户遍历 fd，对就绪者发起实际 IO。  
5. 循环。

---

## 二、特点

### 优点

- **单线程可盯多个 fd**，比「一线程一连接」省线程。  
- **POSIX 常见**，跨平台认知度高。  
- **阻塞在 `select` 上**，无 NIO 那种无脑空转（仍可能有别的设计上的忙等，那是应用层问题）。

### 缺点

- **`nfds` 与 `fd_set` 规模**：可监视的 fd **个数**常受 **`FD_SETSIZE`** 等编译期/实现限制（教材常说 1024，**以目标平台 `select` 文档为准**）。  
- **每次调用常需把 fd 集合在用户态与内核态之间传递**，fd 多时成本高。  
- **就绪后仍需用户遍历** 找出谁就绪（与 `poll` 类似痛点）。  
- 大 fd 值时，**遍历上限到 maxfd** 也可能浪费。

---

## 三、API 备忘（C）

```c
int select(int nfds, fd_set *readfds, fd_set *writefds,
           fd_set *exceptfds, struct timeval *timeout);
```

- `nfds`：一般为 **最大 fd + 1**（用于界定检查范围）。  
- `timeout`：`NULL` 表示无限等；`{0,0}` 表示轮询式探测（仍要付 `select` 成本）。

---

## 四、典型场景

- 小规模、兼容性要求高的代码路径。  
- 读懂 **为什么需要 `poll`/`epoll`** 的台阶。

---

## 五、极简总结

- `select`：**一次等多个 fd**，返回后应用遍历处理。  
- **优点**：模型简单、POSIX；**缺点**：fd 规模与拷贝/遍历成本。  
- 高并发 Linux 服务多迁移到 **`poll` / `epoll`**。
