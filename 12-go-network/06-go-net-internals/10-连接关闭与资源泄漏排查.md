---

## 1. `Close()` 大致做什么（概念）

1. **停止**后续在该 `netFD` 上的正常读写路径。  
2. 从 **netpoll / poll** 上**注销**兴趣，避免已关闭 fd 仍被唤醒。  
3. **关闭**底层 socket / fd，释放内核资源。  
4. 唤醒可能仍阻塞在 **`pollWait`** 上的等待方，使其带着 **`closed` / `use of closed network connection`** 一类错误返回。

> 具体顺序与错误文案以版本为准；抓主线：**先脱离多路复用，再关 fd，再让等的人失败返回**。

---

## 2. 常见泄漏形态

| 原因 | 表现 |
|------|------|
| **Goroutine 永久阻塞在读** | 对端不发、本端不设 deadline、也不关 conn |
| **未 `Close` 即丢引用** | `TIME_WAIT` / `CLOSE_WAIT` 堆积（视角色与协议） |
| **只关一半** | 读协程退出，写协程仍在写已关连接 → 错误风暴 |
| **HTTP Body 未读完即复用连接** | Keep-Alive 下连接状态异常，见 `09-http-internals` 相关篇 |

## 3. 排查工具箱

- **`netstat` / `ss`**：看状态分布、`Recv-Q`/`Send-Q`。  
- **`lsof -p <pid>`**：进程打开了哪些 fd。  
- **Go `pprof` goroutine / block profile**：看谁卡在 `pollWait`、`Read`。  
- **`runtime/trace`**：调度与网络事件时间线。

与 **调优章** 交叉阅读：`12-tuning-and-issues` 下 TIME_WAIT、内存等篇。

## 4. 与 07 章闭环

关闭路径必须让 **netpoll 侧**与 **G 等待队列**一致，避免「fd 已关仍被唤醒」或「永远不醒」，见 [10-netpoll常见坑与优化](../07-go-netpoll/10-netpoll常见坑与优化.md)。
