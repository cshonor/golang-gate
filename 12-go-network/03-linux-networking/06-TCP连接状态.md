# TCP 连接状态

## 一、为什么背状态机？

线上 **`netstat`/`ss`**、抓包、防火墙 **conntrack** 都围绕 **TCP 状态**。能 **口述状态迁移** + **知道异常态含义**，排障速度差一个数量级。

---

## 二、状态总览（复习表）

| 状态 | 典型含义 |
|------|----------|
| **`CLOSED`** | 无连接 |
| **`LISTEN`** | 服务端监听 |
| **`SYN_SENT`** | 客户端已发 SYN |
| **`SYN_RCVD`** | 服务端收到 SYN，已回 SYN-ACK |
| **`ESTABLISHED`** | 已建立，可传数据 |
| **`FIN_WAIT_1`** | 主动关：已发 FIN |
| **`FIN_WAIT_2`** | 主动关：对端已 ACK FIN，等对端 FIN |
| **`CLOSE_WAIT`** | 被动关：收到 FIN，**等本端应用 close** |
| **`CLOSING`** | 双方几乎同时关闭 |
| **`LAST_ACK`** | 被动关：已发 FIN，等 ACK |
| **`TIME_WAIT`** | 主动关：等 2MSL |

---

## 三、客户端 vs 服务端视角

- **客户端主动连**：`SYN_SENT` → `ESTABLISHED`。  
- **服务端**：`LISTEN` → `SYN_RCVD` → `ESTABLISHED`。  
- **关闭**：谁主动 **发 FIN**，谁更可能经历 **`FIN_WAIT_*` / `TIME_WAIT`**（一般规律，具体仍看方向）。

---

## 四、异常与调试

- **`SYN_SENT` 堆积**：连不上、DNS/路由/防火墙问题。  
- **`SYN_RCVD` 暴涨**：SYN flood 或半连接队列异常。  
- **`CLOSE_WAIT` 暴涨**：**应用未 close** 或阻塞在业务。  
- **`TIME_WAIT` 暴涨**：短连接、高 QPS、主动关多 → 调内核参数与架构（见第 12 章）。

命令示例（因系统而异）：

```bash
ss -tan state close-wait
ss -tan state time-wait | wc -l
```

## 六、极简总结

- **状态机** = TCP 可靠性的「交通规则」。  
- **`ESTABLISHED`** 才稳传数据；**`CLOSE_WAIT`/`TIME_WAIT`** 是线上两大常客。
