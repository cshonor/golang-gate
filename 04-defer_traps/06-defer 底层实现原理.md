# 06 - defer 底层实现原理

## 1. 你要知道的实现事实

- `defer` 不是“零成本”
- 高频路径里大量 defer 会有额外开销
- Go 新版本对 defer 做了优化（open-coded defer 等）

## 2. 对业务开发的意义

- 普通函数：优先可读性，放心用 defer
- 极端热点函数：可 benchmark 后考虑显式释放

## 3. 实战建议

- 先写对，再测性能
- 不要为了“可能更快”牺牲正确性和可维护性

---

## 4. runtime 视角（速记）

- 每个 goroutine 维护 **defer 链表**：新注册的 `defer` 插在表头，函数返回时按链表**从表头依次执行**，因此表现为 **后进先出（LIFO）**。
- 与 `defer` 相关的记录可理解为挂在栈上的 **`_defer` 结构**（保存调用点、函数指针、参数副本等）；具体字段随 Go 版本演变，以 `src/runtime/runtime2.go` 及发行说明为准。
- **open-coded defer（开放编码）**：编译器对部分简单 `defer`（如固定调用、无复杂参数）做**内联式**生成，减少链表分配与运行时开销；因此「defer 一定很慢」不成立，但**大量复杂 defer** 仍可能可观。
- 这与 [02](./02-defer%20执行顺序规则.md)、[04](./04-defer%20与%20return%20执行流程.md) 中的语义一致：**先搞清语义与正确性，再谈优化**。
