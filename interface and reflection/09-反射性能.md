# 反射的性能（为什么慢？何时用？）

## 1）反射为什么性能差？

1. **运行时才解析类型**  
   普通代码：编译期确定类型，便于内联、逃逸分析、死代码消除。  
   反射：运行时查 `_type`、字段/方法表、名字字符串，**动态路径多**。

2. **难以做编译期优化**  
   编译器不知道你最终访问哪个字段、调哪个方法，优化空间小。

3. **更多堆分配与 GC 压力**  
   `reflect.Value`、`reflect.Type` 等对象常在堆上分配；配合接口、字符串，**分配与 GC 开销**明显高于直接代码。

4. **多层间接**  
   典型路径：接口 → `eface`/`iface` → 元数据 → 实际数据，**跳转多、缓存不友好**。

## 2）性能差距大概多少？（量级，勿背死数字）

- 简单字段读写：反射通常比直接访问**慢一个数量级以上**很常见。
- 遍历结构体、序列化框架路径：**慢几十倍**不奇怪。  
  **以 `benchmark` 与业务场景为准。**

## 3）什么时候能用？

- **框架 / 通用库**：JSON/XML、`encoding/json`、ORM、依赖注入、配置映射、泛型不足时的兜底。
- **热路径业务逻辑**：能不用就不用；优先具体类型、泛型、生成代码（`go generate`）。

## 4）面试标准句

> 反射在运行时做类型解析与派发，绕过编译期优化，间接寻址多、堆分配多，所以慢；适合框架与通用能力，业务热点路径应少用。

## 复习速记

| 慢因 | 关键词 |
|------|--------|
| 运行时 | 查 `_type`、名字、表 |
| 优化 | 难内联、难专项优化 |
| 分配 | `Value`/`Type`、接口转换 |
