# 锁实现 2：自旋锁（spin lock）

> 课表：**锁实现原理 · 基础版** — 自旋锁

## 1. 行为

- 获取锁失败时 **不睡眠**，在 CPU 上 **循环重试**（常配合 `atomic` / CAS）。  

- 核心逻辑：获取锁失败时不睡眠、不阻塞，在CPU上循环重试（常配合 atomic/CAS 实现）
​
- 优势：锁持有时间极短时，可避免线程阻塞、上下文切换的开销，性能远高于阻塞锁
​
- 本质：用「CPU空转」换「无阻塞」，是典型的空间换时间优化

## 2. 典型实现直觉


// 伪代码：Go 自旋锁核心实现

for !atomic.CompareAndSwapUint32(&lock, 0, 1) {

    // 自旋优化：空转或PAUSE指令，降低CPU功耗、减少流水线冲刷

    runtime.Gosched() // 或 CPU PAUSE 指令（如 x86 的 pause）

}

// 释放锁：原子写回0，需保证内存序正确
atomic.StoreUint32(&lock, 0)


- **释放**：`store(lock, 0)` 或原子写回（需与架构内存序一致）。

- 关键细节：自旋时不能死循环空转，必须加 PAUSE / Gosched() ，否则会占满CPU、影响调度

runtime.Gosched()  是 Go 标准库  runtime  包提供的函数，作用是：
主动让出当前 goroutine 占用的 CPU（逻辑处理器 P），让调度器切换到其他 goroutine 执行。
 
- 它不会阻塞当前 goroutine，只是主动放弃时间片
​
- 调度器会将当前 goroutine 放到队列末尾，下次调度再执行
​
- 本质是主动触发一次协程调度

​
- 内存序要求：释放锁的写操作必须带正确的内存屏障，保证临界区代码对其他线程可见

## 3. 适用场景

- 临界区 **极短**、竞争 **不极端**（内核短路径、部分 runtime 内部）。  
- 用户态业务层长时间自旋通常 **不划算**。

## 4. 自检

一.自旋锁在 **单核** 上为何容易出问题（持锁者无法运行）？  

核心原因：
 
1. 单核CPU同一时间只能运行一个线程
​
- 自旋线程会占满CPU时间片，持续循环抢锁
​
- 持锁线程无法获得CPU时间，永远无法释放锁
​
- 最终导致死锁（自旋线程永远抢不到，持锁线程永远跑不完）

​
2. 多核场景无此问题
​
- 自旋线程在一个核心空转，持锁线程在另一个核心正常执行
​
- 持锁线程释放锁后，自旋线程可立即获取




二. Go 的 `sync.Mutex` 为何会 **先自旋再阻塞**（与课表后几节衔接）？
核心设计思想：快路径+慢路径结合，兼顾性能与CPU效率
 
1. 快路径（自旋）
​
- 抢锁时先尝试CAS+短时间自旋（Go 1.x 中默认自旋4次）
​
- 若自旋期间锁被释放，直接获取锁，避免内核态阻塞、上下文切换
​
- 适合锁竞争不激烈、临界区极短的场景，性能拉满
​
2. 慢路径（阻塞）
​
- 自旋失败后，进入内核态，通过信号量将goroutine挂起、让出CPU
​
- 避免长时间自旋浪费CPU，让其他goroutine正常运行
​
- 适合锁竞争激烈、临界区长的场景，保证系统整体调度效率
 
补充拓展：
 
- Go  sync.Mutex  的自旋是自适应的：会根据CPU核心数、锁竞争情况动态调整自旋次数
​
- 这就是你之前学的「CAS是零件，Mutex是CAS+阻塞组成的成品锁」的完美体现
​
- 业务层绝对不要手写自旋锁，直接用 sync.Mutex ，它已经做了最优适
