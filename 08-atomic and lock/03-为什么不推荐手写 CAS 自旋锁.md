# 为什么不推荐手写 CAS 自旋锁

> Go 调度器视角：**死循环 CAS = 自旋锁**，容易霸占 P、饿死同 P 上其他 goroutine，与 GMP 协作模型冲突。

## 1. 手写 CAS + 死循环 = 自旋锁

典型形态：

```go
for !atomic.CompareAndSwapUint32(&flag, 0, 1) {
    // 空转，不让出
}
// 临界区
atomic.StoreUint32(&flag, 0)
```

在循环里**一直重试 CAS**、**不调用调度让出**，就是用户态自旋锁。

## 2. 后果：不让出 CPU，霸占当前 P

- Go 里 goroutine 跑在某个 **M** 上，M 绑定 **P**。
- 自旋 goroutine **占着 M 不放手**，同 P 上其他就绪 G **长时间得不到执行**。
- 若多个 goroutine 在不同 P 上互相抢同一把「自旋锁」，还可能造成 **CPU 打满** 而业务无进展。

## 3. 危害（和 GMP 的关系）

- **破坏调度公平性**：等锁的 G 可能「饿死」。
- **与 Mutex 设计目标相反**：官方 Mutex 在争用时会走向**阻塞/排队**，把 CPU 让出来。
- 线上表现：延迟尖刺、P 上排队、CPU 高但吞吐不涨。

## 4. 结论（工程与面试）

- **业务与绝大多数库代码：用 `sync.Mutex`，不要手写 CAS 自旋锁。**
- 需要极致无锁时：用 **`sync/atomic` 的官方 API** 做**计数器、状态位**等，且避免无限自旋；复杂场景仍考虑 Mutex 或无锁数据结构专著。
