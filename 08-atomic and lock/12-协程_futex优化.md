# 协程与 futex 优化（用户态快路径 + 内核等待）

> 课表：**锁实现原理 · 基础版** — 协程 · futex 优化  
> （以 Linux **futex** 为典型；课程常用它讲「为什么 mutex 能又快又省」）

## 1. futex 解决什么

- **无竞争**：在用户空间用原子变量完成 **加锁/解锁**，**不 syscall**。  
- **有竞争**：只在需要睡眠/唤醒时 **进入内核**，由内核维护等待队列。

## 2. 直觉流程（简化）

1. 快路径：`CAS` 把 futex word 从 0 → 1，成功 → 获得锁。  
2. 失败：可能 **自旋几次**，再尝试。  
3. 仍失败：`futex(FUTEX_WAIT)` **挂起**（内核记录「我在等这个地址」）。  
4. 解锁：原子置 0，若有等待者 → `futex(FUTEX_WAKE)` **唤醒**。

## 3. 与「协程」在课里的联系

- **OS 线程** 的 pthread mutex 常基于 futex。  
- **Go**：M 仍跑在 OS 线程上；`sync.Mutex` 等与 runtime、调度器配合（不等价于「Go 直接用 futex 当 G」）。理解 **用户态快路径 + 内核慢路径** 有助于理解 **为何不要在高竞争下狂抢一把大锁**。
