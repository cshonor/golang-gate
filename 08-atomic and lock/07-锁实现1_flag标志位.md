# 锁实现 1：flag 标志位（朴素想法与问题）

> 课表：**锁实现原理 · 基础版** — `flag` 标志位

## 1. 最朴素模型

```text
locked = false   // 是否被占用
加锁：若 locked 则等；否则 locked = true
解锁：locked = false
```

用普通变量当「标志位」，**在多核/并发下不正确**。

## 2. 为何不可靠

- **读-改-写** 非原子：`if !locked { locked = true }` 可能被两个线程交错执行，**都以为拿到了锁**。  
- **可见性/重排序**：没有同步时，一个线程的写入对另一线程的观测顺序无保证（需内存屏障/原子）。

内存屏障，就是给 CPU/编译器 下死命令：
不许乱排指令，必须把缓存刷干净，再往下走。

1. 它管两件事（你只要记这两个）
 
1. 禁止指令重排
屏障前面的代码，必须先执行完，才能执行屏障后面的。
→ 解决指令重排序问题。
​
2. 强制刷新缓存
​
- 屏障前的所有写入，立刻刷回主内存
​
- 屏障后的所有读取，必须从主内存重新读
→ 解决可见性问题。
 
所以：
内存屏障 = 同时解决：重排序 + 可见性
## 3. 正确方向

- 用 **原子操作** 实现「测试并设置」或 **CAS（compare-and-swap）** 更新标志位。  
- 仅靠 flag 不够时，还要解决 **等待队列 / 阻塞唤醒**（见后续章节）。
