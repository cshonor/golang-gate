# 01 - 错误介绍

这是一份关于 **Go 语言错误处理哲学** 的学习笔记：Go 与其他语言在错误设计上的区别，以及常见实践方向。

---

## 一、核心设计理念

Go 放弃了传统的 `try` / `catch` / `finally` 式异常模型，采用 **「错误作为返回值」的显式处理**，这是 Go 最具辨识度的设计之一。

### 1. 核心设计原则

- **显式处理**：错误是函数的普通返回值，由调用方主动判断与处理，而不是「隐式抛出、上层捕获」。
- **就地处理**：错误在发生处附近处理，控制流不依赖长距离栈回溯，更易跟读。
- **可组合性**：错误是实现 `error` 接口的值，可以包装、传递、组合，适配复杂业务。

---

## 二、为什么 Go 选择这种设计？

### 1. 控制流更清晰

- `try` / `catch` 会打断线性执行，异常与栈回溯在大型服务里更难追踪。
- 基于返回值的错误让执行顺序更直观，便于调试和维护。

### 2. 错误处理不「隐形」

- 传统异常若缺少约束，容易被忽略，错误被静默吞掉后难以排查。
- Go 鼓励（并通过工具辅助）显式处理 `err`，减少「看不见的错误」。

### 3. 更适合服务端长期维护

- Go 常见于服务端、云原生、分布式场景，需要稳定与可运维性。
- 显式错误让边界清晰，便于熔断、降级、日志与监控。

---

## 三、核心补充知识

### 1. `error` 接口本质

Go 中的错误类型需实现 `Error() string`：

```go
type error interface {
	Error() string
}
```

任何满足该方法的类型都可作为错误返回，这是错误可组合的基础。
```go
package main

import "fmt"

// 自定义错误类型
type MyError string

// 实现Error()方法
func (e MyError) Error() string {
    return string(e) // 直接返回字符串内容
}

func main() {
    var err error = MyError("这是一个自定义错误")
    fmt.Println(err) // 输出：这是一个自定义错误
}
```
### 2. 常见处理方式

- **基础判断**：调用后立即 `if err != nil`，最常见。

```go
_, err := os.Open("test.txt")
if err != nil {
	log.Fatalf("open file failed: %v", err)
}
```


- **错误包装（Go 1.13+）**：用 `%w` 保留下层错误，便于链式判断与追溯（详见 [04 - 错误包装与错误链](./04%20-%20错误包装与错误链.md)）。

```go
return fmt.Errorf("open file failed: %w", err)
```

- **自定义错误**：实现 `error` 接口，携带业务上下文（详见 [03 - 自定义错误类型](./03%20-%20自定义错误类型.md)）。

### 3. 与 `panic` / `recover` 的区别

- **`error`**：用于可预期的失败（如文件不存在、参数不合法），属于正常控制流的一部分。
- **`panic` / `recover`**：用于不应继续执行的严重问题；不能替代常规错误返回（详见 [06 - panic 与 recover](./06-panic%20与%20recover%20捕获异常.md)）。

## 四、显式错误 vs 传统 try/catch（简要对比）

| 特性 | Go 显式错误处理 | 传统 try/catch |
|------|-----------------|----------------|
| 控制流 | 线性清晰 | 抛出/捕获打断流程，栈回溯复杂 |
| 错误可见性 | 倾向显式处理 | 易被忽略，存在静默失败风险 |
| 代码形态 | 常见重复的 `if err != nil` | 表面更短，逻辑有时更隐蔽 |
| 维护性 | 长期服务、协作场景较友好 | 小项目里有时写得更快 |
| 可组合性 | 错误是普通值，易包装传递 | 异常模型相对固定 |

---

## 五、实践建议（摘要）

1. **尽早处理**：在合适层级处理或向上返回，避免无意义地堆积后再暴露。
2. **带上下文包装**：如 `fmt.Errorf("user login failed: %w", err)`，便于日志定位。
3. **区分可恢复与不可恢复**：业务失败用 `error`；真正不可恢复再考虑 `panic`。
4. **慎丢错误**：避免 `_ = err`，除非能证明无需处理且已文档化。
