# 04 - 错误包装与错误链

Go 1.13+ 起，`fmt.Errorf` 的 **`%w`** 与 **`errors.Is` / `errors.As` / `errors.Unwrap`** 配合，实现**错误包装（wrap）**与**错误链（error chain）**：在保留底层错误的前提下附加上下文，便于追溯与按类型/哨兵判断。自定义类型与 `As` 的配合见 [03](./03%20-%20自定义错误类型.md)；`Is`/`As` 速查见 [05](./05-errors.Is%20与%20errors.As%20使用.md)。

---

## 1. 为什么要包装错误

- **定位更清晰**：底层错误带上「在哪一步、操作什么失败」（如 `query user id=123`），避免日志里只有 `connection refused` 却不知道是哪条业务路径。
- **保留原始错误**：不丢底层类型或哨兵值，上层仍能判断「文件不存在」「超时」等。
- **形成错误链**：多层包装后成链，`Is`/`As` 可沿链查找。

---

## 2. 标准写法：`%w`

### 2.1 基础用法

```go
import (
	"errors"
	"fmt"
)

func queryUser(id string) error {
	return errors.New("db connection refused")
}

func getUser(id string) error {
	err := queryUser(id)
	if err != nil {
		return fmt.Errorf("get user id=%s: %w", id, err)
	}
	return nil
}
```

- 打印效果类似：`get user id=123: db connection refused`。
- **`%w` 对应位置必须是 `error` 类型**，否则无法通过编译。

### 2.2 多层包装（错误链）

```go
func handleRequest(id string) error {
	err := getUser(id)
	if err != nil {
		return fmt.Errorf("handle request: %w", err)
	}
	return nil
}
```

语义上的链路：`handle request` → `get user id=…` → 底层原因。

---

## 3. 错误链上的三个常用操作

### 3.1 `errors.Is`：链上是否「包含」某错误

沿链判断是否与目标错误**相等**（含目标实现了 `Is` 时的自定义行为）。适用于**哨兵错误**（如 `os.ErrNotExist`、`io.EOF`）。

```go
var ErrDBConn = errors.New("db connection refused")

func queryUser(id string) error {
	return ErrDBConn
}

// err 可能经多层 %w 包装
if errors.Is(err, ErrDBConn) {
	// 仍可能为 true
}
```

**不要**对包装后的错误用 `err == ErrDBConn` 代替 `errors.Is`（包装后外层是新的 `error` 值）。

### 3.2 `errors.As`：提取某类型实例

将链中的错误**断言为某类型**，并赋值给指针指向的变量。适用于**自定义错误结构体**（见 [03](./03%20-%20自定义错误类型.md)）。

```go
type PathError struct {
	Path string
	Err  error
}

func (e *PathError) Error() string {
	return fmt.Sprintf("path %s: %v", e.Path, e.Err)
}

func openFile(path string) error {
	return &PathError{Path: path, Err: os.ErrNotExist}
}

var pe *PathError
if errors.As(err, &pe) {
	// 使用 pe.Path、pe.Err
}
```

### 3.3 `errors.Unwrap`：手动解包一层

返回被 `%w` 包住的**下一层**错误；多层需循环或依赖 `Is`/`As` 遍历。一般业务代码更多直接用 `Is`/`As`。

```go
inner := errors.Unwrap(err)
```

## 4. 规则与实践摘要

1. **有新上下文再包装**；否则可直接 `return err`，避免无意义嵌套。
2. **习惯把 `%w` 放在格式串末尾**，与「一个底层 err」对应，可读性更好。
3. **`%w` 与 `%v`**  
   - **`%w`**：保留链，供 `Is`/`As`/`Unwrap`。  
   - **`%v`**：把错误变成字符串参与格式化，**不保留**可解包链；适合「只写入日志、不需要再判型」的场景。
4. **少数哨兵**（如 `io.EOF`）在 API 合同里常约定**不包装**，由调用方直接 `errors.Is(err, io.EOF)`；具体以库文档为准。

错误链在实现上可理解为**单向链表**；`Is`/`As` 会沿链查找。

---

## 5. 可运行示例（哨兵 + 自定义类型 + 包装）

```go
package main

import (
	"errors"
	"fmt"
)

var ErrDBConn = errors.New("db connection refused")

type UserError struct {
	UserID string
	Reason string
}

func (e *UserError) Error() string {
	return fmt.Sprintf("user %s: %s", e.UserID, e.Reason)
}

func dbQuery(id string) error {
	return ErrDBConn
}

func validateUser(id string) error {
	if id == "" {
		return &UserError{UserID: id, Reason: "empty id"}
	}
	return nil
}

func getUser(id string) error {
	if err := validateUser(id); err != nil {
		return fmt.Errorf("validate user: %w", err)
	}
	if err := dbQuery(id); err != nil {
		return fmt.Errorf("query user id=%s: %w", id, err)
	}
	return nil
}

func main() {
	// 路径 A：底层为 ErrDBConn
	errA := getUser("123")
	fmt.Println("最终错误：", errA)
	if errors.Is(errA, ErrDBConn) {
		fmt.Println("→ 检测到 DB 连接错误（Is）")
	}
	fmt.Println("→ Unwrap 一层：", errors.Unwrap(errA))

	// 路径 B：底层为 *UserError
	errB := getUser("")
	var ue *UserError
	if errors.As(errB, &ue) {
		fmt.Printf("→ 用户错误：UserID=%q Reason=%s\n", ue.UserID, ue.Reason)
	}
}
```

---

## 6. 常见误区

| 误区 | 说明 |
|------|------|
| 用 `%v` 代替 `%w` 却仍要做 `Is`/`As` | 链断裂，判定失败 |
| 多层 `fmt.Errorf("failed: %w", err)` 且无新上下文 | 噪音大；不如直接返回 `err` 或合并一层 |
| 对包装后的错误用 `==` 与哨兵比较 | 应使用 `errors.Is` |
| 向 `%w` 传入 `nil` | 易与「无错误」混淆；不同 Go 版本行为需查文档，工程上应避免 |
