# 13 - 第三方错误库实战

原生 `error` + `%w` + `Is`/`As` 足以应对多数服务；需要 **调用栈**、**结构化 cause**、或与日志深度集成时，会引入第三方库。本节列常见选型与注意点。前置：[05](./05%20-%20错误包装与错误链.md)、[11](./11%20-%20错误与日志集成.md)。

---

## 1. `github.com/pkg/errors`（经典，维护态弱）

- 曾广泛使用：`errors.Wrap`、`Cause`、`StackTrace`。
- 仓库已标记 **归档**；新项目更推荐 **标准库 `%w`** 或下文替代方案。
- 若维护老代码：读栈时注意与 `errors.Is` 迁移策略（逐步改为 `%w`）。

---

## 2. `github.com/cockroachdb/errors`（带栈 + 兼容 `Is`/`As`）

- 提供链、栈、安全报告等能力，与 Go 1.13+ 错误链模型较合拍。
- 适合需要 **一致栈信息** 的中大型服务（按需引入，避免到处混用多种风格）。

---

## 3. 日志：`zap` / `logrus` / `slog`

- **不要**只打 `err.Error()` 丢栈；若使用 zap 等，可用 `zap.Error(err)`；若错误类型实现 `fmt.Formatter` 的 `%+v` 展开（部分库支持），在开发环境打开。
- 与 **结构化字段** 配合：`trace_id`、`code` 与 `err` 分列，见 [11](./11%20-%20错误与日志集成.md)。

---

## 4. 选型建议

| 场景 | 建议 |
|------|------|
| 普通业务错误、可判定 | 标准库 + 哨兵/自定义类型 |
| 要栈、仍要 `Is`/`As` | `cockroachdb/errors` 或自研薄封装 |
| 老项目大量 `pkg/errors` | 渐进迁移到 `%w`，新代码不再 Wrap |
