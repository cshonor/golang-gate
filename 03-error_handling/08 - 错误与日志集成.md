# 08 - 错误与日志集成

面向**可观测、可排查**：错误从业务函数冒泡到 HTTP/RPC 边界时，如何与日志配合，而不重复、不丢上下文。前置：[04 - 错误包装](./04%20-%20错误包装与错误链.md)、[07 - 最佳实践与反模式](./07%20-%20错误处理最佳实践与反模式.md)。

---

## 1. 原则

1. **错误值负责传递**；**日志负责留痕**。二者互补，不要每层都 `log + return err` 重复刷同一件事。
2. **在边界或决策点**记录：进入外部系统失败、请求失败、需要人工关注的异常；带上**请求 ID、租户、操作名**等（注意脱敏，见下）。
3. **结构化日志**：用键值字段（`slog`、`zap`、`zerolog` 等）而不是只拼一句字符串，便于检索与告警。

---

## 2. 典型分层做法（简图）

```
Handler / 中间件     →  记录：HTTP 状态、trace_id、err 摘要
Service / 用例层     →  可选：关键业务失败一条；或仅 return fmt.Errorf(...: %w)
Repository / 客户端  →  少打或不打；把 err 往上包上下文
```

- **底层**：优先 `return fmt.Errorf("op detail: %w", err)`。
- **顶层**：`log.Error("request failed", "err", err, "trace_id", id)` 并映射 4xx/5xx。

---

## 3. 与标准库 `log` / `slog` 的衔接示例

```go
if err != nil {
    slog.Error("create order failed",
        "err", err,
        "order_id", orderID,
        "trace_id", traceID,
    )
    return fmt.Errorf("create order: %w", err) // 或映射后不再向上传
}
```

要点：`err` 作为字段交给日志库；若使用 `%w` 包装，上游仍可用 `errors.Is` / `As` 判定类型。

---

## 4. 生产环境注意点

- **关联 ID**：`trace_id` / `request_id` 贯穿日志与链路，便于对齐单次请求的多条日志。
- **敏感信息**：密码、token、证件号不要打进日志；必要时打掩码或哈希。
- **级别**：预期内的业务错误（如校验失败）可用 `Warn`/`Info`；意外失败、依赖不可用用 `Error`。
- **不要重复**：已在 Handler 打 ERROR 的，下层不必再对同一失败打 ERROR（除非补充**不同**维度信息）。

---

## 5. 与错误链的配合

包装保留 `%w` → 日志里打印 `%+v` 或专用库展开栈（若使用）时仍可读；判错仍用 `errors.Is`/`As`，与日志字段独立。
