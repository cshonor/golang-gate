# 14 - 实战：完整项目错误处理设计

把前面章节收束成 **可落地的分层约定**（HTTP / gRPC / 领域层 / 存储）。前置：[10](./10%20-%20错误处理最佳实践与反模式.md)、[11](./11%20-%20错误与日志集成.md)。

---

## 1. 分层职责（简图）

```
Handler / Middleware  →  记日志、trace、映射 HTTP 状态码 / RPC code
Service / UseCase     →  业务语义、fmt.Errorf 带场景、return err
Repository / Client   →  外部 IO 失败，包装 %w，少打日志
```

- **领域层**：尽量不依赖具体日志实现；用 `error` 表达失败。
- **边界**：统一把 `error` 转成对外码与文案，避免把内部栈泄露给客户端。

---

## 2. HTTP 服务（示例策略）

| 错误类型 | 典型映射 |
|----------|----------|
| `errors.Is(…, ErrNotFound)` | 404 |
| 校验 / 哨兵 `ErrInvalid` | 400 |
| 认证 / 授权 | 401 / 403 |
| 上游超时、依赖不可用 | 502 / 503 + 重试头策略 |

实现上可在中间件或 `errors.As` 到自定义 `*HTTPError` 再映射。

---

## 3. 数据库与并发

- **DB**：`sql.ErrNoRows` 是否在业务层视为 404，要在 **repository 边界** 决定，避免每层都包装同一语义。
- **并发**：`errgroup` 收集多个错误时，可用 `errors.Join`（Go 1.20+）或显式聚合，再在上层统一记录一次日志。

---

## 4. 检查清单（上线前）

- [ ] 边界外是否不会暴露敏感内部信息？
- [ ] 关键路径是否都有 `trace_id`？
- [ ] 需判定的失败是否都走 `Is`/`As` 而非字符串？
- [ ] `defer Close` 是否与返回 `err` 协同（见 [09](./09%20-%20defer与错误处理协同.md)）？
