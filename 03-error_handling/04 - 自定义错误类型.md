# 04 - 自定义错误类型

前置：[02 - 错误基本语法](./02%20-%20错误基本语法.md)、[03 - 错误创建](./03%20-%20错误创建-errors.New与fmt.Errorf.md)。若错误会被 `fmt.Errorf("...: %w", err)` 包装，提取类型请用 [06 - errors.As](./06%20-%20errors.Is与errors.As使用.md)。

---

## 1. 一句话对比

| | 普通错误 | 自定义错误 |
|---|----------|------------|
| 常见写法 | `errors.New("...")` / `fmt.Errorf("...")` | 自定义结构体 + 实现 `Error() string` |
| 能带什么 | 基本只有**文案**（链路由 `%w` 扩展） | **字段**：错误码、业务字段名、ID 等 |

直觉对比：

- 仅字符串：「参数错误」  
- 带数据：`Code=400`，`Field=name`，`Msg=用户名不能为空` —— 便于映射 HTTP 状态码、结构化日志与监控。

---

## 2. 为什么要自定义？

`errors.New` 只能表达**固定字符串**，例如：

```go
return errors.New("参数错误")
```

业务里往往需要机器可读的 **Code**、给人看的 **Msg**、以及 **Field** / **ID** 等上下文。这些需要**结构体字段**承载，再让该类型实现 `error` 接口。

---

## 3. `error` 接口与 `Error()` 方法

```go
type error interface {
	Error() string
}
```

任意类型只要实现了 `Error() string`，返回值就可以当作 `error` 使用。

### 3.1 最小示例

```go
type BizError struct {
	Code int
	Msg  string
}

func (e *BizError) Error() string {
	return e.Msg
}
```

---

## 4. 返回与解析：类型断言与 `errors.As`

### 4.1 直接返回、未包装时：类型断言

```go
func login(name string) error {
	if name == "" {
		return &BizError{
			Code: 400,
			Msg:  "用户名不能为空",
		}
	}
	return nil
}

// 调用方
err := login("")
if myErr, ok := err.(*BizError); ok {
	fmt.Println("错误码：", myErr.Code)
	fmt.Println("错误信息：", myErr.Msg)
}
```

### 4.2 中间曾用 `%w` 包装时：用 `errors.As`

若链路中存在 `fmt.Errorf("login: %w", err)`，`*BizError` 可能被包在错误链里，此时 `err.(*BizError)` 可能失败。应使用 `errors.As`（见 [06](./06%20-%20errors.Is与errors.As使用.md)）。

```go
var be *BizError
if errors.As(err, &be) {
	fmt.Println(be.Code, be.Msg)
}
```

### 4.3 易混对照：接口断言、`type switch`、`reflect` 不是一回事

Go 里常见**三种「看类型」的写法**，底层能力与成本不同：

| 方式 | 形式 | 是否 `import "reflect"` | 典型用途 |
|------|------|-------------------------|----------|
| **接口类型断言** | `v, ok := err.(*BizError)` | 否 | 接口值底层**就是**该类型时，还原为具体类型 |
| **`type switch`** | `switch v := x.(type) { case *BizError: ... }` | 否 | 同一套机制的分支写法，**只能写在 `switch` 里** |
| **`reflect` 包** | `reflect.TypeOf(x)`、`reflect.ValueOf(x)` | 是 | 运行期动态看类型/字段/方法，**能力强、成本高** |

**要点**：

- `x.(T)`、`switch x.(type)` 属于 **语言原生**的接口断言，**不是** `reflect` 包 API；编译器可做部分检查，日常解析自定义错误**优先断言 + `errors.As`** 即可。
- 业务里为判断 `*BizError` **不必**写 `reflect.TypeOf(err) == reflect.TypeOf((*BizError)(nil))` 这类代码，属于杀鸡用牛刀。

下面三段代码**并列对照**（均为「从 `error` 里认出 `*BizError`」的意图，但写法不同）：

**写法一：接口类型断言（轻量，链上无包装时）**

```go
if myErr, ok := err.(*BizError); ok {
	_ = myErr.Code
}
```

**写法二：`type switch`（与断言同源，多分支更清晰）**

```go
switch v := err.(type) {
case *BizError:
	_ = v.Code
default:
	// 其它类型
}
```

**写法三：`reflect`（一般不用于普通业务分支）**

```go
import "reflect"

t := reflect.TypeOf(err)
// 仅示意：还要处理 err 为 nil、接口内动态类型为 nil 等边界
_ = t
```

自定义错误在**可能被 `%w` 包装**的场景，仍应优先 **`errors.As`**（§4.2），而不是依赖 `reflect`。更深反射主题见 [`02-interface-and-reflection`](../02-interface-and-reflection/README.md)。

## 5. 带 Code、Field、Msg 的完整示例（可直接改项目用）

```go
package main

import (
	"errors"
	"fmt"
)

type BizError struct {
	Code  int
	Msg   string
	Field string // 可选：哪个字段不合法
}

func (e *BizError) Error() string {
	if e.Field != "" {
		return fmt.Sprintf("[%d] %s (field=%s)", e.Code, e.Msg, e.Field)
	}
	return fmt.Sprintf("[%d] %s", e.Code, e.Msg)
}

func validateName(name string) error {
	if name == "" {
		return &BizError{
			Code:  400,
			Msg:   "用户名不能为空",
			Field: "name",
		}
	}
	return nil
}

func main() {
	err := validateName("")
	var be *BizError
	if errors.As(err, &be) {
		fmt.Println("code:", be.Code, "msg:", be.Msg, "field:", be.Field)
	}
}
```

说明：`Field` 为空时，`Error()` 可只输出 code + msg，避免日志里出现无意义的 `field=`。

---

## 6. 这样做的好处

- **按类型分流**：参数类 / 权限类 / 系统类映射不同 HTTP 状态码或重试策略。  
- **结构化日志与告警**：把 `code`、`field` 打成 JSON 字段，比纯字符串更好检索（与 [11 - 错误与日志集成](./11%20-%20错误与日志集成.md) 配合）。  
- **与 `errors.Is` 配合**：可为包定义哨兵值；需要**字段**时再用自定义类型 + `As`（见 [06](./06%20-%20errors.Is与errors.As使用.md)、[07](./07%20-%20Sentinel%20Error哨兵错误设计.md)）。

---

## 7. 速记

- 普通错误：多为**字符串**（或包装链）。  
- 自定义错误：**结构体 + `Error()`** → 仍是 `error`。  
- 调用方：**直接类型**用 `err.(*T)` 或 `switch err.(type)`；**可能被包装**用 `errors.As`。  
- 判错**不必**上 `reflect.TypeOf`；断言与 `type switch` **不是**反射包。
