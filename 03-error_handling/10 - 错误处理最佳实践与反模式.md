# 10 - 错误处理最佳实践与反模式

在已掌握 [02](./02%20-%20错误基本语法.md)、[05](./05%20-%20错误包装与错误链.md)、[06](./06%20-%20errors.Is与errors.As使用.md) 的基础上，用几条**可执行**的习惯减少线上坑。

---

## 1. 推荐做法（实践）

### 1.1 边界处理，内部向上传

- **库 / 领域层**：尽量 `return fmt.Errorf("...: %w", err)`，把决策（重试、HTTP 状态码、是否告警）留给上层。
- **HTTP / RPC / main**：在**边界**统一记录日志、映射状态码、打点；避免每一层都打满日志又原样返回（见 [11](./11%20-%20错误与日志集成.md)）。

### 1.2 包装时补「谁在什么场景失败」

```go
return fmt.Errorf("load user id=%s: %w", id, err)
```

便于排查；链路上用 `errors.Is` / `errors.As` 识别底层原因（见 [06](./06%20-%20errors.Is与errors.As使用.md)）。

### 1.3 可判定的错误用哨兵或自定义类型

- 包级变量：`var ErrNotFound = errors.New("not found")`，上层 `errors.Is(err, ErrNotFound)`（规范见 [07](./07%20-%20Sentinel%20Error哨兵错误设计.md)）。
- 需要字段时用自定义结构体并实现 `Error()`（见 [04](./04%20-%20自定义错误类型.md)）。

### 1.4 `panic` 留给真·不可恢复场景

业务可预期失败一律 `error`；`panic` 见 [08](./08%20-%20panic与recover捕获异常.md)。

## 2. 常见反模式（避免）

| 反模式 | 为何有问题 | 更好方向 |
|--------|------------|----------|
| `_ = err` 或忽略返回值 | 失败被静默，线上难查 | 显式处理或注释说明为何可忽略 |
| 用 `err.Error() == "..."` 判断 | 文案一变就错；包装后字符串变长 | `errors.Is` / 哨兵错误 |
| 业务失败滥用 `panic` | 打断控制流、难测试 | 返回 `error` |
| 只 `log` 再 `return nil` | 调用方以为成功 | `return err` 或返回包装后的错误 |
| 同一错误在每一层都打 ERROR | 日志刷屏、同一事件多条 | 在**一层**记关键上下文（常是边界） |
| 包装时滥用 `%v` 丢掉链 | 无法用 `Is`/`As` | 需要保留原因时用 `%w`（见 [05](./05%20-%20错误包装与错误链.md)） |
| `defer` 里吞掉错误 | 资源释放失败被隐藏 | `Close` 错误合并或记录（见 [09](./09%20-%20defer与错误处理协同.md)） |
