# 二、逃逸分析（Escape Analysis）：GC优化的核心手段

逃逸分析是Go编译器在编译阶段执行的**静态分析**，作用是：**判断变量是否会逃逸出函数栈，决定分配到栈还是堆**，是减少GC开销的最核心手段。

## 1. 逃逸分析的核心目标

- 尽可能让变量分配到栈（零GC开销），只有必须分配到堆的变量才逃逸到堆。
- 避免不必要的堆分配，减少GC压力，提升程序性能。

## 2. 逃逸分析的触发与查看

- **编译时自动执行**：无需手动配置，Go编译器默认开启。
- **手动查看逃逸结果**：

```bash
# 查看变量逃逸情况，-l 关闭内联（避免内联干扰分析结果）
go build -gcflags="-m -l" main.go
```

输出示例：

```
./main.go:5:6: moved to heap: x  # x逃逸到堆
./main.go:7:2: foo does not escape # foo函数的参数不逃逸
```

## 3. 常见逃逸规则（面试必背）

### 一定会逃逸的场景

1. **函数返回局部变量的指针**（最常见）

```go
func foo() *int {
    x := 10 // x逃逸到堆，因为返回指针给外部使用
    return &x
}
```

2. **变量大小动态不确定**（如slice长度/容量是变量，非常量）

```go
func foo(n int) {
    s := make([]int, n) // n是变量，slice大小不确定，逃逸到堆
}
```

3. **变量被闭包引用**

```go
func foo() func() {
    x := 10
    return func() { // 闭包引用x，x逃逸到堆
        fmt.Println(x)
    }
}
```

4. **变量赋值给接口类型（`interface{}`）**  
   接口是动态类型，运行时才能确定具体类型，通常需要堆上分配（具体以编译器输出为准）。

```go
func foo() {
    x := 10
    var i interface{} = x // x 往往逃逸到堆
}
```

5. **变量被发送到channel/存储到全局变量**  
   变量会被其他goroutine访问，生命周期超出当前函数，通常必须分配到堆。

6. **大对象（超过栈可用空间等）**  
   栈空间不足时可能改为堆分配（以编译器与运行时策略为准）。

### 不会逃逸的场景（典型）

- 函数内局部变量，不返回指针、不被外部引用、大小确定。
- 常量大小的slice/map，不返回指针。
- 闭包不引用外部变量，或引用的变量不逃逸。
- 函数参数按值传递，不返回指针。

## 4. 逃逸分析的面试考点

- 问：怎么减少Go程序的GC开销？  
  答：核心是**通过逃逸分析减少堆分配**：
  1. 避免返回局部变量的指针，尽量按值传递（小对象）。
  2. 用常量大小的slice/map，避免动态大小（在业务允许时）。
  3. 减少闭包对外部变量的引用。
  4. 避免不必要的接口类型转换。
  5. 用`sync.Pool`复用频繁创建销毁的小对象，减少堆分配。
