# 什么是逃逸（Go 语言版）

> 前置阅读：[01-内存分配堆与栈.md](./01-内存分配堆与栈.md)（栈/堆与分配器直觉；`mcache`/`mcentral`/`mheap` 见 [02](./02-mcache%20mcentral%20mspan.md)、[03-mheap.md](./03-mheap.md)）

在 Go 语言里，**逃逸**指的是：  
原本定义在函数内部的局部变量，因为被外部引用、生命周期超出当前函数，**从「栈内存」跑到了「堆内存」**，这个过程就叫**变量逃逸**。

编译器在编译时自动分析变量会不会「跑出去」，这个行为叫做**逃逸分析（Escape Analysis）**。

---

## 一、为什么要关心逃逸？

Go 里内存分两种：

1. **栈（Stack）**
   - 函数一结束，帧弹出，局部变量随调用结束而失效（由编译器安排，不经 GC）
   - 分配/回收极快，没有 GC 压力
2. **堆（Heap）**
   - 需要 **GC** 参与回收
   - 分配与扫描成本更高，可能影响延迟与 CPU

**逃逸分析的作用：**

- 变量**不逃逸**：尽量放栈（或寄存器/编译器认为的等价路径）
- 变量**逃逸**：只能放堆（或必须可被 GC 追踪）

一句话：**逃逸越多，堆分配越多，GC 压力通常越大；在热点路径上值得关注。**

---

## 二、什么情况会发生逃逸？

下面是 Go 里最常见、面试常考的逃逸场景。

### 1. 返回局部变量的指针（最典型）

```go
func f() *int {
	x := 10
	return &x
}
```

- `x` 是函数内部局部变量
- 你把**指针返回给外部**了
- 函数返回后栈帧不再有效，但外面还要用这个地址
- 所以 `x` 必须**逃逸到堆**

### 2. 被闭包捕获

```go
func closure() func() int {
	x := 10
	return func() int {
		x++
		return x
	}
}
```

- 闭包会让 `x` 在函数返回后仍可能被访问
- 生命周期超过原函数
- `x` **逃逸**

### 3. 变量大小不确定 / 运行时才知道

```go
func makeSlice(n int) []int {
	s := make([]int, n)
	return s
}
```

- 编译期不知道 `n` 多大时，底层数组大小在运行期决定
- **切片头**可在栈，**底层数组**往往在堆（以 `-gcflags=-m` 为准）

### 4. 太大的变量

```go
func big() {
	var arr [100000]int
	_ = arr
}
```

- 单个对象过大，超出栈帧合理限制时，编译器可能改为**堆分配**（阈值随版本/实现变化，以编译器输出为准）

### 5. 被指针间接引用（结构体字段等）

```go
type T struct {
	p *int
}

func foo() {
	x := 5
	t := &T{p: &x}
	_ = t
}
```

- `x` 的地址流入堆上可达对象时，**`x` 常随引用链逃逸**（具体以 `-m` 输出为准）

### 6. 发送指针到 channel

```go
func ch() {
	c := make(chan *int)
	x := 10
	c <- &x
}
```

- 指针可能跨 goroutine 使用，编译器保守处理 → **常逃逸**

### 7. 使用反射、`fmt` 等走 `interface{}` 的路径

```go
func fmtDemo() {
	x := 10
	fmt.Println(x)
}
```

- `fmt` 等 API 内部常把实参以 `interface{}`（或等价）传递，触发**装箱**，实参往往**逃逸**（并非所有打印都等价，以 `-m` 为准）

### 8. 赋给接口类型（补充）

```go
func iface() {
	x := 42
	var i any = x
	_ = i
}
```

- 值装入接口通常需要在堆上放置可共享、带类型信息的表示（**以编译器输出为准**）

### 9. 全局变量、长生命周期容器（补充）

- 写入 `package` 级变量、`sync.Map`、长生命周期 map/slice 且元素含指针时，相关分配常发生在堆上。

## 三、怎么看变量有没有逃逸？

```bash
go build -gcflags="-m -l" .
# 或针对单文件
go build -gcflags="-m -l" main.go
```

- **`-l`**：关闭内联，避免内联把逃逸信息「藏」掉，读输出更稳。

输出里出现：

```text
moved to heap: x
```

说明 **`x` 发生了逃逸**（或与其相关的分配在堆上）。  
也可能看到 `does not escape`，表示编译器认为可留在栈上路径。

---

## 四、逃逸和 Rust 生命周期有什么关系？（极简对比）

- **Go 逃逸**：编译器自动分析，变量会不会**逃出函数**（能否在栈上结束生命）
- **Rust 生命周期**：用类型系统标注，引用会不会**活得比被引用数据更久**

**目标一致**：避免悬空指针、保证内存安全。  
**差异**：Go 由编译器推断；Rust 由程序员写出可检查的约束。

---

## 五、一句话总结（适合背）

**在 Go 里，逃逸就是：局部数据因仍可能被函数外使用，而不能只活在当前栈帧里，于是放到堆上由 GC 管理的过程。**  
**逃逸分析决定「栈 vs 堆」；热点路径上少逃逸、少堆分配，通常更利于性能。**

## 六、优化与面试补充（与本目录衔接）

- **减少无意义的 `interface{}` / `any`、反射、热路径上的 `fmt`**（或改用定型 API），可降低意外装箱与逃逸。
- **小结构体**在允许时**按值返回**，避免「为了省拷贝反而返回指针」导致逃逸。
- **复用对象**：`sync.Pool`、预分配 slice 容量等（见 [08-GC触发与性能优化.md](./08-GC触发与性能优化.md)）。
- 面试常问「如何减 GC」：**先看 `go build -gcflags="-m -l"` 的堆分配热点，再改数据结构与 API 形状。**
