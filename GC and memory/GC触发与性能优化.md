# 六、GC触发时机与性能优化（面试高频）

## 1. GC触发时机

Go的GC是**自动触发**，无需手动管理，触发条件（概念层面）：

1. **堆内存阈值触发**：默认`GOGC=100`，即堆增长达到设定比例时触发下一轮 GC（具体计算以实现为准）。
   - 可通过环境变量`GOGC`调整：`GOGC=200`表示更「懒」的 GC（通常更少 GC、更多占用）；`GOGC=50`更激进（更频繁、占用更低趋势）。
2. **定时触发**：有周期性兜底触发，避免长时间不 GC 导致占用持续偏高（细节见 runtime）。
3. **手动触发**：`runtime.GC()`，会驱动一次 GC，**生产环境慎用**，多用于测试/诊断。

## 2. Go GC性能优化（面试必问）

### （1）减少堆分配（最核心）

- 通过逃逸分析，让尽可能多的变量分配到栈（零GC开销）。
- 用`sync.Pool`复用频繁创建销毁的小对象（如临时 buffer），减少堆分配和 GC 压力。
- 避免不必要的大对象分配。

### （2）优化GC参数

- 调整`GOGC`：根据业务内存情况，平衡 GC 频率与内存占用。
- 调整`GOMEMLIMIT`（Go 1.19+）：限制堆内存上限，配合调度器行为更适合容器化部署。

### （3）排查GC瓶颈

用`pprof`等工具分析 GC 开销：

```bash
go tool pprof http://localhost:6060/debug/pprof/heap
# GC 相关：关注 trace、heap、alloc 等；具体路径以你的服务暴露为准
```

重点关注：GC频率、STW时间、堆分配率、逃逸情况。

### （4）避免内存泄漏

- 避免goroutine泄漏（如channel未关闭、无限阻塞），导致关联对象无法回收。
- 避免全局变量长期持有大对象/闭包捕获大对象，导致无法及时释放。
