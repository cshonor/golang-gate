# 一、Go内存分配：堆 vs 栈（GC的基础前提）

Go的内存分配直接决定了GC的开销，核心原则是：**栈分配无GC开销，堆分配需要GC管理**。

## 1. 栈分配（Stack Allocation）

栈是goroutine私有的内存空间，用于存储函数调用的局部变量、参数、返回值等。

### 核心特性

- **自动回收**：函数执行结束后，栈空间自动释放，**完全不需要GC参与**，零开销。
- **连续栈设计**：Go 1.4之后废弃了早期的分段栈，改用**动态扩容的连续栈**：
  - 初始栈大小：2KB（Go 1.4+），远小于C的8MB默认栈，节省内存。
  - 栈扩容：当goroutine栈空间不足时，自动分配2倍大小的新栈，拷贝旧栈数据，更新所有指针指向新栈，无用户感知。
  - 栈缩容：函数执行结束后，若栈空间利用率低于25%，自动缩容，避免内存浪费。
- **分配规则**：只有满足「逃逸分析不逃逸」的变量，才会分配到栈。

## 2. 堆分配（Heap Allocation）

堆是全局共享的内存空间，用于存储需要跨goroutine访问、生命周期不确定的变量，**必须由GC管理回收**。

### Go堆分配的层级架构（面试必问）

Go的堆分配采用**三级缓存+全局管理**的无锁/低锁设计，大幅提升分配效率：

| 层级 | 所属 | 作用 | 锁状态 | 适用对象 |
|------|------|------|--------|----------|
| `mcache` | 每个P（Processor）私有 | 缓存小对象内存块，无锁分配 | 无锁 | 小于32KB的小对象（tiny/small） |
| `mcentral` | 全局共享 | 按大小类（size class）管理`mspan`，为`mcache`补充内存 | 有锁（每个大小类独立锁） | 所有大小对象 |
| `mheap` | 全局共享 | 管理整个堆内存，向操作系统申请/释放内存 | 全局大锁（仅在申请内存时加锁） | 大于32KB的大对象 |
| `mspan` | 内存块单元 | 堆内存的最小管理单元，由多个连续页（page，8KB）组成 | 无 | 所有对象 |

### 大小对象分配规则

- **Tiny对象（<16B）**：分配到`mcache`的tiny块，多个tiny对象复用一个内存块，减少内存碎片。
- **Small对象（16B ~ 32KB）**：从`mcache`的对应大小类`mspan`中分配，无锁，速度极快。
- **Large对象（>32KB）**：直接从`mheap`分配页对齐的内存，需要加全局锁，开销较大。

> 注：具体阈值与实现细节会随 Go 版本微调，面试以「小对象走 mcache、大对象走 mheap」为主线，读源码请以当前 `src/runtime/malloc.go` 等为准。
