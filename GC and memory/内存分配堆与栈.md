# 一、Go内存分配：堆 vs 栈（GC的基础前提）

Go的内存分配直接决定了GC的开销，核心原则是：**栈分配无GC开销，堆分配需要GC管理**。

## 1. 栈分配（Stack Allocation）

栈是goroutine私有的内存空间，用于存储函数调用的局部变量、参数、返回值等。

### 核心特性

- **自动回收**：函数执行结束后，栈空间自动释放，**完全不需要GC参与**，零开销。
- **连续栈设计**：Go 1.4之后废弃了早期的分段栈，改用**动态扩容的连续栈**：
  - 初始栈大小：2KB（Go 1.4+），远小于C的8MB默认栈，节省内存。
  - 栈扩容：当goroutine栈空间不足时，自动分配2倍大小的新栈，拷贝旧栈数据，更新所有指针指向新栈，无用户感知。
  - 栈缩容：函数执行结束后，若栈空间利用率低于25%，自动缩容，避免内存浪费。
- **分配规则**：只有满足「逃逸分析不逃逸」的变量，才会分配到栈。

## 2. 堆分配（Heap Allocation）

堆是全局共享的内存空间，用于存储需要跨goroutine访问、生命周期不确定的变量，**必须由GC管理回收**。

### Go堆分配的层级架构（面试必问）

Go的堆分配采用**三级缓存+全局管理**的无锁/低锁设计，大幅提升分配效率：

| 层级 | 所属 | 作用 | 锁状态 | 适用对象 |
|------|------|------|--------|----------|
| `mcache` | 每个P（Processor）私有 | 缓存小对象内存块，无锁分配 | 无锁 | 小于32KB的小对象（tiny/small） |
| `mcentral` | 全局共享 | 按大小类（size class）管理`mspan`，为`mcache`补充内存 | 有锁（每个大小类独立锁） | 所有大小对象 |
| `mheap` | 全局共享 | 管理整个堆内存，向操作系统申请/释放内存 | 全局大锁（仅在申请内存时加锁） | 大于32KB的大对象 |
| `mspan` | 内存块单元 | 堆内存的最小管理单元，由多个连续页（page，8KB）组成 | 无 | 所有对象 |

#### 最顺的主线（从头串一遍）

你可以按这条链路背下来（背会就“通了”）：

**mcache → mcentral → mheap → mspan → slot**

换成更“总览”的说法就是：

**mheap（总仓库） → mcentral（公共池） → mcache（P 小抽屉） → mspan（内存块） → slot（小格子）**

---

### 2.1 每个角色到底是干嘛的（超直白版）

#### ① `mheap` —— 整个堆的“大总管 / 总仓库”

- 全局只有 **一个**
- 负责向 **操作系统** 申请/归还内存（arena 等）
- 负责把大块内存切成 `mspan`，并维护 span 的分配/回收

#### ② `mcentral` —— “公共池子”（按 size class 分桶）

- 全局有很多个：每个 **size class** 对应一个 `mcentral`（数量随版本略变）
- 里面放着一堆 **可用的 `mspan`**
- 作用：给各个 P 的 `mcache` **补给 `mspan`**
- 访问通常需要锁，但因为“按 size class 分桶”，锁竞争相对可控

#### ③ `mcache` —— “P 私有的小抽屉”（最快路径）

- 每个 **P 绑定一个 `mcache`**
- 里面缓存了各 size class 对应的可用 `mspan`
- **完全私有，不加锁，极快**
- 你当前 goroutine 分配小对象：**优先走这里**

#### ④ `mspan` —— “一段真实的连续内存”（只服务一种 size class）

- 由若干个连续 page 组成（page 大小以实现为准；你只要记住“span 是连续页”即可）
- 一个 `mspan` **只服务一种 size class**（只切一种大小的格子）
- 内部会被切成很多个等大的 **slot**

#### ⑤ `slot` —— 最小分配单元（槽位）

- `mspan` 里切出来的等大格子
- **一个 slot 存一个对象**
- 比如 8B 的 size class，就是从对应的 span 里取一个 8B slot

**一句话拍死结论**：

- ✅ **一个 `mspan` 只管“一种大小”，不管类型**
- ❌ 不是“一个 mspan 只能存一种类型”，而是“一个 mspan 只能切一种大小的 slot”

所以只要 **字节大小一样**，什么类型都能放进同一个 size class 的 `mspan/slot` 里（`int`、指针、小结构体等都可能落在同一档）。

**自测题（面试秒答）**：

- 一个 `int8`（1B）和一个 `int64`（8B），会放在同一个 `mspan` 里吗？  
  - **不会**：它们会映射到不同的 size class，落在不同大小的 `mspan`/slot。

---

### 2.2 分配一个 `int`（8B）的完整流程（你最关心的）

1. 你 `new(int)`（或等价分配）要一个 8B 对象
2. 当前 goroutine 正在某个 P 上运行
3. 用这个 P 自带的 **`mcache`**
4. 找 `mcache` 里 **8B size class 对应的 `mspan`**
5. 从这个 `mspan` 里拿一个 **空 slot**
6. 返回该 slot 的地址（对象就放在这里）

如果 `mcache` 里 8B 对应的 `mspan` 满了：

- 去 **`mcentral`（8B 的公共池）** 拿一个新的 `mspan`
- 如果 `mcentral` 也空了 → 去 **`mheap`** 申请/切一个 `mspan` 再往下发
- 如果 `mheap` 也不够 → `mheap` 向 **OS 要内存**，再切 span 下发


### 大小对象分配规则
在 Go 内存分配里：
对象 = 一块在堆上、用来存数据的内存
跟 OOP、class、实例 一毛钱关系都没有

Go 内存分配把对象的大小分成了 68 档：
大概是这样：
0～8B → 第 1 档
8～16B → 第 2 档
16～24B → 第 3 档
…
一直到 32KB
一共 68 个尺寸档位
在 Go 内存分配里，“对象”就是一块在堆上存数据的内存（不是 OOP 的 class/实例概念）。对象大小会被映射到 **size class**（档位数量随版本略变），直到 32KB 左右都走“小对象”路径。

- **Tiny对象（<16B）**：分配到`mcache`的tiny块，多个tiny对象复用一个内存块，减少内存碎片。
- **Small对象（16B ~ 32KB）**：从`mcache`的对应大小类`mspan`中分配，无锁，速度极快。
- **Large对象（>32KB）**：直接从`mheap`分配页对齐的内存，需要加全局锁，开销较大。


二、整体运作逻辑（自上而下）
 
1. 启动阶段：mheap 初始化，向操作系统预申请大块虚拟内存空间。

​
2. 细分管理：mheap 将内存按大小分类（size class），划分成不同的mspan，每种大小类对应一个 mcentral。
​

3. 线程本地缓存：每个P（逻辑处理器）持有一个 mcache，从对应的 mcentral 抓取内存块缓存到本地。
​

4. 业务分配：Go程序申请内存时，优先从当前P的 mcache 中获取空闲内存块（无锁操作，最快）；mcache不足时，向 mcentral 申请；mcentral不足时，向 mheap 申请；mheap不足时，向OS申请。

 
三、关键设计亮点

 
1. 多级缓存（Cache-Like）：从OS到P再到线程，层层缓存，最大化减少锁竞争和内存碎片。

​
2. 大小类（Size Class）：根据对象大小预分配不同规格的内存块，避免外部碎片（external fragmentation）。

​
3. 无锁设计：利用P的私有mcache，将分配压力分散到每个P上，从根本上减少了全局锁的开销。


> 注：具体阈值与实现细节会随 Go 版本微调，面试以「小对象走 mcache、大对象走 mheap」为主线，读源码请以当前 `src/runtime/malloc.go` 等为准。
