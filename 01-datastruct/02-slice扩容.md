# Go 语言 slice 扩容机制深度解析

这是一份非常经典的 Go 切片（slice）扩容知识点总结：核心原理、易错点与实践建议。

---

## 一、核心知识点解读

### 1. 何时触发扩容

- **触发条件**：当执行 `append` 操作后，切片的**长度（len）超过了当前的容量（cap）**时，Go 会自动触发扩容。
- **底层行为**：
  1. 分配一块新的、更大的连续内存作为新的底层数组；
  2. 将旧数组中的所有元素完整拷贝到新数组；
  3. 更新当前 slice 的指针，使其指向新数组，同时更新 `len` 和 `cap`；
  4. 旧数组如果没有其他 slice 引用，会被 GC 回收。

### 2. 新容量的计算规则（Go 1.18+ 标准）

扩容策略在 `runtime/slice.go` 的 `growslice` 函数中实现，核心逻辑如下：

| 旧容量范围 | 扩容系数 | 说明 |
|------------|----------|------|
| `cap < 256` | 约 2 倍 | 小切片直接翻倍，减少扩容次数 |
| `cap ≥ 256` | 约 1.25 倍（并受对齐等影响） | 大切片降低扩容系数，避免内存浪费 |

- **补充细节**：
  - 最终容量会向上取整到内存对齐的大小，同时会根据元素类型（是否含指针、元素大小）做优化，平衡内存占用和 GC 扫描效率。
  - 不同 Go 版本的系数有微调，**永远以源码 `growslice` 为准**。

### 3. 与「共享切片」的关系（高频易错点）

- **扩容前**：多个 slice 可能共享同一个底层数组。此时修改其中一个 slice 的元素，会影响所有共享该数组的 slice。

```go
s1 := []int{1, 2, 3}
s2 := s1[:2] // s1 和 s2 共享底层数组
s2[0] = 100  // s1[0] 也会变成 100
```

- **扩容后**：当前 slice 会指向新数组，而其他共享旧数组的 slice 仍指向旧数组，两者彻底分离。

```go
// s2 原 len=2、cap=3 时，单次 append 一个元素往往不扩容；要演示「分离」需追加到超出 cap
s2 = append(s2, 4, 5, 6) // 触发扩容后 s2 指向新数组
s2[0] = 200              // 仅修改 s2，s1 不受影响
```

- **额外易错**：即使写成 `_ = append(s2, x)` 不接收返回值，只要**未扩容**且写入仍在共享数组的 `cap` 范围内，`append` 仍可能把数据写到兄弟 slice「看得见」的位置。正确做法是 **`s = append(s, x)`** 并按需使用返回的 slice。

### 4. 最佳实践建议

1. **预分配减少扩容**：已知最大长度时，用 `make([]T, 0, n)` 初始化，一次性分配足够容量，避免多次扩容带来的性能损耗。

```go
// 推荐：预分配 1000 个元素的容量
s := make([]int, 0, 1000)
```

2. **避免大切片内存泄漏**：大切片 `append` 后若长期只保留一小段，底层大数组仍会占用内存。可通过 `copy` 到新小切片释放原大数组（视场景而定）。

```go
// 从大切片中截取前 10 个元素，并拷贝到新切片，释放原大数组引用
newS := make([]T, 10)
copy(newS, oldS[:10])
```

3. **永远用返回值接收 append 结果**：`s = append(s, x)` 是强制规范，因为扩容后 `s` 的指针、`len`、`cap` 都会变化，不赋值会导致数据与语义不一致；且未赋值时仍可能在共享底层数组上写入。

## 二、自检问题完整解答

### 问题 1：`append` 是否可能修改「无关」的另一个 slice？在什么条件下会 / 不会？

- **会修改的条件**：两个 slice 共享同一个底层数组，且本次 `append` **未触发扩容**（写入落在当前底层数组已分配容量内）。本质是仍在同一块数组上写，共享该数组的 slice 会看到变化。
- **不会修改的条件**：
  1. 两个 slice 不共享底层数组；
  2. 共享数组，但本次 `append` 触发了扩容，执行 `append` 的那一侧指向新数组，与旧数组上的兄弟 slice 分离。

### 问题 2：`s = append(s, x)` 为什么要赋值回 `s`？

- **核心原因**：`append` 的返回值是**新的 slice 头**（可能包含新的指针、`len`、`cap`）。
- **若不赋值**，原 `s` 仍保存旧的 slice 头（旧指针、旧 `len`/`cap`），会导致：
  1. 看不到扩容后的新底层数组与正确长度；
  2. 后续逻辑基于旧的 `cap`/`len`，可能重复扩容、覆盖或逻辑错误。
- **补充**：即便某次未触发扩容，也应始终用返回值接收，保持风格一致并避免与「共享数组上的隐式写入」混淆。

## 三、补充：slice 结构体本质

Go 中的 slice 不是数组，而是一个**结构体**（概念上包含三个字段）：

```go
type slice struct {
	array unsafe.Pointer // 指向底层数组的指针
	len   int            // 切片当前长度
	cap   int            // 切片最大容量
}
```

（以上为便于理解的逻辑模型；具体布局以 `runtime` 实现为准。）

- **扩容的本质**：可能替换这个「头」里的 `array` 指针，并更新 `len`、`cap`，因此必须用返回值更新原变量。

## 四、常见面试题延伸

1. **切片和数组的区别**：数组是固定长度的值类型；切片是动态视图，slice 头是值拷贝，但内部指针指向共享的底层数组。
2. **切片作为函数参数**：传的是 slice 头的副本（指针、`len`、`cap` 拷贝）。函数内改元素可能反映到调用方；函数内 `append` **触发扩容**后，只有接收了返回值的局部变量指向新数组，调用方的 slice 头不会自动更新。
3. **`nil` 切片和空切片**：`var s []int` 是 nil 切片（指针为 nil，`len=0`，`cap=0`）；`s := []int{}` 常为非 nil 但长度为 0 的空切片。在 `len(s)==0` 时许多用法一致，但反射、JSON、与 `nil` 比较等场景可能有差别。
