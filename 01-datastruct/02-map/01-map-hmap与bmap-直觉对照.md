# map：hmap / bmap 与「数组 + 链表」模型对照（速查）

把教材里的「**哈希表 = 底层数组 + 冲突链表**」和 Go 的 **`hmap` / `bmap`** 对上号；并纠正 **HPM、BPM** 等常见口误（实际只有 **`hmap` 与 `bmap`**）。

**完整字段、`tophash`、扩容与面试题**见主篇：[02-map数据结构旧版.md](./02-map数据结构旧版.md)。

---

## 新手先看：一句话总纲 + 面试速背

**一句话总纲**：Go map 底层就是 **`hmap`（总控头） + `buckets` 指向的 `bmap` 桶数组`**；每个桶里放多对 key/value，**满了就 `overflow` 溢出桶链表**；扩容走 **渐进式搬迁**（见 [03-map渐进式扩容旧版本.md](./03-map渐进式扩容旧版本.md)），避免「一口气搬完」容易卡。

**超简速背（先背再往下读细节）**：

1. **结构**：**`hmap` 管全局，`bmap` 管存**。  
2. **桶里（教学常记）**：**`tophash` + keys + values + overflow**（槽位数以当前 `runtime` 为准，讲义里常写 **8**）。  
3. **key / value 怎么摆（经典讲法）**：常是 **一段 keys、再一段 values**，不是把 `k,v,k,v` 在整桶里交叉塞满那种简笔画（利于对齐与扫描；**细节以 `src/runtime/map.go` 为准**）。  
4. **为啥快**：先用 **`tophash`（哈希高位摘要）快速过滤**，再比**完整 key**。  
5. **冲突**：**同下标桶 + 溢出 `bmap` 链表**（链地址法思路）。  
6. **扩容**：**渐进式**，`oldbuckets` / `buckets` 协作（细读 [03](./03-map渐进式扩容旧版本.md)）。  
7. **遍历**：**不保证顺序**，见 [04-map遍历顺序不固定.md](./04-map遍历顺序不固定.md)、[05-map乱序原理.md](./05-map乱序原理.md)。  
8. **并发**：**非并发安全**，见 [06-map并发不安全.md](./06-map并发不安全.md)、[07-map并发安全方案.md](./07-map并发安全方案.md)。

## 1. 先把名字说准：`hmap` 不是「数组本体」

- **`hmap`**：整张 map 的**总控头结构体**（管理员），**不是**桶数组本身。它保存元素个数、桶数量参数 `B`、哈希种子、标志位等元数据，并通过字段 **`buckets`** 指向**真正的桶数组**（逻辑上是「`[]bmap`」一块连续区域，源码里用 `unsafe.Pointer` 表示，见下文示意）。
- **`bmap`**：**一个桶**。经典模型里每桶 **8 组** key/value；**满**了再挂 **overflow bmap**，形成**溢出桶链表**，对应教材里的「冲突链表」，只是链上每个结点是一个「能装多对的桶」而不是「一对 key/value 一个结点」。

容易混淆的点：**存数据的「数组」是 `buckets` 指向的桶数组；`hmap` 是管这张表的头。**

### 1.1 `hmap` 里大概记什么？（人话 =「户口本 / 总控」）

可以记：**`count`**（元素个数，对应 **`len(m)`** 语义）、**`B`**（桶数 **\(2^B\)**）、**`buckets` / `oldbuckets`**、**`hash0`**（哈希种子）等——**`hmap` 自己不装 key/value**，只**掌控元数据 + 指针**。

### 1.2 `bmap`：一个桶里有什么？（抓直觉）

教学版常画这些（与最终实现一一对应以 `map.go` 为准）：

| 部分 | 作用（人话） |
|------|----------------|
| **tophash 槽** | 存**高位摘要**，先**快速对照**，减少无脑比完整 key |
| **keys / values** | 真存数据；常是 **先一片 keys，再一片 values** |
| **overflow** | 槽满了，**再挂一串溢出 `bmap`**，继续装 |

概念布局（帮助记「不是 k,v 交叉塞满」）：

```text
[ key0 key1 key2 ... ]   ← 一片
[ val0 val1 val2 ... ]   ← 一片
```

## 2. 「数组 + 链表」模型在 Go 里成立吗？

经典教学模型：

1. 对 key 做哈希；  
2. 对数组长度取模（或等价运算）得到**下标**；  
3. 冲突用**链表**挂下去。

Go map **同属这一模型**，只是在「数组的每一格」里又做了 **多槽位桶 + 溢出桶链** 等优化。下面按**真实定位步骤**与教材用语一一对应。

---

## 3. 哈希定位三步（与「哈希 → 取模 → 下标」对照）

### 步骤 1：对 key 做哈希

运行时对 key 做哈希，得到**足够宽的哈希值**（常见讨论里常按 64 位理解）。计算会用到 **`hmap.hash0`**（建 map 时随机生成），减轻针对性哈希碰撞攻击。

### 步骤 2：得到桶下标（等价于对桶数取模）

桶个数为 **\(2^B\)**（`B` 在 `hmap` 里）。用哈希值的**低 `B` 位**直接得到 **\(0 \sim 2^B-1\)** 的下标，**数学上等价于** `hash % 2^B`；因桶数恒为 2 的幂，Go 用**位运算**实现，比通用取模更快。

例：`B=5` 时有 32 个桶，取哈希**低 5 位**即得到 `0~31` 的桶编号。

**对应教材说法**：「哈希后对**桶数组长度**取模得数组下标」——这里「数组」指 **`bmap` 桶数组**，不是对 `hmap` 这个结构体取模。

### 步骤 3：定位到 `bmap`，再处理冲突

1. 用步骤 2 的下标找到**那一个桶**（`bmap`）。  
2. 桶内用哈希**高 8 位**作 **`tophash`**，先快速筛选再必要时比较完整 key（主篇第三节细讲）。  
3. 桶内有空槽则插入；**8 个槽都满**则分配 **overflow bmap**，挂在当前桶的链表上，继续查找或插入。

### `val := m[key]`：超简流程（只看到现象时用这个串）

1. 算 key 的哈希（会用到 **`hash0`**）。  
2. **低 `B` 位** → 选中**哪一个桶**（等价 `hash % 2^B`）。  
3. 在桶里看 **`tophash`** → 对不上就跳过，对上再比**完整 key**。  
4. 本桶没有 → 沿 **overflow 溢出桶链**继续找。  
5. 找到返回值；没找到返回**零值**（是否需要 `ok` 看你是 `v :=` 还是 `v, ok :=`）。

## 4. 进阶校准（避坑）

### 4.1 为什么常说「低 B 位」而不是写 `%`？

桶数恒为 **\(2^B\)**，**低 B 位**与 **`% 2^B`** 等价，位运算实现更高效；逻辑与「取模定桶」一致。

### 4.2 `hmap` 里和「数组在哪」相关的字段（教学简化）

以下为**抓直觉**的节选，**完整定义以 `src/runtime/map.go` 为准**：

```go
type hmap struct {
	count  int    // 元素个数
	B      uint8  // 桶个数 = 2^B
	hash0  uint32 // 哈希种子（示意）

	buckets    unsafe.Pointer // 指向桶数组；逻辑上是 []*bmap / []bmap 布局
	oldbuckets unsafe.Pointer // 渐进扩容时的旧桶数组
	// ... flags、nevacuate、extra 等
}
```

要点：**「数组本体」是 `buckets` 所指的桶序列；`hmap` 是持有指针并管理元数据的头。**

### 4.3 哈希冲突

与「数组 + 链表」一致：同一桶下标内用 `bmap` 存；**桶满则溢出桶链表**延伸。`tophash` 是为**减少全 key 比较**做的实现层优化，不改变「链地址」抽象。

### 4.4 渐进式扩容为什么是「慢慢搬」？（人话）

- **一次性**把整张表全搬完：**很容易造成明显停顿**。  
- Go 用 **渐进式搬迁**：在**普通读写**路径里顺带搬一点，用 **`oldbuckets` / `buckets`** 协作（细则见 [03-map渐进式扩容旧版本.md](./03-map渐进式扩容旧版本.md)）。

## 5. 和教材用语的一览表

| 经典说法 | Go 里大致对应 |
|----------|----------------|
| 整张表、元数据、指向底层存储的指针 | **`hmap`** |
| 底层定长数组的「第 i 格」 | **`bmap` 数组**里下标选中的**那一个桶** |
| 冲突时的链表 | **`overflow` 链**（多个 **`bmap`**） |

---

## 6. 一句话锁死记忆

**`hmap` 是 map 的总控头，通过 `buckets` 管理 `bmap` 桶数组；key 哈希后，用低位（等价于对 \(2^B\) 取模）得到桶下标，定位到对应 `bmap`，冲突用溢出 `bmap` 链表解决。**

**换一句更顺口的收尾**：**`hmap` 管全局，`bmap` 管存储，`tophash` 管筛选速度，溢出桶管冲突，渐进扩容管性能；遍历不保证顺序，并发要自己管安全。**

---

## 7. 极简文字结构图

```
hmap
 └── buckets ──► [ bmap | bmap | bmap | ... ]     ← 桶数组（下标由哈希低位决定）
                      │
                      └── overflow ──► bmap ──► bmap ──► nil   ← 同一逻辑桶的溢出链
```

## 8. 速记

- **`hmap`**：总头，管元数据 + **指向桶数组的指针**（像「户口本 / 总控」）。  
- **`bmap`**：**数组里的一格**（一个桶，多槽 + **overflow**）。  
- **溢出 `bmap`**：同一桶上的**链表**延伸（链地址法）。  
- **`tophash`**：先比对**摘要位**，减少整 key 比较。  
- **扩容**：**渐进式**，见 [03](./03-map渐进式扩容旧版本.md)。  
- **遍历 / 并发**：见 [04](./04-map遍历顺序不固定.md)、[05](./05-map乱序原理.md)、[06](./06-map并发不安全.md)。

更细的字段、`tophash` 枚举、搬迁规则与面试题见 [02-map数据结构旧版.md](./02-map数据结构旧版.md) **第一～四节**。
