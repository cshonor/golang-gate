# 08 - map 并发安全方案

前置：[06-map并发不安全](./06-map并发不安全.md)（为何不能并发写、与 `sync.Map` 简介）。本篇归纳**常用替代方案**与选型直觉；**细节以标准库文档与 benchmark 为准**。

---

## 1. 结论先说

- **普通 `map`**：并发读写需**外加同步**或改用 **`sync.Map`** 等专用结构。  
- **没有万能方案**：按「读/写比例、key 集合是否稳定、是否需强一致遍历」选型。

---

## 2. `sync.Map`

**定位**：为「**读多、写少**，且 key 集合**相对稳定**」的场景优化；**不是**对所有并发 map 的普适替换。

**实现要点（直觉）**：内部维护**可读快照**与**脏数据**分支，配合互斥与原子操作；读路径常可避开全局锁，写路径更重。**不适合**写极多、或 key  churn 很大的场景（易退化成锁竞争）。

**适用**：

- 缓存、只增不改的注册表等；  
- 读远多于写，且愿意接受 API 为 `Store/Load/Range` 而非下标语法。

**不适用 / 慎用**：

- 需要**原生 `map` 语法**与复杂事务逻辑；  
- 写多读少：往往 **`sync.RWMutex` + map** 更直观且易推理。

---

## 3. `sync.RWMutex` + `map`

**写法**：一把锁保护一个 `map`；读用 `RLock`，写用 `Lock`。

**优点**：语义简单，与业务代码一致，易测。  
**缺点**：写多或 `RLock` 持有过久会阻塞读；**锁粒度大**，高并发下可能成为瓶颈。

**适合**：中小并发、逻辑以互斥 map 为主的服务内状态。

---

## 4. 分片（Sharding）：多把锁 + 多个 map

将 key 哈希到 **N 个分片**，每片一把锁（或每片一个 `map`）。**写分散到不同片**时可降低锁竞争。

**优点**：高并发写入场景可扩展。  
**缺点**：实现与测试成本高；**跨分片遍历/统计**需合并；`Range` 语义需自行定义。

**适合**：明确需要横向扩展 map 吞吐、且愿意维护分片逻辑的中大型服务。

---

## 5. 选型直觉（简表）

| 场景 | 常考虑方案 |
|------|------------|
| 读极多、写少、key 相对稳定 | `sync.Map`（先 profile） |
| 读写适中、逻辑简单 | `RWMutex` + `map` |
| 写很多、单机要撑更高并发 | 分片 map，或数据层用专门存储 |
| 跨进程 | 不要依赖内存 map；用 Redis 等 |

---

## 6. 实践提醒

- 任何方案都要 **`go test -race`** 与压测验证。  
- **`sync.Map` 的 Range** 语义与使用约束见官方文档。  
- 业务上常把「并发字典」需求收敛为 **单 goroutine 拥有 map**（actor 模型），从根源去掉锁竞争。
