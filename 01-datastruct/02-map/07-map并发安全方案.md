# 07 - map 并发安全方案

前置：[06-map并发不安全](./06-map并发不安全.md)（为何不能并发写、`sync.Map` 与自检题人话答案）。本篇用**最简单、最实战**的方式：**会选、会用**；**细节以标准库文档与 benchmark 为准**。

---

## 新手先看：一句话总纲

- **普通 `map`**：**不能**在未同步时**并发读写**（见 [06](./06-map并发不安全.md)）。  
- 想安全，工程上常走 **三条大路**：  
  1. **`sync.Map`** —— **读特别多、写特别少**，key 相对稳定。  
  2. **`map + sync.RWMutex`** —— **最通用、最好维护**，多数业务**先选它**。  
  3. **分片 map** —— **写压力巨大**、单机还要硬扛吞吐时再上。  
- **第四条（补充）**：**`atomic.Value` + 整表替换 / 快照**——只解决「**发布新版本 map**」，不是万能并发 map（见下文 **§5**）。

---

## 1. 结论先说（和总纲同一件事，换句人话）

- **没有万能方案**：按「读/写比例、key 是否频繁增删、要不要 `m[k]` 语法、是否跨进程」选型。  
- **默认心态**：**不要上来就用 `sync.Map`**；大多数项目 **`RWMutex + map` 最简单、最不容易出 bug**。

---

## 2. `sync.Map`（官方自带「并发 map」形态）

### 适合什么？

- **读特别多、写特别少**；  
- **key 集合相对稳定**（缓存、配置表、注册表一类）。

### 特点（直觉）

- 内部 **`read` 快照 + `dirty` + 原子操作`** 那一套；读路径**常常**能少碰全局锁，**写路径更重**。  
- **没有**舒服的 **`m[k]`** 语法，只能 **`Load` / `Store` / `Delete` / `Range`**。

### 不适合什么？

- **频繁增删 key**、**写比读还多**（容易**退化成锁竞争**，还不如 `RWMutex + map`）。  
- 需要 **原生 `map` 语义**、复杂事务式更新。

---

## 3. `sync.RWMutex` + 普通 map（最常用、最稳）

### 适合什么？

- **绝大多数业务**：读写比例一般、代码要**简单好测**。  
- 团队更熟 **`map` + 锁**，比先上 `sync.Map` 更不容易写飘。

### 原理（背这个就够）

- **读**：`RLock` → **多个 goroutine 可同时读**（读与读不互斥）。  
- **写**：`Lock` → **独占**，别人**不能读也不能写**。

### 优缺点

- **优点**：语义清晰，**`-race` 好写**；性能对大多数服务**够用**。  
- **缺点**：**写很多**或锁持有太久时，**锁竞争**会成为瓶颈。

---

## 4. 分片 map（高并发写入向）

### 适合什么？

- **写压力巨大**、单机还要把内存 map 榨干（不想立刻上 Redis 时的一类优化）。  
- 能接受 **实现 + 测试成本**。

### 原理（人话）

- 把 key **哈希到 N 个分片**（教学里常听到 **16 / 32 / 64** 之类，**N 按压测取**），**每片一把锁 + 一个小 `map`**。  
- **不同 key 落到不同片** → **锁冲突大幅下降**。

### 缺点

- **代码复杂**；**全量遍历 / 统计**要**扫所有分片**再合并；**Range 语义**要自己定义清楚。

---

## 5. `atomic.Value`：整表替换 / 快照（概念）

对应 [06](./06-map并发不安全.md) 自检第二题：**可以**，但只能玩 **「整体换一张 map」**，不能多人并发改**同一张** map。

**典型约束**：

1. **读**：`cur := v.Load().(map[string]int)` 一类（类型按你实际存的为准），只对这份引用做**只读**使用。  
2. **写**：**复制/新建**下一张表，改完后 **`Store`** 换成新 map；旧表无引用可被 GC。  
3. **代价**：写往往带 **O(n) 量级复制**（取决于你怎么做），只适合**写极少**。  
4. **一致性**：读方可能短暂看到**旧快照**——业务要能接受。

**一句话**：**`atomic.Value` 管的是「指针/引用这一格」的原子发布**，**不是** map 内部的并发安全魔法。

---

## 6. 超级实用选型表（背这个）

| 场景 | 推荐方案 |
|------|----------|
| **读超多、写极少**，key 相对稳定 | `sync.Map`（上之前 **profile**） |
| **普通业务**，要简单可靠 | **`RWMutex` + `map`**（默认首选） |
| **高并发写**、单机硬扛吞吐 | **分片 map**，或下沉到专门存储 |
| **跨机器 / 分布式** | **Redis** 等，**别**指望进程内 map 当分布式字典 |

---

## 7. 实践提醒

- **不要上来就用 `sync.Map`**：先 **`RWMutex + map`**，真有瓶颈再换。  
- 任何方案都要 **`go test -race`** 与压测验证。  
- **`sync.Map` 的 `Range`** 语义与坑见官方文档。  
- 业务上常把「并发字典」收敛成 **单 goroutine 拥有 map**（actor），从根源去掉锁竞争。

---

## 8. 极简总结（面试一句话）

- Go **原生 map** 不支持**未同步的并发读写**；  
- **读多写少**再考虑 **`sync.Map`**；  
- **通用场景**用 **`RWMutex + map`**；  
- **高并发写**用 **分片 map** 或专用存储；  
- **跨机**用 **Redis** 等。
