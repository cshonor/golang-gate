# map 渐进式扩容（旧版经典模型）· 深度解析

> **定位**：教学提纲与面试口径，对应「旧版经典」两表 + 渐进搬迁直觉。**当前实现以 `src/runtime/map.go` 与所用 Go 版本的 release note 为准**（大版本可能调整触发条件、批量迁移、桶布局等）。  
> 前置：[02-map数据结构旧版.md](./02-map数据结构旧版.md)（桶、tophash、bucket 直觉）

---

## 一、为什么需要扩容？

map 的性能与**装载因子（负载因子）**强相关：

> **装载因子** ≈ 已存元素个数 / **主桶**数量（教学上常这样记；精确口径以源码为准）

### 核心问题

- 元素持续增多 → 装载因子升高 → **哈希冲突**增多  
- 冲突多 → 查找/插入要扫更多槽位与**溢出桶** → 性能变差  
- 极端时接近链表扫描，均摊复杂度恶化

### 扩容在做什么

扩容 ≈ **更大的桶数组** + 对 key **重新哈希（rehash）** + 把条目**搬迁**到新桶，目的是把装载因子拉回舒适区，恢复接近均摊 O(1) 的读写。

---

## 二、「渐进式」是什么意思？

### 传统一次性扩容的痛点

若在某次操作中 **一口气** 搬完所有旧桶：

- map 很大时，单次操作延迟极高，像一次 **长停顿**（对延迟敏感的服务不友好）

### Go 的经典思路：渐进（增量）搬迁

1. 触发扩容时先申请 **约 2 倍** 规模的新桶区（概念级；是否等量翻倍以源码为准）。  
2. **旧桶区**挂在 `oldbuckets`，**新桶区**挂在 `buckets`，进入 **两表共存**。  
3. 在后续 **读/写/删** 等会触碰 map 的路径上，**顺带搬迁少量旧桶**（常见讲法是每次 1～2 个桶量级，具体以实现为准），把成本**摊**到多次调用。  
4. 旧桶搬完后释放对 `oldbuckets` 的引用，由 GC 回收，扩容阶段结束。

### 核心优势

- **摊薄**「触发扩容」那一下的峰值延迟；对业务 **透明**。  
- **总工作量**（每个 key 参与搬迁与重定位）与一次性全量搬迁 **同一量级**，只是 **时间分布**不同。

## 三、经典两表模型（扩容核心结构）

### 1. 读写与遍历（概念）

| 操作 | 概念逻辑 |
|------|----------|
| **读** | 根据 **该桶是否已搬迁**，从 **旧表或新表** 查找；未搬完前可能要查两张表思路。 |
| **写** | 常需先保证 **本桶相关区域** 与搬迁状态一致，再写入 **新表**（避免新旧长期不一致）。 |
| **遍历** | 需覆盖 **尚未搬迁的旧数据** 与 **新表数据**，保证不丢 key；顺序因此更「乱」（见 [04-map遍历顺序不固定.md](./04-map遍历顺序不固定.md)、[05-map乱序原理.md](./05-map乱序原理.md)）。 |

### 2. 搬迁粒度与状态

- **以桶（及挂着的溢出链）为单位**搬迁，而不是逐个 key 乱迁（效率与一致性）。  
- 用 **tophash 等标记** 表示槽位/桶是否已 **evacuated**（已搬迁），避免重复搬。  
- **`nevacuate`** 一类进度用于推进「下一个该搬的旧桶下标」，直到搬完。

## 四、与并发安全的关系

- map **本身不保证**并发安全；**扩容期**两表并存，实现更复杂，**更不能**多 goroutine 同时写。  
- 并发写 → **数据竞争**，常见直接 **`fatal error: concurrent map writes`**（细节见 [06-map并发不安全.md](./06-map并发不安全.md)）。  
- 工程上：**`sync.Mutex` / `RWMutex`** 包一层，或 **`sync.Map`**（场景有限），或 **分片**降低锁竞争。

## 五、自检：标准答法

### 1. 渐进式扩容降低的是哪一次操作的延迟？总工作量变了吗？

- **降低的**：主要是 **触发扩容当次** 若「全量搬完」会产生的 **超长延迟**；改为在多次操作中 **各摊一小段**。  
- **总工作量**：**基本不变**——该 rehash、该搬的条目总量仍在，只是 **拆开执行**。

### 2. 为什么扩容时遍历更「微妙」？

- 两表共存 + 搬迁进行中：**有的 key 还在旧侧、有的已在新侧**；遍历器要按规则扫过，**顺序更不可依赖**。  
- 这与「map 遍历无序」的 **语言语义**、**随机起点** 等叠加（见 05、06）。

## 六、新版 Go 与源码阅读提示

不同大版本可能对以下方面持续调整，**不要背死某版细节当永久真理**：

- **触发条件**：除装载因子外，是否还看 **溢出桶比例** 等（教材常提 **~6.5** 一类阈值，请以 `map.go` 中常量与注释为准）。  
- **每次搬迁多少桶**：与调用路径、优化策略相关。  
- **内存布局**：如是否引入更紧凑的桶/组结构（随版本查 release note）。

**做法**：面试答「两表 + 渐进搬迁 + 总工作量不变」；追问细节说 **打开当前版本的 `runtime/map.go` 对照 `growWork` / `evacuate` 等**。

## 七、高频面试题（简答）

1. **为什么要渐进式扩容？**  
   避免（或减轻）一次性全量搬迁带来的 **单次超长延迟**，把搬迁摊到多次 map 操作中，利于 **尾延迟**。

2. **两表模型是什么？**  
   **`buckets`（新）与 `oldbuckets`（旧）** 共存；按 **搬迁进度** 决定从哪读、写时如何推进搬迁，直到旧表清空。

3. **常见触发条件（旧版口径）？**  
   **装载因子超过阈值**（常记 **6.5** 为教学值）或 **溢出桶过多**；**以当前源码为准**。

4. **总工作量变吗？**  
   **不变**，只是 **时间分布**变。

5. **遍历为何无序？**  
   **语义不保证顺序** + **随机起点** + **扩容/搬迁改变布局** 等（见 05、06）。

## 八、最佳实践

1. **预分配**：`make(map[K]V, hint)`，减少扩容次数（hint 语义见官方文档）。  
2. **并发**：加锁或 `sync.Map`（慎用场景），不要裸写共享 map。  
3. **大 map**：尽量避免在峰值路径上依赖「刚扩容完」的瞬时行为；容量规划 + 预分配。  
4. **业务**：**永远不要**依赖 `range map` 的顺序。
