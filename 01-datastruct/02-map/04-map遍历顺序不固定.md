# map 遍历顺序不固定

用**最简单、最直白**的方式记：**依赖 `for range map` 的顺序写逻辑，一定是错的。** 乱序的**刻意设计**与随机起点等机制见 [05-map乱序原理.md](./05-map乱序原理.md)；扩容、两表共存见 [03-map渐进式扩容旧版本.md](./03-map渐进式扩容旧版本.md)。

---

## 新手先看：极简背诵版

1. Go map 遍历**顺序不保证**，**每次运行都可能不同**。  
2. **绝不**依赖顺序写业务逻辑（包括「先插的先出来」这种假设）。  
3. 要**稳定顺序** → 先抽 **keys** → **排序** → 再按 key 访问。  
4. **不固定**是**语言刻意设计**，用来逼你写不依赖顺序的代码。  
5. **扩容 / rehash** 会进一步改变布局，顺序**更不可依赖**。

---

## 1. 核心结论（必须记住）

- **`for k, v := range m`** 的遍历顺序**永远不固定**，**不能**指望「先插入的先遍历到」。  
- 你写的代码如果**依赖遍历顺序**，在 Go 里**一定是错的**（迟早会在别的机器、别的版本、别的负载下翻车）。

---

## 2. 为什么不固定？

Go **故意**设计成不保证顺序，常见原因叠在一起：

- **语言语义**：规范**不保证**迭代顺序。  
- **实现策略**：例如遍历**起始位置随机化**，防止大家依赖「看起来稳定」的顺序（细节见 [05](./05-map乱序原理.md)）。  
- **扩容与 rehash**：会**重新排桶**；渐进式扩容时还可能**两表共存**，遍历规则更复杂 → 顺序**更不可预测**（见 [03](./03-map渐进式扩容旧版本.md)）。

**人话**：**顺序不稳定 = 规范不保证 + 防依赖顺序的实现手段 + 扩容机制** 一起作用的结果。

---

## 3. 想要固定顺序怎么办？

**标准做法**：把 key 拿出来 → 排序 → 再按键访问 `m[k]`。

Go 1.21+ 可用 **`slices.Sort`**；更老版本可用 **`sort.Strings`** 等（键类型要匹配）。

```go
import (
	"fmt"
	"slices"
)

func main() {
	m := map[string]int{"a": 1, "b": 2, "c": 3}

	keys := make([]string, 0, len(m))
	for k := range m {
		keys = append(keys, k)
	}
	slices.Sort(keys)

	for _, k := range keys {
		fmt.Println(k, m[k])
	}
}
```

需要**插入顺序**时，用 **`slice` 记顺序 + `map` 查重/查值`** 或独立结构，不要指望 map 自带顺序。

---

## 4. 扩容为什么会「更乱」？

- 扩容会 **rehash、重新分配桶**，物理布局变掉。  
- 渐进式扩容时**新旧结构会共存一段时间**，遍历要兼顾两阶段，**顺序更不能指望**。

所以：**扩容 = 更不能依赖遍历顺序**（与 [03](./03-map渐进式扩容旧版本.md) 对照读）。

---

## 5. 自检（面试高频）

### 问：为什么「固定顺序」反而可能让依赖顺序的错误代码悄悄通过测试？

**答（人话版）**：

- 若遍历**总是同一种固定顺序**，你**错误地依赖顺序**的代码，在本地可能**每次都碰巧通过**。  
- Go **故意不固定**（并配合随机起点等），是为了让这类 bug **更容易在开发/测试阶段暴露**，而不是上线后在别的环境才爆。

**一句话**：**故意不固定 = 逼你写不依赖顺序的正确代码**，而不是靠「碰巧顺序稳定」蒙混过关。
