# 08 - map 内存模型与 GC（直觉）

用**最直白**的方式记：**map 吃内存主要在桶 + 溢出桶**；**GC 关心的是「还有没有引用」**；所谓「泄漏」多半是 **key 没删、大 value 一直可达**——不是 runtime 故意坑你。前置：[01-map-hmap与bmap-直觉对照](./01-map-hmap与bmap-直觉对照.md)、[02-map数据结构旧版](./02-map数据结构旧版.md)、[03-map渐进式扩容旧版本](./03-map渐进式扩容旧版本.md)。**细节以 `runtime/map.go` 与当前版本为准**。

---

## 新手先看：一句话总纲 + 极简背诵

**总纲**：

- map 占内存大头在 **桶 + 溢出桶**（由 runtime 分配）。  
- **GC** 要判断 **key/value 里是否含指针、是否还被 map 引用**。  
- **「泄漏」**多数是：**不 `delete` key**、或一直握着 **大 value**（大切片、大字符串等）→ **业务侧可达性**问题。  
- **想省内存 / 想清空**：**预估 hint**、**删无用 key**、或直接 **`m = make(map[K]V)` 换一张新表**。

**极简背诵（面试 + 工作）**：

1. 内存主体：**桶 + 溢出桶**；**渐进扩容**时**新旧表会短暂共存**，占用可能**短暂升高**。  
2. **含指针的 key/value** 会让 GC **多扫描**；**纯小标量**（如 `int`）相对轻（**注意**：`string`、切片等**在 GC 眼里不是「零负担纯值」**）。  
3. **未 delete 的 entry** 会让对应 value **一直可达** → 常被说成 **map 泄漏**，本质是**用法**。  
4. **清空最快**：旧 map **无人引用**时整表回收；常用写法是 **`m = make(map[K]V)`**。  
5. **`make(map[K]V, hint)`**：有把握时带上 **hint**，**减少反复扩容**（见 [../03-common-optimization/01-数据结构性能优化.md](../03-common-optimization/01-数据结构性能优化.md)）。

---

## 1. map 内存怎么长？（人话）

- 一开始很小；元素变多后 runtime **自动加桶、挂溢出桶**。  
- **渐进式扩容**时，**新旧两张表会共存一段时间**，内存曲线可能**先抬一截再回落**（见 [03](./03-map渐进式扩容旧版本.md)）。  
- 桶怎么切、怎么搬，**没有稳定对外 API** 给你数「当前几块」——把它当成**黑盒 + pprof** 去看即可。

---

## 2. map 和 GC 的关系

- **key/value 里带指针**（例如 **指针字段**、**引用堆上对象** 的类型）→ GC 必须沿着 **可达性** 去扫。  
- **大 value**（大 `[]byte`、大切片、长生命周期对象）只要 **entry 还在 map 里** → 就和「**大对象一直有人指着**」一样占内存。

**一句话**：**指针多、对象大 → GC 与内存压力都更明显**；**小标量 entry** 相对省心，但别误以为 **`string` 也零成本**（它有**数据指针**语义）。

---

## 3. map「内存泄漏」到底是什么？

**不是 Go 有 bug**，而是：

- key 放进 map 后 **从不 `delete`**；  
- 这个 key **一直可达** → 绑在上面的 **value 再大也不会被 GC**；  
- value 是 **大切片 / 大字符串 / 堆上大对象** 时，现象最明显。

这叫 **业务侧可达性泄漏**（你让对象**不该活那么久却还活着**）。

---

## 4. 怎么正确释放内存？

**两种最实用**：

1. **遍历 `delete(m, k)`**  
   适合：**还要复用同一个 map 变量**，只清内容。

2. **直接换新 map（常最干净）**

   ```go
   m = make(map[string]int)
   ```

   旧 map **没有别的引用** → 下一轮 GC **整表可回收**；代码也最好懂。

---

## 5. 日常优化（必记）

- **预估容量**：`make(map[string]int, 1024)` —— **hint 不是上限**，但能**减少扩容次数**与碎片压力。  
- **大对象做 value**：权衡 **「值拷贝 vs 指针共享」**——指针常能**少拷贝**，但只要 **entry 还在**，**堆上对象照样被 map 指着**；该 **`delete` / 换 map** 的别省。  
- **不用的 key 及时 `delete`**，或改设计（TTL、外置缓存等）。
