# Go 数组常见坑与最佳实践

## ⚠️ 常见坑点与解决方案

### 1. 数组传参造成全量复制
**问题**：函数接收大数组参数时，会复制整个数组，造成巨大性能开销。
```go
// 糟糕：接收 [10000]int 数组，会复制 10000 个元素
func sum(arr [10000]int) int { return 0 }

// 最佳：接收数组指针
func sum2(arr *[10000]int) int { return 0 }
```

### 2. 数组长度是类型的一部分
**问题**：`[3]int` 和 `[5]int` 是完全不同的类型，不能直接赋值或比较。
```go
a := [3]int{1,2,3}
b := [5]int{1,2,3,4,5}
_ = a
_ = b
// a = b // 编译错误：不同类型
```

### 3. 数组越界访问 (Index Out of Range)
**问题**：访问超出数组长度的索引，会直接导致 panic。
```go
arr := [3]int{1,2,3}
_ = arr
// fmt.Println(arr[3]) // panic: index out of range
```
**防范**：使用 `len(arr)` 校验索引范围。

### 4. 多维数组声明误区
**问题**：行尾必须加逗号，否则编译错误。
```go
arr := [2][3]int{
	{1, 2, 3},
	{4, 5, 6}, // 这里必须有逗号
}
_ = arr
```

## ✅ 最佳实践 (Best Practices)

### 1. 优先使用切片
除非有明确的固定长度需求，否则默认使用切片 `[]T`。

### 2. 大数组传递使用指针
对于长度较大的数组，始终使用指针 `*[n]T` 传递，避免内存复制。

### 3. 数组作为结构体字段时谨慎
结构体中的数组会占用整个结构体的空间。如果数组很大，考虑改用切片或指针。

### 4. 数组初始化规范
```go
arr1 := [3]int{1, 2, 3}
arr2 := [...]int{1, 2, 3} // 自动推断长度，推荐
_ = arr1
_ = arr2
```

### 5. 数组遍历效率
Go 中数组遍历效率很高，使用 `range` 更简洁。
```go
arr := [5]int{1, 2, 3, 4, 5}
for i, v := range arr {
	_, _ = i, v
}
```

## 🎯 总结
数组是 Go 的基石，但在现代 Go 开发中，**切片是主角，数组是配角**。
