# Go 数组内存布局

## 📍 内存模型核心
数组在内存中是**一段连续的、大小固定**的存储空间。数组的大小 = 元素大小 × 数组长度。

## 🧮 元素存储与对齐
### 1. 连续存储
假设 `var arr [3]int`，在 64 位系统中（int 占 8 字节），内存布局如下：
```
+--------+--------+--------+
|  arr[0]|  arr[1]|  arr[2]|
+--------+--------+--------+
  0x0000   0x0008   0x0010   (地址)
```

### 2. 内存对齐 (Memory Alignment)
Go 编译器会根据机器架构（32/64 位）和元素类型进行内存对齐，以保证 CPU 访问效率。
- **基本类型**：按自身大小对齐
- **数组**：按其最大元素类型的大小对齐

### 3. 数组与指针
数组变量本身代表整个数组的**首地址**。
- `&arr` 获取的是整个数组的指针 (类型为 `*[n]T`)
- `arr` 与 `&arr[0]` 的值相等，但类型不同

## 🧱 多维数组内存布局
多维数组是**数组的数组**，内存连续且按**行优先**存储。
```go
// 2x3 的二维数组
var arr [2][3]int
```
内存布局 (按行展开):
```
arr[0][0] arr[0][1] arr[0][2] arr[1][0] arr[1][1] arr[1][2]
```

## ⚠️ 值传递的内存代价
由于数组是值类型，传递给函数时会**完整复制**整个数组。
- 小数组（如 [10]int）代价可接受
- 大数组（如 [10000]int）会造成严重的栈溢出或性能损耗
- **最佳实践**：传递数组指针 `*[n]T` 或切片，避免全量复制

## 🔚 总结
数组的内存布局是理解 Slice 底层的关键。理解其**连续存储**和**值复制**的特性，是写出高性能 Go 代码的基础。
