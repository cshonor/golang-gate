# 03 - slice 常见坑与最佳实践

前置：[01-slice数据结构](./01-slice数据结构.md)、[02-slice扩容](./02-slice扩容.md)。本篇偏**工程踩坑**与习惯；map 侧见 `02-map/`。

---

## 1. 共享底层数组

切片头是「指针 + len + cap」，多个切片可**共用同一底层数组**。

- **改元素会互相影响**：`s2 := s1[1:3]` 后改 `s2[i]` 可能改到 `s1` 可见区域。
- **`append` 可能「断连」**：若 `append` 触发**扩容**，新切片指向**新数组**，旧切片仍指向旧数组，二者不再共享后续写入。

**习惯**：要独立副本时用 `copy` 或 `append([]T(nil), s...)` / `append(make([]T, len(s)), s...)`，并明确是否需要**截断 cap**（见第 4 节）。

---

## 2. `nil` 切片 vs 空切片

| | `nil` 切片 | 空切片（非 nil） |
|---|------------|------------------|
| 典型写法 | `var s []int` | `s := []int{}` 或 `make([]int, 0)` |
| `len`/`cap` | 多为 0 | 0 |
| 底层指针 | `nil` | 常为非 nil 的「空数组头」（实现相关） |

对多数业务：`len==0` 时行为接近；**与 JSON / 反射 / 需要区分「未设置」与「空列表」**时要显式约定（例如指针 `*[]T` 或单独标志位）。

---

## 3. 传参：为什么改元素生效、「扩容」像是不生效？

函数参数是**值传递**，拷贝的是**切片头**（指针、len、cap），不是拷贝底层数组。

- **改下标**：`func f(s []int) { s[0]=1 }` 改的是**共享底层数组**，调用方可见。
- **`append` 导致头变化**：若在 `f` 里 `s = append(s, x)`，**只改了 `f` 里的头副本**；若发生扩容，新数组的引用也只在 `f` 内。要让调用方看到新切片，需**返回** `[]T` 或传 `*[]T`。

---

## 4. 子切片导致「大数组」释放不掉

`s2 := s1[a:b]` 会让 `s2` 仍引用**整块旧数组**，只要 `s2` 存活，**整段 backing array** 无法被 GC，即使只用了少量元素。

**缓解**：

- 只需小片段时：**`copy` 到新切片**（`make` 成精确 `len`），让大数组无引用。
- 或 `s2 = append([]T(nil), s1[a:b]...)` 得到独立小数组（按需预分配 `cap`）。

---

## 5. 实践习惯（简表）

| 场景 | 建议 |
|------|------|
| 已知元素上限 | `make([]T, 0, n)` 预分配，减少反复扩容 |
| 需要独立副本 | `copy` 或 `append` 到新切片 |
| 频繁 `append` 且长度可估 | 设足够 `cap`，避免多次拷贝 |
| 对外返回内部 `[]byte` | 警惕调用方保留子切片；必要时 `copy` 再返回 |
