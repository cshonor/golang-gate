# Go 切片（slice）数据结构 · 深度解读

> 延伸阅读：扩容行为概要见 [02-slice扩容.md](./02-slice扩容.md)（与本篇第 3、5 节呼应）。

切片是 Go 里最高频的结构之一。本篇把 **slice header**、与数组的差异、**共享底层数组**与 **append 脱钩**、截取公式、**`len=0, cap>0`** 以及工程避坑串在一起；**扩容的具体阈值与倍数随 Go 版本调整**，以当前 `src/runtime/slice.go` 为准。

---

## 1. 切片的三字段头（slice header）

切片在 Go 中**不是数组本身**，而是一个**对底层数组的视图描述符**。runtime 里可理解为（字段名以源码为准，见 `runtime/slice.go`）：

```go
type slice struct {
	array unsafe.Pointer // 指向底层数组的起始元素
	len   int            // 当前切片可见的元素个数
	cap   int            // 从起始位置到底层数组末尾，最多可容纳的元素数
}
```

### 字段详解

| 字段 | 作用 | 关键说明 |
|------|------|----------|
| `array` | 指向底层数组某一起始元素 | 切片头里不「存满」元素，只存指针；数据在底层数组中 |
| `len` | 切片的「长度」 | 合法下标为 `s[0]`～`s[len-1]`，越界 panic |
| `cap` | 切片的「容量」 | 从 `array` 所指位置起到底层数组末尾，还能容纳的元素个数；恒有 **`cap >= len`** |

## 2. 切片 vs 数组：核心区别（面试必问）

| 特性 | 数组 `[n]T` | 切片 `[]T` |
|------|-------------|------------|
| **类型本质** | 值类型，**长度 n 是类型的一部分** | 由 **slice 头（三字段）** + 底层数组共同体现行为；常口语称「引用语义」 |
| **内存** | 连续存储全部元素，整体拷贝 | 拷贝/传参时主要是 **拷贝头**（仍指向同一底层数组，除非发生扩容换新数组） |
| **长度** | 固定 | `len` 可变；靠 `append` 等增长（可能触发扩容） |
| **初始化** | `var arr [5]int`、`arr := [3]int{1,2,3}` | `s := []int{1,2,3}`、`s := make([]int, 3, 5)` |

### 关键结论

- **数组**：赋值、传参会拷贝**整个数组**；改副本不改原数组（若未共享）。
- **切片**：赋值、传参拷贝的是 **header**；多个 slice 可**共享同一底层数组**，一方改下标可能影响另一方可见区间；**扩容后**当前 slice 可能指向**新数组**，与其它 slice **脱钩**。

## 3. 共享底层数组的后果（高频踩坑）

### ① 截取 `s[low:high]` 仍共享底层数组

```go
s := []int{1, 2, 3, 4, 5}
s1 := s[1:3] // s1 可见 [2,3]
s1[0] = 100
fmt.Println(s) // [1 100 3 4 5]
```

一方改共享区间内的元素，另一方在同一段底层数组上的视图也会变。

### ② `append` 与「脱钩」

- **`len < cap`**：往往在**原底层数组**上追加，仍可能与兄弟 slice 共享同一块数组（注意是否重叠区间）。
- **需要更大容量**：会 **分配新底层数组**，拷贝旧元素再追加；**当前** `append` 返回的 slice 通常指向新数组，与未扩容的其它 slice **脱钩**，互改不再相干。

更细的「谁会改到谁」见 [02-slice扩容.md](./02-slice扩容.md)。

## 4. 自检：标准答法（面试高频）

### 问题 1：`s[low:high]` 新切片的 `len`、`cap` 怎么算？

- **`len = high - low`**
- **`cap = 原切片的 cap - low`**（从 `low` 起到底层数组末尾还能放多少元素）

```go
s := make([]int, 5, 5) // len=5 cap=5
s1 := s[1:3]          // low=1 high=3
// len(s1)=2, cap(s1)=5-1=4
```

**三索引截取** `s[low:high:max]`：**`cap = max - low`**，用于限制新 slice 的可写区间，避免后续 `append`「长」进父切片尾部，是工程里常用安全写法。

### 问题 2：`len=0, cap>0` 有意义吗？

**有。** 典型用途：

1. **预分配**：`s := make([]int, 0, n)`，`append` 在容量内不扩容，减少反复 `grow`。
2. **与 `nil` 区分**：`nil` slice `len=0, cap=0`，指针字段为 nil；`len=0, cap>0` 有底层数组，**`s != nil`**（若你用 JSON / 反射等在意 nil vs 空）。
3. **复用缓冲区**：如 `s = s[:0]` 清空但保留 `cap`，继续 `append` 复用同一块数组。

| 类型 | `len` | `cap` | `array` | `s == nil` | `append` |
|------|-------|-------|---------|------------|----------|
| `nil` 切片 | 0 | 0 | nil | `true` | 通常会分配新底层数组 |
| 空切片 `make([]T,0,n)` | 0 | >0 | 非 nil | `false` | 在容量内用原数组 |
| 普通切片 | >0 | ≥len | 非 nil | `false` | 视是否扩容而定 |

## 5. 扩容规则（延伸，与版本强相关）

逻辑入口在 `runtime` 的 **`growslice` / `nextslicecap`**（名称与实现随版本微调）。**教材式归纳**通常是：

- 容量不足时：**新数组 + 拷贝旧元素 + 再写入新元素**。
- **小容量**：增幅往往较大（常听到「约翻倍」类描述）；**大容量**：增幅放缓，减少内存浪费。
- **元素大小**、是否含指针等也会影响策略（与 GC、对齐有关）。

**不要背死某一个「256 / 1024 / 1.25」到具体版本**——面试可答「查 `runtime/slice.go`」+ 上面两条直觉。概要对照：[02-slice扩容.md](./02-slice扩容.md)。

## 6. 工程实践避坑

1. **大序列传参**：用 **切片**（传的是头 + 共享数组），避免传 **大数组** 造成整段拷贝。（64 位上 header 一般为 **三个机器字**：指针 + 两个 `int`，常说成约 24 字节；以 `unsafe.Sizeof` 自证即可。）
2. **截取后还要 `append`**：考虑 **`s[low:high:high]`**（或合适的 `max`）**限制 cap**，避免意外写到父切片「后面」的槽位。
3. **已知元素规模**：`make([]T, 0, n)` 或 `make([]T, n)` 预分配，减少扩容。
4. **内存泄漏**：从大 slice **截取一小段**长期持有，会令**整段底层数组**无法回收；需要时用 **`copy` 到小切片** 只保留小块。
