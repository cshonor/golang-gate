# Go 零值机制（完整版）

## 1. 定义

声明变量但**不显式初始化**时，Go 会把它设为该类型的**零值（zero value）**。  
和 C/C++ 里“栈上未初始化可能是随机脏数据”不同，Go 里**保证是确定的零值语义**，不是垃圾比特。

## 2. 各类型零值速查

| 类型 | 零值 |
|------|------|
| `bool` | `false` |
| 整数（`int` / `int64` / `uint` …） | `0` |
| 浮点（`float32` / `float64`） | `0.0` |
| `string` | `""` |
| 指针 `*T` | `nil` |
| 数组 `[N]T` | 每个元素都是对应类型的零值 |
| 切片 `[]T` | `nil` |
| `map` | `nil` |
| `chan` | `nil` |
| 接口 `interface{...}` | `nil`（无动态类型与动态值） |
| 函数类型 `func(...)` | `nil` |
| `struct` | 各字段按各自类型取零值 |

## 3. 示例

```go
var a int
var s string
var p *int
var sli []int

fmt.Println(a, s, p, sli)
// 0  <nil> []   （空串 s 在 Println 里表现为连续空格）
// *int 的 nil → <nil>；nil 切片 → []（不是 <nil>）
```

## 4. 核心意义（工程直觉）

- **消灭“未初始化就读”的脏内存模型**：默认值可预期。  
- 很多场景下**零值就够用**（例如 `bytes.Buffer` 的零值可用、`sync.Mutex` 零值可用——具体以类型文档为准）。  
- 和 GC、并发模型搭配时，**“先声明、后慢慢填”**仍然要遵守类型自己的规则（尤其 `map`）。

## 5. 必考大坑

### ① `nil` 的 `map`：不能写

```go
var m map[string]int // m == nil
// m["a"] = 1 // panic: assignment to entry in nil map
m = make(map[string]int)
m["a"] = 1 // OK
```

读不存在的 key 会得到值类型的零值，不会 panic；**写**才会炸。

### ② `nil` 的切片：和 `nil` map 不一样

- `len(nilSlice) == 0`，`cap(nilSlice) == 0`  
- **`append(nilSlice, x)` 合法**（会分配新底层数组并得到非 nil slice）  
- **按下标读写** `nilSlice[i]` 会 panic（因为没有元素）

### ③ 零值结构体往往“就能用”

不需要为了“初始化”而强行 `new` / `&T{}`；是否可用取决于类型的 API 约定（例如零值 `mutex` 可直接 `Lock()`）。

### ④ “零值”≠“`nil`”

- `int` 零值是 `0`，**没有** `nil` 这种取值  
- `string` 零值是 `""`，**也不是** `nil`  
- **`nil` 主要出现在**：指针、切片、map、channel、接口、函数等类型的零值

## 6. 一句话口诀

**Go 声明就有零值，语义确定不摸瞎；`map` 未 `make` 别写入，`nil` 切片别乱按下标。**
