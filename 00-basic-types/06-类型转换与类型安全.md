# Go 类型转换与类型安全（完整版）

## 1. 强制规则（先把“强类型”说清楚）

对**已经带上具体类型的变量**来说，Go **不会像 C/Java 那样**在赋值时悄悄把 `int64` 塞进 `int`：

- **`int`、`int32`、`int64` 等是不同类型**，哪怕“都是整数”，也**不能自动互赋值**。
- 需要就用 **显式转换**：`T(x)`。

```go
var a int = 10
var b int64 = 20

// a = b // 编译报错
a = int(b) // 显式转换（仍要注意范围，见 §4）
```

**重要例外（面试常考）**：**无类型常量（untyped constant）**在不少上下文里更“灵活”，编译器会在**能表示、能赋值**的前提下帮你落到目标类型上；一旦写成 `var b int64 = 20` 这种**已具类型变量**，规则立刻变严。  
一句话：**别把“常量很宽松”当成“所有类型都能隐式互转”。**

## 2. 统一语法

只有这一种形式：**`T(表达式)`**  
没有 C 的 **`(T)val`** 写法。

## 3. 高频常用转换

- **整型互转**：`int` ↔ `int32` / `int64` / `uint` …（按目标类型重新解释；注意范围）
- **浮点与整型**：`float64` ↔ `int`（浮点转整型按**向零截断**）
- **字符串与字节**：`string` ↔ `[]byte`
- **字符串与码点**：`string` ↔ `[]rune`

## 4. 高危风险点（必背）

1. **大范围 → 小范围整型：可能“静默失真”**  
   超出目标类型可表示范围时，不是编译期必报错（非常量场景下常见是**按目标类型规则截断/重解释**），结果可能完全不符合直觉。**跨平台时 `int` 位宽还依赖实现**，更要小心。

2. **`float` → `int`：直接丢小数，不是四舍五入**  
   `int(3.99) == 3`（向零取整）。

3. **`string` ↔ `[]byte` / `[]rune`：通常要付拷贝成本**  
   心智模型按「**会复制/会分配**」准备；别当零成本视图转换。

4. **`bool` 与数字：永远不能互转**  
   不能 `bool(1)`，也不能 `int(true)`；条件里必须写 `x != 0` 这种显式比较。

## 5. 额外核心考点

- **接口值还原具体类型**：用**类型断言** `v, ok := i.(T)`，或 **`switch i.(type)`**。
- **指针类型**：不能像在 C 里那样“随便 cast”；跨类型指针转换走 **`unsafe.Pointer`** 就是另一条赛道（可读性/可维护性/可移植性都很差，业务代码应尽量避免）。
- **数组 vs 切片**：`[N]T` 与 `[]T` **不是**“直接 `T(x)` 一下就行”的关系；常见是 `s := arr[:]`（得到指向同一数组的 slice）、`copy`、或逐元素构造。
- **别名类型**：`type UserID int` 与 `int` 也是不同类型，通常也要显式转（或别乱转，优先用构造函数/方法约束）。

## 6. 一句话总结

**Go 对“已具类型变量”非常苛刻：不一样就不能硬塞，靠 `T(x)` 显式转换；转换后的范围、截断、拷贝成本你自己负责。**
