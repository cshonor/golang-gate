# 09 - 泛型性能与底层原理

Go 泛型在实现上**不是**「像 C++ 模板那样，每个类型都复制一整份独立代码」那么简单；编译器会在**类型安全**、**编译时间**、**二进制体积**和**运行速度**之间折中。本篇用**新手能听懂**的人话记结论；**分组细节随版本变化**，以 `cmd/compile`、release note 为准。

前置：[03 - 泛型函数](./03-泛型函数.md)、[08 - 泛型与标准库](./08-泛型与标准库结合.md)。

---

## 新手先看：一句话抓住三条线

| 路子 | 人话 | 常见代价 / 好处 |
|------|------|-------------------|
| **C++ 模板** | 每种类型常常**一份独立实例** | 运行可很快，但**编译慢、体积大**（示意对比，不是讲 C++ 细节） |
| **Go 泛型** | **长得像（GC 形状相近）的类型，可能共用一套生成代码** | **兼顾**类型安全、体积、编译速度；热点上常**接近手写** |
| **`interface{}` / `any`** | **运行时**才知道具体类型、走动态派发 | **灵活**（异构、插件），但**有间接调用等开销** |

---

## 1. 底层到底在干什么？（gcshape / 特化）

Go **不会**简单搞成「每个具名类型都一份完全独立的巨型副本」那种刻板印象；代码生成阶段会按 **gcshape（GC 形状）** 一类信息做 **特化 / 共享**：

- **大意**：编译器把「在 GC / 代码生成眼里**足够像**」的类型归组，**能合并就合并**；差得远的再分开实例化。  
- **直觉例子（不背具体名单）**：很多**指针形态**的类型常能**共享**同一份 stencil；**具体哪些类型并成一组**是编译器实现细节，升级编译器后可能变。  
- **对你意味着什么**：二进制和编译时间通常**比「每个类型全复制」温和**；性能上又往往**比到处 `interface{}` 再断言**更直给。

原文档里说的「按 gcshape 做特化 / 共享」，指的就是这套思路。

---

## 2. 泛型 vs `interface`：性能上记什么？（最重要）

### （1）泛型

- **编译期**就尽量知道「这里的 `T` 实际是谁」，许多路径可以**单态化 / 去虚化**。  
- **不走** `interface` 那条「查方法表 + 间接跳转」时，通常**没有**为接口而做的**装箱**（你自己把值赋给 `any` 另说）。  
- **人话**：热点循环里，泛型常常**几乎当手写非泛型**用。

### （2）`interface` / 动态调用

- **运行时**才知道具体实现，**方法分派、间接调用**；需要把具体值塞进接口值里时，还有**装箱**成本。  
- **人话**：**灵活**换**一点性能**；异构集合、插件化，接口仍然合适。

### 一句话

**泛型 ≈ 编译期把类型钉死，尽量按手写速度跑。**  
**`interface` ≈ 运行时再认类型，动态路径有开销。**

**怎么选（粗规则）**：同一套算法、类型集合清楚、又要性能 → **泛型**；要装很多种完全不同的东西、极度灵活 → **`interface`**。

---

## 3. 性能到底要不要操心？

- **普通业务 CRUD**：一般**不用**先焦虑泛型底层。  
- **高频循环、热点路径**：用 **benchmark**、**pprof** 说话，再决定要不要改写法。  
- Go 泛型**很少**天然成为第一瓶颈；更常见的是 IO、锁、分配等。

---

## 4. 面试可背的一段（够用的版本）

> Go 泛型**不是** C++ 那种「每个类型无脑全复制一份」的刻板模板；实现上按 **gcshape 等形状信息**做 **特化 / 共享**，在**体积、编译时间**和**速度**之间折中。编译期尽量固定 `T`，许多场景**接近手写**；相比 **`interface` 动态派发**，通常更省间接成本，但**仍要以版本与实测为准**。

---

## 5. 极简三条（新手永久够用）

1. **Go 泛型 ≠ C++ 模板**：核心是 **shape / 共享** 思路，不是「每个类型一份完整副本」那么简单。  
2. **同场景下泛型往往比 `interface` 更「硬」**：类型在编译期更确定，少一层动态派发。  
3. **日常不用钻实现**：知道「**安全 + 复用 + 热点可测**」即可。
