# 09 - 泛型性能与底层原理（提要）

Go 泛型在实现上**不是**简单等同于「为每个类型各生成一份完整 C++ 模板实例」，而是 **编译器 + 运行时** 协作：在**类型安全**与**二进制体积/编译时间**之间折中。本节只记**工程上够用**的结论；细节随版本演变，以 `cmd/compile` / release note 为准。

---

## 1. 编译期在做什么

- 类型检查阶段会展开类型参数、验证约束满足情况。
- 代码生成阶段会对不同 **gcshape（GC 形状）** 或调用形态做 **特化 / 共享**：**相似内存布局**的类型可能**共享同一份生成代码**，而不是「每个具名类型一份完全独立副本」。

---

## 2. 与 `interface` 调用对比

- **泛型**：许多场景下可**去虚化**，在编译期确定具体类型，**接近手写非泛型**性能。
- **`interface` 动态调用**：有接口装箱、间接调用等成本；**表达力**（插件化、异构集合）有时更好。

---

## 3. 何时需要关心性能

- **热点路径**里大量抽象：应用 **pprof**、**benchmark** 再下结论。
- **二进制体积**：极端多实例化场景可能增大体积；一般业务远小于依赖库与调试信息。

---

## 4. 面试可答的一句话

> 泛型在 Go 里主要解决**编译期类型安全与复用**；实现上按 **shape / 特化** 生成代码，通常比 `interface{}` + 断言更直接，但**不是**无成本抽象，仍以实测为准。
