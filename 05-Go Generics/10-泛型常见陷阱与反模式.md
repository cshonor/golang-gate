# 10 - 泛型常见陷阱与反模式

本篇用**新手最直白的人话**讲常见坑：**一看就懂、一用就少踩雷**。细节规则仍以编译器与 [05](./05-泛型类型约束基础.md)、[06](./06-泛型类型约束进阶.md)、[07](./07-泛型方法.md) 为准。

前置：[06 - 约束进阶](./06-泛型类型约束进阶.md)、[07 - 泛型方法](./07-泛型方法.md)、[05 - 约束基础](./05-泛型类型约束基础.md)。

---

## 新手先看：超简短总结（背这个）

1. **别为炫技乱泛型**：只服务一种类型就别硬上 `[T]`；**真有重复**再抽。  
2. **约束别乱写**：`[T any]` + 体内全是 `type switch` → **不如接口**；联合写太长 → **难实例化、报错难读**。  
3. **编译期 vs 运行期**：泛型管**编译期类型安全**；**多态、插件、高度动态**常是 **接口** 的主场。  
4. **反射**：ORM、插件、动态 schema 等，**新手阶段**优先 **接口 + 反射**，别指望用泛型「模拟一切动态」。  
5. **泛型方法**：**不能**给普通类型随意「事后挂一套类型参数方法」——规则见 [07](./07-泛型方法.md)。

---

## 1. 过度泛型（最常见）

| 现象 | 问题 |
|------|------|
| 全仓库 `Do[T any]`，其实只给一种类型用 | 读者多记一层类型参数，**收益为零** |
| 为「以后可能复用」提前抽象 | **YAGNI**，和 Go **简单优先** 习惯相反 |

**正确做法（人话）**：先写 **`int`、`string`、具体 `struct`**；**第二、第三处出现同样逻辑**再抽成泛型。**够用就行，不提前抽象。**

---

## 2. 约束太宽 or 太窄

- **`[T any]` + 函数里一堆 `type switch`、断言**  
  → 等于**没吃到泛型红利**，读起来还绕 → 考虑 **接口** 或 **拆成多个具体函数**。

- **联合类型列巨长**，再和**方法约束**硬拧在一起  
  → **调用方很难满足**、编译器报错难啃 → 见 [06](./06-泛型类型约束进阶.md)；能 **`cmp.Ordered` / `comparable`** 解决的别手写一长串。

**简单记**：

- **同一套算法、多类型、运算一致** → 泛型 + **合适约束**。  
- **多种不同行为、多实现** → **接口**，别硬套泛型。

---

## 3. 泛型和反射别乱混

- **泛型**：主要在**编译期**把类型关系钉死。  
- **反射**：在**运行时**对着 **`interface{}` / `any`** 那一套值去「看类型、调方法」。

**人话**：别把泛型当成「动态类型系统的替代品」。要做 **动态解析、插件化、ORM 一类高度动态** 的事，**优先 接口 + 反射**（对照 [02-interface-and-reflection](../02-interface-and-reflection/README.md)）；泛型与反射混用时，更要心里有数：**反射 API 和泛型编译期规则不是同一套心智模型**。

---

## 4. 泛型 vs 接口：什么时候用哪个？

| 更适合泛型 | 更适合接口 |
|------------|------------|
| 对**切片、map** 写通用工具：`Max`、`Sort`、`Filter` 等 | 多个结构体**行为不同**（多态） |
| **同一段逻辑**，只是类型不同；要强类型、少断言 | **依赖注入、Mock、运行时换实现** |
| 和 **`cmp.Ordered` / `comparable`** 等约束天然合拍 | **插件、异构集合**、强动态 |

**一句话**：**同逻辑、多类型 → 泛型**；**同签名、多行为 → 接口**。

---

## 5. 方法上的坑（死规则一条）

**不能**给**非泛型类型**随便「事后」加一组「带类型参数的方法」——泛型方法要么挂在**泛型类型**上，要么是包级的**泛型函数**。具体形状与合法写法见 [07 - 泛型方法](./07-泛型方法.md)。
