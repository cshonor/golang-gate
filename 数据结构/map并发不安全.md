# map 并发不安全

## 1. 语言规则

- **未同步的并发读写 map** → **未定义行为**，运行时可能 **直接 fatal**（`fatal error: concurrent map read/write` 等）。  
- **并发只读**（无写）在较新版本中有更细规定，但 **不要依赖**；生产仍应用 `sync.RWMutex` 或 `sync.Map` 等。

## 2. 为什么不安全

- 内部在 **扩容、搬迁、更新槽位** 时修改指针与计数；与其他 goroutine 同时动同一结构 → 竞态。  
- 即使「看起来只读」，若另一路在写，仍可能读到中间状态。

## 3. 正确做法

| 场景 | 方案 |
|------|------|
| 少量 key、读写比不极端 | `map` + `sync.RWMutex` |
| 读多写少、key 集合稳定 | `sync.Map`（注意适用面） |
| 分片 | **分片 map**（每片一把锁）降低锁竞争 |

## 4. 自检

- `sync.Map` 为什么不是「万能替换 `map`」？  
- 能否用 `atomic.Value` 存整个 map 指针实现无锁替换？
