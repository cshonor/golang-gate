# 自旋锁优化：阻塞与唤醒

> 课表：**锁实现原理 · 基础版** — 自旋锁优化 · 阻塞唤醒

## 1. 混合策略（常见 mutex 思路）

1. **快路径**：CAS 一次成功 → 直接进临界区。  
2. **短自旋**：失败则自旋若干次（持锁者可能马上释放）。  
3. **慢路径**：仍失败则 **阻塞**：进入等待队列，**挂起** 线程/协程，让出 CPU。

## 2. 唤醒

- 解锁时检查是否有等待者 → **唤醒一个**（或广播，视语义）。  
- 被唤醒的线程再次 **竞争锁**（可能又有自旋/睡眠）。

## 3. 与 Go 的联想

- `sync.Mutex`：**自旋（有限） + 排队 + 饥饿模式** 的组合演进；与「纯自旋」不同。  
- 协程阻塞 ≠ 一定占 OS 线程（网络等路径不同），但 **mutex 阻塞** 会占住 M 等（概念上对接 GMP）。

- 唤醒后的线程为什么要 **重新抢锁** 而不是「直接把锁交给它」？
