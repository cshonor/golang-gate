# 03 - Background、TODO、上下文树

文末保留 [精简背诵要点](#精简背诵要点) 便于速记。

## 一、两个根上下文：`Background()` & `TODO()`

### 1. 源码签名

```go
func Background() Context
func TODO() Context
```

### 2. 底层本质（重点）

- Go 标准库里各有一个**包级单例**：`backgroundCtx`、`todoCtx`（变量名随版本可读源码确认）。  
- 二者底层**都是** `emptyCtx`（`int` 型的别名实现），行为一致：  
  1. 没有截止时间、不能取消、不携带 `WithValue` 数据；根 ctx 上 `Value` 查不到业务键。  
  2. `Deadline()` → `0, false`  
  3. `Done()` → **`nil`**（永远阻塞的语义：对 `nil` channel 读会永远阻塞，故不要对根 ctx `select` 等取消）  
  4. `Err()` → `nil`  
  5. `Value(any)` → `nil`  

> **物理行为一致**，区别主要在**文档语义与代码可读性**：一个表示「正式根」，一个表示「还没接好 ctx」。

### 3. `context.Background()` 详解

#### 使用场景

1. `main`、初始化、服务进程启动入口  
2. 没有更外层父 ctx 时，作为**派生整棵请求树的起点**  
3. 测试、工具程序里作为**明确的根**

#### 特性

- **不会**因调用某个 `cancel` 而取消（根本没有 `CancelFunc`）  
- **不会**自带 deadline（除非你再包一层 `WithDeadline` 等）  
- 业务里所有「有取消/超时」的 ctx，**祖先链最终**常会回到 `Background`（或来自 `Request.Context()` 等框架根）

#### 定位

> 项目里**正式、稳定语义**的「根」——表示从这里开始建立上下文树。

---

### 4. `context.TODO()` 详解

#### 使用场景

1. 开发阶段：尚不确定父 ctx 从哪传进来，先占位  
2. 重构：函数签名已加 `ctx`，上游分批改造中  
3. 调用第三方 API 必须传 `Context`，当前栈上**暂时没有**合适父 ctx（仍应尽快替换）

#### 禁止场景

- **线上长期**把 `TODO` 当根、当「随便传一个」——排查链路、静态分析、可读性都会变差

#### 定位

> **语义占位**：提醒维护者「这里应接入真实请求 / 任务生命周期 ctx」。

### 5. 二者完整对比表

| 对比项 | `Background()` | `TODO()` |
|--------|------------------|----------|
| 底层类型 | `emptyCtx` 单例 | `emptyCtx` 单例 |
| 可取消 | 否（无 `CancelFunc`） | 否 |
| 超时 / deadline | 无 | 无 |
| `Done()` | `nil` | `nil` |
| 典型使用 | 程序与业务的**正式根** | **临时**占位、待改造 |
| 语义 | 正规根节点 | 待补全 ctx 的标记 |

### 6. 面试标准答案（深挖版）

> 两者底层都是**不可取消的 `emptyCtx`**，接口行为一致；**差别在语义**：  
> `Background` 表示应用或模块的**正式根**；`TODO` 表示**尚未接入正确父 ctx 的临时占位**，不应在线上代码中长期保留。

## 二、上下文树（完整心智模型 + 传播规则）

### 1. Context 设计思想

- 实践中通常有**一个进程级或服务器级的根**（常用 `Background()` 派生）。  
- 业务可用 ctx 均由父 ctx 通过 **`WithCancel` / `WithTimeout` / `WithDeadline` / `WithValue`** 派生，形成**有向树**（父指针在子中，多子分支常见）。

### 2. 派生关系图示

```text
Background（根，自身不携带 cancel）
    │
    └── HTTP Request 根 ctx（框架注入）
          ├── WithTimeout(500ms) 子 ctx
          │     └── WithValue(traceId) 孙 ctx
          └── WithCancel 兄弟子 ctx（与上并列）
```

### 3. 取消传播规则（重中之重）

1. **自上而下**  
   - **父**取消或超时 → **所有子孙**的 `Done()` 关闭（或等价可感知），`Err()` 有原因。  
2. **子不反向取消父**  
   - 对**某个子**调用其 `cancel()`：**只取消该子及其后代**；父、兄弟另一棵子树**不受影响**。  
3. **兄弟互不影响**  
   - 同一父下多个 `WithCancel` 分支，彼此独立，除非它们共享监听同一个被父取消的 ctx。

### 4. 继承规则（心智模型）

子 ctx **挂在父链上**，因此会：

- **感知**父级传来的取消 / 超时（协作式监听 `Done()`）  
- **`Deadline()`** 可能体现父链上**更紧**的截止时间（子可再缩短，不能单方面「放松」父已决定的取消）  
- **`Value(k)`** 沿父链查找（见 [07-WithValue 数据透传.md](./07-WithValue%20数据透传.md)）

## 三、中间件到底在做什么（原理打通）

### 1. 中间件完整工作流程

1. 框架为每个请求提供 **Request 级 ctx**（通常从服务根派生，语义上仍是「请求子树」的根）。  
2. 中间件**不修改**传入的 ctx，而是以它为 **parent**，`WithTimeout` / `WithValue` / `WithCancel` 等**再包一层**，得到**新的子 ctx**。  
3. 把**子 ctx** 写入 `Request` 或继续传给下一层中间件 / `Handler`。  
4. 多层中间件 = **多层包装**，请求生命周期与元数据沿树向下可见。

### 2. 一句话本质

> **不原地改 ctx，只派生新 ctx**；向下追加超时、取消、trace 等，形成**请求专属**上下文子树。

### 3. 为什么不能「修改原 ctx」？

- 对外暴露的 `Context` 接口是**只读协作模型**：`WithX(parent)` **返回新值**，不把 `parent` 改掉。  
- 有利于**并发安全**与**可组合**：不同中间件包不同层，互不踩内存。

更多写法见 [08-context在中间件中的实战.md](./08-context在中间件中的实战.md)。

## 四、串联打通（与 WithCancel 联动）

1. `Background` 作为常见**全局语义根**，自身不提供 `cancel`。  
2. 业务用 `WithCancel` / `WithTimeout` 等从根或 `Request.Context()` **不断派生**树枝（见 [04-WithCancel 手动取消.md](./04-WithCancel%20手动取消.md)、[05-WithTimeout 超时控制.md](./05-WithTimeout%20超时控制.md)）。  
3. **父取消向下传**；子 `cancel` **只砍自己这棵子树**。  
4. 中间件 = **反复派生子节点** + 挂载元数据 + 控制 SLA。  
5. 每个 `WithCancel` 返回的 `cancel`，**只管当前子 ctx 及其后代**，不能「取消父」。

## 五、高频面试深挖追问

### Q1：能不能用 `TODO` 当做根上下文？

能编译、能跑，但**语义错误、不推荐**：读者会以为此处仍待接入；**规范上根应使用 `Background`**（或框架提供的请求根 ctx）。

### Q2：为什么 ctx 常说成「树」而不是单纯一条「链」？

一条请求下常有 **DB / RPC / 缓存** 等多路并发；**树（或森林）**表达：  
**父一次取消，可通知所有分支**；各分支又可各自再包超时、独立 `cancel`，**隔离**清晰。  
（实现上每个节点仍有指向父的链接，整体是多叉派生，不是只能线性串一行。）

### Q3：父 ctx 没有 cancel，子 ctx 可以自己 `cancel` 吗？

**可以。** 子树用子 ctx 的 `CancelFunc` 结束**本分支**即可；父若从未取消，父链其他部分仍存活——这正是「子不杀父」的体现。

---

## 精简背诵要点

- **Background** = 正式根；**TODO** = 临时占位；底层都是 **`emptyCtx`**，`Done()` 为 **`nil`**。  
- **树**：`WithX` 派生；**父取消 → 子孙**；**子 cancel → 只影响子树**；兄弟独立。  
- **中间件**：**派生新 ctx**，不原地改父 ctx。
